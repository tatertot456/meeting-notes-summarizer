File Operations Overview

▶

Files allow data to persist between program runs. Requires #include <fstream>.

ifstream

Input only. File cannot be written. open fails if file doesn't exist.

ofstream

Output only. File cannot be read. Created if it doesn't exist. Existing content is ERASED.

fstream

Both input and output. Must specify mode flags on open. Most flexible option.

Multiple Files

A program can have multiple files open simultaneously. Define a separate file stream object for each file.

#include <fstream>

// ifstream — input only
ifstream gradeList("grades.txt");

// ofstream — output only
ofstream reportFile("report.txt");

// fstream — specify mode explicitly
fstream dFile;
dFile.open("class.txt", ios::in | ios::out);

// Copy 10 numbers between files
fstream infile("input.txt", ios::in);
fstream outfile("output.txt", ios::out);
int num;
for (int i = 1; i <= 10; i++)
{
    infile >> num;
    outfile << num;
}
infile.close();
outfile.close();

Always close files when done — buffered output may not be written to disk until the file is properly closed. Always test for open failure before reading or writing. File stream objects evaluate to false if the open failed.

// Combine flags with | operator
fstream file("data.txt", ios::in | ios::out);

// Test for open failure
ifstream gradeList("grades.txt");
if (!gradeList)
    cout << "Error: could not open file\n";

if (gradeList.fail())
    cout << "Open failed\n";

File Output Formatting

▶

File stream objects support the same formatting manipulators as cout. Requires #include <iomanip>.

#include <iomanip>

ofstream outFile("report.txt");

outFile << fixed << setprecision(2);
outFile << setw(10) << "Name"
        << setw(8)  << "Score" << endl;
outFile << setw(10) << "Alice"
        << setw(8)  << 95.5 << endl;

Everything you can do with cout formatting (setw, fixed, setprecision, showpoint, left, right) works identically with file output stream objects. Always pass file stream objects to functions by reference — never by value.

Detailed Error Testing

▶

Stream state bits provide detailed information about what went wrong with a file operation.

Member FunctionReturns true when...
eof()End of file has been reached
fail()Operation failed (bad data, open failure)
bad()A serious unrecoverable error occurred
good()No error bits are set — stream is OK
clear()Clears all error state bits (not a test — resets state)
ifstream dataFile("numbers.txt");

int num;
while (dataFile >> num)
{
    cout << num << endl;
}

if (dataFile.eof())
    cout << "Reached end of file normally\n";
else if (dataFile.fail())
    cout << "Read failed — bad data?\n";

// Clear error state before seeking (required after eof)
dataFile.clear();
dataFile.seekg(0L, ios::beg);  // go back to beginning

Member Functions for Reading & Writing

▶

These functions handle whitespace and single characters — things that >> and << can't handle cleanly.

// getline: read a full line including spaces
string name, address;
getline(myFile, name);              // reads until '\n'
getline(myFile, address, '\t');     // reads until tab character
// default delimiter is '\n' if third argument omitted

// get: read a single character (including whitespace)
char letterGrade;
gradeFile.get(letterGrade);         // reads next char, even spaces

// put: write a single character to a file
reportFile.put(letterGrade);

// Example: copy file character by character
char ch;
while (inFile.get(ch))
    outFile.put(ch);

The >> operator skips whitespace before reading — you can't use it to read spaces, tabs, or newlines as data. Use get() for single character I/O that preserves whitespace. Use getline() for entire lines. These functions are essential for processing text files where formatting and whitespace matter.

Binary Files

▶

Binary files store raw, unformatted data — no ASCII text conversion. Use the ios::binary flag and the read/write member functions instead of << and >>.

// Open in binary mode
fstream binFile("nums.dat", ios::in | ios::binary);
fstream outBin("output.dat", ios::out | ios::binary);

// Read/write a char
char ch;
binFile.read(&ch, sizeof(ch));   // read sizeof(ch) bytes into ch
outBin.write(&ch, sizeof(ch));   // write sizeof(ch) bytes from ch

// Read/write non-char types: must cast address to char*
int num;
binFile.read(reinterpret_cast<char *>(&num), sizeof(num));
outBin.write(reinterpret_cast<char *>(&num), sizeof(num));

double value;
binFile.read(reinterpret_cast<char *>(&value), sizeof(value));

reinterpret_cast<char *>(&num) tells the compiler "treat the address of num as if it were the address of a char array." The read/write functions work with raw bytes, so they need a char* address. sizeof(num) tells them exactly how many bytes to read or write. This is the standard pattern for all non-char binary I/O.

Creating Records with Structures

▶

Structs can be written to and read from binary files as complete records — all members at once in a single read/write call.

struct TestScore
{
    int studentId;
    double score;
    char grade;
};

TestScore oneTest;
oneTest.studentId = 1001;
oneTest.score = 95.5;
oneTest.grade = 'A';

// Open binary file
fstream gradeFile("grades.dat", ios::out | ios::binary);

// Write entire struct as one record
gradeFile.write(reinterpret_cast<char *>(&oneTest),
                sizeof(oneTest));

// Read it back
fstream readFile("grades.dat", ios::in | ios::binary);
TestScore readTest;
readFile.read(reinterpret_cast<char *>(&readTest),
              sizeof(readTest));
cout << readTest.studentId << " " << readTest.score;

Each struct written to a binary file becomes a fixed-size record — sizeof(TestScore) bytes. Since all records are the same size, you can calculate the position of any record and jump directly to it — the foundation of random access file processing.

Random Access Files

▶

Sequential access reads from start to end in order. Random access jumps directly to any position — like accessing array elements by index.

FunctionPurpose
seekg(offset, mode)Set READ position (seek get) for input files
seekp(offset, mode)Set WRITE position (seek put) for output files
tellg()Return current byte position in input file
tellp()Return current byte position in output file
Mode FlagStarting Point
ios::begBeginning of file
ios::curCurrent position
ios::endEnd of file
// Jump to 26th byte from beginning
inData.seekg(25L, ios::beg);

// Move 10 bytes before current position
outData.seekp(-10L, ios::cur);

// Go to beginning of file
gradeFile.seekg(0L, ios::beg);

// Jump to the Nth struct record (random access by record number)
int recordNum = 4;   // want record #5 (0-based)
gradeFile.seekg(recordNum * sizeof(TestScore), ios::beg);
gradeFile.read(reinterpret_cast<char *>(&oneTest), sizeof(oneTest));

// Get current position
long whereAmI = inData.tellg();

// IMPORTANT: clear eof flag before seeking
if (gradeFile.eof())
{
    gradeFile.clear();    // must clear before seeking
    gradeFile.seekg(0L, ios::beg);
}

Random access only works reliably with fixed-size records — that's why binary struct files are perfect for it. If record sizes vary (like text lines), you can't calculate positions directly. Always clear the eof flag with clear() before calling seekg or seekp — seeking after eof without clearing will fail silently.

Opening a File for Both Input and Output

▶

A file can be open for both reading and writing simultaneously — useful for updating records in place without creating a separate output file.

// Open for both input and output
fstream gradeList("grades.dat", ios::in | ios::out);

// Open for both in binary mode
fstream binData("records.dat", ios::in | ios::out | ios::binary);

// Typical update pattern:
// 1. Seek to the record position
// 2. Read the record
// 3. Modify the data in memory
// 4. Seek back to the same position
// 5. Write the updated record

int recordNum = 2;
TestScore record;

// Read record #3
binData.seekg(recordNum * sizeof(TestScore), ios::beg);
binData.read(reinterpret_cast<char *>(&record), sizeof(record));

// Modify
record.grade = 'B';

// Write back to same position
binData.seekp(recordNum * sizeof(TestScore), ios::beg);
binData.write(reinterpret_cast<char *>(&record), sizeof(record));

Opening with ios::in | ios::out does NOT erase the file — it opens the existing file for modification. This is the key difference from ios::out alone (which erases the file). Use fstream (not ifstream or ofstream) when you need both input and output on the same file.
