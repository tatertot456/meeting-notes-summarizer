What Is Dynamic Programming?

▶

Dynamic programming (DP) solves problems by breaking them into overlapping subproblems and storing results so each subproblem is solved only once.

The problem with plain divide-and-conquer on overlapping subproblems: the same sub-computation gets redone exponentially many times. DP fixes this by caching results — trading redundant computation for memory.

DP applies when a problem has two key properties:

  • Optimal substructure: An optimal solution contains optimal solutions to subproblems
  • Overlapping subproblems: The same subproblems recur many times (unlike divide-and-conquer where subproblems are independent)

Top-Down vs Bottom-Up

▶

Top-Down (Memoization)

Write the natural recursive solution. Add a cache (memo table). Before computing, check if result is already cached. Return cached result or compute and cache.

Bottom-Up (Tabulation)

Solve smallest subproblems first. Fill a table in order of increasing subproblem size. Each entry uses only already-computed entries. No recursion — just loops.

Bottom-Up, No Memory

For some problems (e.g., Fibonacci), only the last few computed values are needed. Discard old entries to reduce space from O(n) to O(1).

Which Is Best?

Bottom-up is usually faster in practice (no recursion overhead, better cache behavior). Top-down is easier to write and only computes needed subproblems.

The Rod Cutting Problem

▶

Given a rod of length n and a price table p[i] for rods of length i, find the maximum revenue by cutting the rod into pieces.

Optimal substructure: The best way to cut a rod of length n uses the best cut of the remaining length after the first piece.

// Bottom-Up Rod Cutting
BOTTOM-UP-CUT-ROD(p, n)
let r[0..n] be a new array
r[0] = 0
for j = 1 to n
    q = -∞
    for i = 1 to j
        q = max(q, p[i] + r[j-i])
    r[j] = q
return r[n]

Runtime: Θ(n²) — two nested loops each running O(n) times. The naive recursive solution runs in Θ(2ⁿ) — exponential. DP provides a dramatic improvement.

Matrix-Chain Multiplication

▶

Matrix multiplication is associative: (A×B)×C = A×(B×C). But the parenthesization dramatically affects the number of scalar multiplications needed.

Given matrices A₁, A₂, ..., Aₙ, find the optimal parenthesization to minimize total multiplications.

4-step DP development:

  • Step 1: Characterize optimal substructure — an optimal solution splits at some index k
  • Step 2: Define recursively: m[i,j] = min cost to multiply Aᵢ through Aⱼ
  • Step 3: Compute m[i,j] bottom-up, filling by increasing chain length
  • Step 4: Reconstruct the optimal solution from the s[i,j] table

Runtime: Θ(n³) — three nested loops. Brute force trying all parenthesizations is Ω(2ⁿ). The DP table has Θ(n²) entries, each computed in O(n) time.

Longest Common Subsequence (LCS)

▶

Given sequences X and Y, find the longest subsequence common to both. Used in DNA comparison, diff tools, version control.

Recursive structure: if xₘ = yₙ, the LCS ends with that character and LCS(X,Y) = LCS(Xₘ₋₁, Yₙ₋₁) + 1. Otherwise, LCS(X,Y) = max(LCS(Xₘ₋₁, Y), LCS(X, Yₙ₋₁)).

Table Construction

Fill an m×n table c[i,j] = length of LCS of X[1..i] and Y[1..j]. Fill row by row left to right.

Reconstruction

Trace back through the table using the b[i,j] pointer table to reconstruct the actual LCS, not just its length.

Runtime: Θ(mn) — fills a table of m×n entries, each in O(1). Space: Θ(mn) for the tables. Can reduce space to O(min(m,n)) if only the length is needed.

Optimal Binary Search Trees

▶

Given keys k₁ < k₂ < ... < kₙ with search probabilities p₁...pₙ and "dummy key" probabilities q₀...qₙ for unsuccessful searches, find the BST minimizing expected search cost.

Unlike standard BST construction, optimal BST doesn't always put the most frequent key at the root. The optimal structure balances frequency against depth across the entire tree.

The expected cost considers every key's depth: E[cost] = Σ (depth(kᵢ)+1)·pᵢ + Σ (depth(dᵢ)+1)·qᵢ

Runtime: Θ(n³) for the straightforward DP formulation. Can be improved to Θ(n²) using Knuth's optimization.

When Should We Use Dynamic Programming?

▶

Use dynamic programming when:

  • The problem asks for an optimal value (min/max) or counting
  • The problem has optimal substructure — solutions to subproblems combine into the overall solution
  • Subproblems overlap — the same sub-computation appears repeatedly in the recursion tree
  • The number of distinct subproblems is polynomial in the input size

Don't use DP if subproblems are independent (use divide-and-conquer instead). Don't use DP if the problem lacks optimal substructure — a greedy solution may work, or the problem may require a different approach entirely.
