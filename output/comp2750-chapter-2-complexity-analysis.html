Computational and Asymptotic Complexity

▶

An algorithm's complexity is a function describing its efficiency in terms of the amount of data it must process. Some algorithms are more efficient than others — we need metrics to compare them.

Time Complexity

Amount of time an algorithm takes in terms of the amount of input. How fast does it run as n grows?

Space Complexity

Amount of memory an algorithm uses in terms of the amount of input. How much RAM does it need?

Asymptotic complexity asks: when n (number of inputs) goes to infinity, what happens to performance? How scalable is the solution?

f(n) = n² + 100n + log₁₀n + 1000

// As n grows, the n² term dominates completely.
// The 100n, log n, and constant terms become negligible.
// Only the DOMINANT TERM matters asymptotically.
// So f(n) = O(n²)

The key insight: for large n, only the fastest-growing term matters. Constants and lower-order terms are dropped. This is why n² + 100n + 1000 is still O(n²) — for large enough n, the n² term so completely dominates that the others are irrelevant to the growth rate.

Big-O Notation

▶

Big-O is the most commonly used asymptotic complexity notation. It provides a formal upper bound on how fast a function grows.

Definition: f(n) = O(g(n)) if and only if there exists a real number c and positive integer N such that 0 < f(n) < c·g(n) for all n > N.

In plain English: after some point N, f(n) never exceeds c times g(n). Constants and lower-order terms don't change the Big-O class.

// Example: f(n) = 2n² + 3n + 1
// Is this O(n²)?

// Need: 2n² + 3n + 1 < c·n²
// Divide by n²: 2 + 3/n + 1/n² < c
// At N=1: 2 + 3 + 1 = 6 → c = 6 works
// So: 2n² + 3n + 1 < 6n² for all n > 1
// Confirmed: f(n) = O(n²) ✓

// Example: 3n² + 4n - 2
// 3 + 4/n - 2/n² < c
// At N=1: 3 + 4 - 2 = 5 → c = 6 works
// So 3n² + 4n - 2 < 6n² for all n > 1 ✓

// All of these are O(n²):
// n² + n,   4n² - n·log n + 12,   n²/5 - 100n,   n·log n? NO, that's O(n log n)

The c and N in the Big-O definition don't have to be unique — many pairs work. The choice of c depends on N and vice-versa. What matters is that SOME pair exists. Big-O gives us a way to compare algorithms quickly by knowing their position in the complexity hierarchy.

Properties of Big-O Notation

▶
  • Transitivity: if f(n) = O(g(n)) and g(n) = O(h(n)), then f(n) = O(h(n))
  • Sum: if f(n) = O(h(n)) and g(n) = O(h(n)), then f(n) + g(n) = O(h(n))
  • Constant factor: a·nᵏ = O(nᵏ) for any a > 0
  • Polynomial: any kth degree polynomial is O(nᵏ)
  • Constant multiple: if f(n) = c·g(n), then f(n) = O(g(n))
  • Limit form: f(n) = O(g(n)) is true if lim(n→∞) f(n)/g(n) is a constant
  • Logarithm base: logₐn = O(logᵦn) for any a, b > 1 — the base of logarithms doesn't matter in Big-O

The logarithm base property is particularly useful: since all logarithms differ only by a constant factor (change of base formula), and Big-O ignores constant factors, we can always write O(log n) without specifying the base. All logarithmic algorithms are in the same Big-O class.

Ω (Omega) and Θ (Theta) Notations

▶

Big-O alone gives too much freedom — selection sort is technically O(n³) since n² = O(n³), but that's not a useful bound. We need tighter notation.

Big-Ω (Lower Bound)

f(n) = Ω(g(n)) if there exist c and N such that f(n) > c·g(n) for all n > N. g is a function that f never falls below (after N). "f grows at least as fast as g."

Big-Θ (Tight Bound)

f(n) = Θ(g(n)) if f(n) = O(g(n)) AND f(n) = Ω(g(n)). Equivalently: c₁·g(n) < f(n) < c₂·g(n) for all n > N. "f grows at exactly the same rate as g."

// Big-Θ means: g(n) is a tight bound — not just an upper bound
// Example: n² + 3n = Θ(n²)
// Because: n² < n² + 3n < 2n² for large n
// c₁ = 1, c₂ = 2, works for n > 3

// Limit forms:
// f(n) = Ω(g(n)) true if  lim(n→∞) g(n)/f(n) is constant
// f(n) = Θ(g(n)) true if  lim(n→∞) f(n)/g(n) is NON-ZERO constant

// Important ordering (slowest to fastest growth):
// O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!)

nᵏ = O((1+ε)ⁿ) for any positive k and ε — any polynomial is bounded above by any exponential. (log n)ε = O(nᵏ) — logarithms to any power grow slower than any polynomial. This gives us the ordering: logarithmic < polynomial < exponential. Prefer log-time algorithms over polynomial, and polynomial over exponential.

Possible Problems with the Notation

▶

Big-O focuses on asymptotic behavior — what happens as n → ∞. This can mislead for practical, finite values of n.

// Algorithm A: 10⁸ · n steps  → O(n)  (linear)
// Algorithm B: 10 · n²  steps → O(n²) (quadratic)

// Big-O says: use Algorithm A (linear is better)
// BUT: Algorithm B is FASTER when n < 10⁷!

// If your data always has n < 1,000,000 (n < 10⁶):
// A requires 10⁸ × 10⁶ = 10¹⁴ steps
// B requires 10 × 10¹² = 10¹³ steps
// B is 10x faster in practice!

"Double-O" notation (OO) has been proposed for this: f(n) = OO(g(n)) if f(n) = O(g(n)) but the constant c is too large to be practically useful. The lesson: Big-O tells you about asymptotic behavior, but for real applications you must also consider the constants. An O(n²) algorithm with a tiny constant can easily outperform an O(n) algorithm with a huge constant for realistic input sizes.

Finding Asymptotic Complexity

▶

Count assignments and comparisons to determine time complexity. Each additional level of loop nesting typically multiplies complexity by n.

// Simple loop — O(n)
for (i = sum = 0; i < n; i++)
    sum = sum + a[i];
// Initialization: 2 assignments (sum=0, i=0)
// Loop body: sum = sum + a[i] → n times
// Loop increment: i++ → n times
// Total: 2 + 2n assignments → O(n)

// Nested loop — O(n²)
for (i = 0; i < n; i++) {
    for (j = 1, sum = a[0]; j <= i; j++)
        sum += a[j];
    cout << sum;
}
// Outer loop: n iterations
// Inner loop: 0 + 1 + 2 + ... + (n-1) = n(n-1)/2 iterations total
// Total assignments: 1 + 3n + 2·n(n-1)/2
// = O(1) + O(n) + O(n²) = O(n²)

// Rule of thumb:
// k nested loops over n → O(nᵏ)
// BUT: not all loops increase complexity — analyze carefully
// Some loops may iterate log n times (binary search) → O(log n)

Best, Average, and Worst Cases

▶

Best Case

The fewest number of steps the algorithm takes. The most favorable input. Often not very informative — almost never actually encountered.

Worst Case

The maximum number of steps the algorithm takes. The most useful measure — guarantees the algorithm won't do worse than this.

Average Case

Performance between extremes. Weight steps for each input by probability of occurrence and sum. Requires knowing the probability distribution of inputs.

Example: Sequential search in an unordered array for target value:

// Best case: target is in position 0 → 1 comparison → O(1)
// Worst case: target is in last position or not found → n comparisons → O(n)

// Average case (uniform distribution — equal probability 1/n for each position):
// Expected steps = (1/n) × (1 + 2 + 3 + ... + n)
//                = (1/n) × (n(n+1)/2)
//                = (n+1)/2
// Average case → O(n) (still linear, just roughly half the array on average)

// Formal average case (weighted sum):
// Σ (steps_for_input_i × probability_of_input_i)
// This is the expected value from probability theory

The worst case is most commonly reported because it provides a guarantee regardless of input distribution. The average case requires knowing the probability distribution of inputs, which is often unavailable or difficult to determine. The best case is usually uninteresting — it's the lucky scenario that rarely represents typical behavior.

Amortized Complexity

▶

When data structures are manipulated by sequences of operations, some early operations impact later ones. Amortized analysis considers the entire sequence to get a more accurate overall performance picture.

// Example: Dynamic array that doubles when full
// Push n elements one at a time

// Worst case for a SINGLE insertion when resizing: O(n)
// (must copy all n existing elements to new array)

// But resizing only happens at sizes 1, 2, 4, 8, 16, ...
// Total copy operations for n insertions:
// 1 + 2 + 4 + 8 + ... + n ≈ 2n

// So n insertions total: O(n) time
// Amortized cost per insertion: O(n)/n = O(1)

// The expensive O(n) resize operations happen so RARELY
// that cheaper O(1) insertions "pay back" the cost.
// This is why vector::push_back is O(1) amortized.

Amortized analysis guarantees worst-case performance over a sequence of operations without making assumptions about input. It's particularly useful when individual operations vary wildly in cost but the expensive ones are rare. The "bankers" and "potential" methods are formal techniques for amortized analysis — the key idea is spreading the cost of expensive operations over many cheap ones.

NP-Completeness

▶

Deterministic Algorithm

Behaves predictably — given the same input, always takes the same steps. One path through the computation.

Nondeterministic Algorithm

Can "guess" what to do next. Different behaviors on different runs. Parallel algorithms may exhibit this via race conditions.

P (Polynomial)

Problems solvable deterministically in O(nᵏ) time. These are "tractable" — feasibly solvable for large inputs.

NP (Nondeterministic Polynomial)

Problems verifiable in polynomial time. A solution can be CHECKED quickly even if it can't be FOUND quickly. P ⊆ NP.

NP-Complete: the hardest problems in NP. A problem is NP-complete if:

  • It is in NP (a proposed solution can be verified in polynomial time)
  • Every other problem in NP can be reduced to it in polynomial time

The million-dollar question: Does P = NP? If ANY NP-complete problem can be solved deterministically in polynomial time, then ALL NP-complete problems can be solved that way — and P = NP. If ANY NP problem is intractable (requires more than polynomial time), then ALL NP-complete problems are intractable. Despite decades of effort, no one has proven P = NP or P ≠ NP — it remains the most famous unsolved problem in computer science.

// Reducibility: Problem A reduces to Problem B if
// every instance of A can be transformed into B in polynomial time
// → if B is solvable efficiently, so is A

// Examples of NP-complete problems:
// - Boolean satisfiability (SAT) — first proven NP-complete
// - Travelling Salesman Problem (find shortest route visiting all cities)
// - Graph coloring (color a graph with k colors so no adjacent nodes match)
// - Knapsack problem

// Current approach: since NP-complete problems can't be solved exactly
// in polynomial time (as far as we know), use:
// - Approximation algorithms (close-enough solutions)
// - Heuristics (good solutions in practice)
// - Special cases that ARE polynomial
