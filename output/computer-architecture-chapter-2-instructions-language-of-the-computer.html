Instruction Sets & ARMv8

▶

An instruction set is the complete repertoire of instructions a computer can execute. Different computers have different instruction sets, but with many aspects in common.

The ARMv8 ISA (using the LEGv8 subset throughout this book) is one of the most commercially successful ISAs ever made:

  • Commercialized by ARM Holdings — largest share of the embedded core market
  • Used in consumer electronics, network/storage equipment, cameras, printers, smartphones
  • Typical of many modern RISC ISAs

Early computers had very simple instruction sets to simplify implementation. Many modern computers do too — simplicity enables higher performance at lower cost.

Arithmetic Operations

▶

All arithmetic instructions have exactly three operands: two sources and one destination.

ADD a, b, c    // a = b + c
SUB a, b, c    // a = b - c

Design Principle 1: Simplicity favors regularity. Every arithmetic instruction has the same form — this regularity makes hardware implementation simpler and faster.

Example: Compiling f = (g + h) - (i + j)

ADD X9, X20, X21    // X9 = g + h
ADD X10, X22, X23   // X10 = i + j
SUB X19, X9, X10    // f = X9 - X10

Registers

▶

Arithmetic instructions operate on registers — small, fast storage directly inside the CPU. LEGv8 has a 32 × 64-bit register file.

Design Principle 2: Smaller is faster. Registers are fast because there are only 32 of them — more registers would mean slower access.

RegistersPurpose
X0 – X7Procedure arguments and return values
X9 – X17Temporaries (not preserved across calls)
X19 – X27Saved registers (callee must preserve)
X28 (SP)Stack pointer
X29 (FP)Frame pointer
X30 (LR)Link register (return address)
XZRConstant value 0

Register optimization is critical — compilers must use registers for frequently accessed variables and only "spill" to memory for rarely used ones. Every memory access costs extra instructions.

Memory Operands & Immediate Values

▶

Main memory stores composite data (arrays, structures, dynamic data). To use memory values in arithmetic, you must first load them into registers, then store results back.

// A[12] = h + A[8];   h in X21, base of A in X22
LDUR X9, [X22, #64]   // load A[8] (offset = 8*8 = 64)
ADD  X9, X21, X9      // X9 = h + A[8]
STUR X9, [X22, #96]   // store to A[12] (offset = 12*8 = 96)

Memory is byte addressed — each address identifies one 8-bit byte. A 64-bit doubleword spans 8 bytes, so array index 8 is at byte offset 64.

Immediate operands embed a constant directly in the instruction:

ADDI X22, X22, #4    // X22 = X22 + 4

Design Principle 3: Make the common case fast. Small constants are extremely common — immediate operands avoid a separate load instruction, saving both time and instructions.

Number Representation

▶

Unsigned Binary

Range: 0 to 2ⁿ–1. Using 32 bits: 0 to 4,294,967,295. All bits contribute positively.

2's Complement Signed

Range: –2ⁿ⁻¹ to 2ⁿ⁻¹–1. Bit 31 is the sign bit. Using 32 bits: –2,147,483,648 to +2,147,483,647.

Signed Negation

To negate: flip all bits, then add 1. Example: +2 = 0000...0010 → –2 = 1111...1110.

Sign Extension

Replicate the sign bit leftward to extend to more bits. LDURSB sign-extends; LDURB zero-extends.

Key insight on signed vs unsigned: the bit pattern 1111...1111 means –1 when interpreted as signed, but 4,294,967,295 when interpreted as unsigned. The hardware is the same — interpretation is everything.

Hexadecimal is used as a compact representation: each hex digit represents exactly 4 bits. Example: eca8 6420 = 1110 1100 1010 1000 0110 0100 0010 0000.

Instruction Formats

▶

All LEGv8 instructions are encoded as 32-bit words. Three main formats:

FormatUsed ForFields
R-formatRegister arithmetic (ADD, SUB, AND...)opcode(11) | Rm(5) | shamt(6) | Rn(5) | Rd(5)
D-formatLoad/store (LDUR, STUR)opcode(11) | address(9) | op2(2) | Rn(5) | Rt(5)
I-formatImmediate arithmetic (ADDI)opcode(10) | immediate(12) | Rn(5) | Rd(5)

Design Principle 4: Good design demands good compromises. Different formats complicate decoding, but allow 32-bit instructions uniformly across all instruction types. Keeping formats as similar as possible minimizes the complexity cost.

Stored Program concept: Instructions are just binary data stored in memory alongside other data. This enables programs that operate on programs — compilers, linkers, loaders — and binary compatibility across machines with the same ISA.

Logical Operations

▶
OperationCLEGv8Use
Shift left<<LSLMultiply by 2ⁱ
Shift right>>LSRDivide by 2ⁱ (unsigned)
Bitwise AND&AND, ANDIMask bits — clear selected bits to 0
Bitwise OR|ORR, ORRISet selected bits to 1
Bitwise XOR^EOR, EORIFlip selected bits (XOR with all 1s = NOT)

LSL by i bits = multiply by 2ⁱ. LSR by i bits = divide by 2ⁱ (unsigned only). Shift instructions are critical for array index calculations — multiplying an index by 8 (for 64-bit elements) becomes LSL #3.

32-bit constants require two instructions since the immediate field is only 12 bits:

MOVZ X9, 255, LSL 16   // load upper 16 bits, zero rest
MOVK X9, 255, LSL 0    // keep upper bits, patch lower 16

Branches & Conditional Operations

▶

Branches change the flow of execution based on conditions:

CBZ  register, L1   // branch to L1 if register == 0
CBNZ register, L1   // branch to L1 if register != 0
B    L1             // unconditional branch to L1

For richer comparisons, use an S-suffix instruction to set condition codes, then branch:

SUBS XZR, X22, X23   // set flags from X22 - X23
B.LT Less            // branch if less than (signed)
B.GE Exit            // branch if greater or equal

Compiling a while loop:

// while (save[i] == k) i += 1;
Loop: LSL  X10, X22, #3      // X10 = i * 8
      ADD  X10, X10, X25     // address of save[i]
      LDUR X9, [X10, #0]     // X9 = save[i]
      SUB  X11, X9, X24      // X11 = save[i] - k
      CBNZ X11, Exit         // if != 0, exit
      ADDI X22, X22, #1      // i++
      B    Loop
Exit:

Branch addresses are PC-relative — the branch instruction stores an offset from the current program counter, not an absolute address. This makes code position-independent and more compact.

Procedures & the Stack

▶

6 steps for a procedure call:

  • Place parameters in X0–X7
  • Transfer control: BL ProcedureLabel (saves return address in X30/LR)
  • Acquire storage for the procedure on the stack
  • Perform the procedure's operations
  • Place result in X0–X7 for the caller
  • Return: BR LR (jumps to address in X30)

Leaf Procedures

Don't call other procedures. Must save any X19–X27 registers they use on the stack, then restore them before returning.

Non-Leaf Procedures

Call other procedures. Must save LR (return address) AND all needed registers on the stack — otherwise the nested BL overwrites LR.

Temporaries (X9–X17)

Not preserved across calls — callee can freely overwrite them. Caller saves if needed.

Saved Registers (X19–X27)

Callee must save and restore these if used. Caller can rely on them being unchanged after a call.

Memory layout from low to high: Text (code) → Static data → Heap (grows up, malloc/new) → Stack (grows down, automatic variables). The stack pointer SP always points to the top of the stack.

Translation, Linking & Loading

▶

Getting from C source to running program involves multiple steps:

Compiler

Translates C to assembly language. Applies optimizations (O1, O2, O3 levels). Optimization level dramatically affects instruction count and CPI.

Assembler

Translates assembly to machine code (object module). Produces: header, text segment, static data segment, relocation info, symbol table, debug info.

Linker

Merges object modules. Resolves labels and external references. Patches location-dependent references. Produces executable image.

Loader

Reads header, creates virtual address space, copies text and data into memory, initializes registers, jumps to startup routine which calls main().

Dynamic linking loads library procedures only when called — avoiding image bloat from static linking and automatically picking up new library versions. Java uses bytecode interpreted by the JVM, with JIT compilation of "hot" methods to native code for performance comparable to optimized C.

Lessons from the C sort example:

  • Instruction count and CPI are not good indicators in isolation — both matter
  • Compiler optimizations are sensitive to the algorithm
  • Java JIT compiled code is significantly faster than interpreted and can approach optimized C
  • Nothing can fix a dumb algorithm — quicksort beats bubblesort by 2500× regardless of language or optimization

ARM vs. x86 & Fallacies

▶

The x86 ISA evolved from the Intel 8080 (1974) through decades of backward-compatible extensions — accumulating enormous complexity along the way.

ARMx86
Date19851974 (8080)
Instruction size32 bits (fixed)Variable length
Registers31 × 64-bit16 × 64-bit
Design philosophyRISC — simple, regularCISC — complex, backward compatible
ImplementationExecute directlyHardware translates to micro-ops

x86 implements complex instructions by translating them to simpler internal micro-operations — essentially running a RISC core inside a CISC shell. Market share makes this economically viable despite the overhead.

Fallacies:

  • Powerful instructions = higher performance: False. Complex instructions are hard to implement and may slow down all instructions including simple ones. Compilers are excellent at generating fast code from simple instructions.
  • Use assembly for high performance: False. Modern compilers understand modern processor pipelines better than most humans and generate better code.
  • Backward compatibility means the ISA doesn't change: False. ISAs only accrete — new instructions are always added, never removed.

Design principles recap: Simplicity favors regularity · Smaller is faster · Make the common case fast · Good design demands good compromises.
