Decision Trees & Lower Bounds for Sorting

▶

Any comparison-based sort can be modeled as a decision tree — a binary tree where each internal node represents a comparison and each leaf represents an output permutation.

  • A correct sort must reach every possible permutation — n! leaves
  • A binary tree with n! leaves has height ≥ ⌈log₂(n!)⌉
  • By Stirling's approximation: log₂(n!) = Θ(n log n)

Theorem: Any comparison-based sorting algorithm requires Ω(n log n) comparisons in the worst case. This means merge sort and heapsort are asymptotically optimal comparison sorts.

To beat Ω(n log n), we must use information beyond comparisons — knowledge of the values themselves. That's exactly what counting sort, radix sort, and bucket sort do.

Counting Sort

▶

Counting sort assumes all input values are integers in the range [0, k]. It counts occurrences rather than comparing elements.

COUNTING-SORT(A, B, k)
// Step 1: Count occurrences
let C[0..k] be a new array initialized to 0
for j = 1 to n
    C[A[j]] = C[A[j]] + 1

// Step 2: Accumulate counts
for i = 1 to k
    C[i] = C[i] + C[i-1]

// Step 3: Place elements
for j = n downto 1
    B[C[A[j]]] = A[j]
    C[A[j]] = C[A[j]] - 1

Runtime: Θ(n + k). When k = O(n), this is Θ(n) — linear time! The iteration in step 3 goes backwards (j = n downto 1) to preserve stability: equal elements appear in their original order.

Stability is crucial because counting sort is used as a subroutine in radix sort — the sort must preserve the relative order of equal elements from previous passes.

Radix Sort

▶

Radix sort sorts multi-digit numbers digit by digit, from least significant to most significant digit, using a stable sort at each pass.

Why least significant digit first? If you sort by most significant digit, you'd need to recursively sort each bucket — expensive. Sorting from least to most significant with a stable sort accumulates the correct order across passes.

Runtime

Θ(d·(n+k)) where d = number of digits, k = range per digit. With d constant and k = O(n), this is Θ(n).

Stability Requirement

The per-digit sort MUST be stable. Counting sort is the natural choice — that's why we studied stability there first.

Bucket Sort

▶

Bucket sort assumes input is uniformly distributed over [0, 1). It divides the range into n equal-width buckets and distributes elements into them.

BUCKET-SORT(A)
let B[0..n-1] be an array of empty lists
for i = 1 to n
    insert A[i] into list B[⌊n·A[i]⌋]
for i = 0 to n-1
    sort list B[i] with insertion sort
concatenate lists B[0], B[1], ..., B[n-1]

Expected runtime: Θ(n). With uniform distribution, each bucket gets O(1) elements on average — insertion sort on each bucket takes O(1) expected time. Total: n buckets × O(1) = O(n).

Unlike counting and radix sort, bucket sort's runtime depends on the input distribution. If inputs cluster in one bucket, that bucket takes longer to sort — worst case degrades to O(n²).
