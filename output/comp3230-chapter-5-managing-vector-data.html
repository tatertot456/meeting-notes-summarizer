Vector Storage Models

▶

Vector data stores geographic features as x-y coordinate pairs. Each pair is a vertex (plural: vertices). Line endpoints are called nodes. A multipart feature contains several separate pieces — Hawaii is one state feature composed of multiple island polygons.

Features are linked to attribute tables by a unique integer (Feature ID / Object ID). A feature class is a collection of similar features stored as a dataset — it contains both the spatial geometry and the table, and can only hold one geometry type (point, line, OR polygon).

Spaghetti Model

Simple format — each feature is stored as a separate unrelated object. Adjacent boundaries stored twice (redundant). Very robust due to simplicity. Commonly used for data transfer between GIS systems. Shapefiles use this model.

Topological Model

Also stores spatial relationships between features — which states are adjacent, what connects or overlaps. Can test adjacency, connectivity, overlap, intersection. Can detect errors like overlapping polygons or county boundaries outside their state.

File Management Best Practices

▶

NEVER use spaces in folder or file names. Not ever. Anywhere. Nix. No. Nein. Nyet. Non. Use letters, numbers, and underscores only. Keep folder names 8–15 characters. Spaces break GIS functions and are extremely difficult to fix without breaking maps and projects.

  • Use well-organized folder structures — NOT the Desktop or Documents folders
  • The project geodatabase is the default save location but not always the best choice
  • Name files descriptively: SnailHabitat, not rastercalc1
  • Separate permanent data libraries from in-progress work
  • Put downloads in separate folders for easy archival or deletion
  • Make file extensions visible in Windows Explorer (uncheck "Hide extensions for known file types")

Data access considerations:

  • GIS data is often shared by multiple users — file locks prevent simultaneous corruption
  • Editing functions won't be available if someone else has an exclusive lock on a file
  • Read-write permissions are controlled at the folder level by the OS
  • Downloaded data is often zipped — must extract before ArcGIS can use it (use 7zip or similar)

Source scale matters when compiling data. Data from different source scales may not align properly — check metadata to confirm source scale is appropriate for your project's objective. Small-scale data is easier to find but may be inadequate for detailed analysis.

Geocoding

▶

Geocoding converts postal addresses into map locations by matching address fields against a reference database (street network with address ranges). Requires a locator — the packaged rules and street data for matching.

Three phases of address matching:

  • Preprocessing — parsing and standardizing the address format
  • Matching — the geocoding engine matches the address to the reference database using a scoring system. Matching conditions can be relaxed to improve hit rates.
  • Plotting — the matched address is plotted as a point using linear interpolation within the address range of the matched street segment

Linear Interpolation

Estimates where an address falls within a street segment's address range. 150 Main St is plotted halfway along a segment with range 100–200. An alternative uses actual building centroid coordinates from a location database.

Offset Options

Side offset: places point to the left or right of the street centerline. End offset: moves point away from the end of a street segment. Both improve point placement for visualization.

Hit Rate

Percentage of addresses successfully matched. 60% is considered unacceptable. 85% is a statistical minimum. 95%+ required for competitive location services. Improve with current, accurate reference data.

Reference Database

Street name, address ranges, ZIP codes per segment. Historically from TIGER/Line files (Census). Now commonly from commercial providers. ArcGIS Online has a World Locator (credits required for batch geocoding).

Geocoding variations:

  • Intersection matching — matches to street intersections rather than address ranges
  • ZIP code geocoding — matches ZIP code to its centroid
  • Parcel-level geocoding — matches parcel number to parcel centroid or boundary
  • Reverse geocoding — converts lat/lon coordinates to descriptive addresses
  • Place name alias geocoding — matches place names (restaurant, museum) to a street address
  • Photo geocoding (geotagging) — attaches location information to photographs

Metadata

▶

Metadata is data about a dataset — it describes the dataset's purpose, content, source, and quality. It travels with the data as a separate file or embedded in the feature class. Always create metadata for every dataset you produce.

SectionContent
TagsSearch terms used to discover the data in catalogs
SummaryBrief description of the purpose of the data
DescriptionLonger description: contents, data sources, processing steps performed
CreditsCitation of data creator and original source information
Use LimitationRestrictions on use or redistribution; disclaimers
Appropriate Scale RangeSource scale or intended display scale of the data
Bounding BoxCoordinate extent of the data in geographic degrees

Standards-based metadata (FGDC, ISO 19115) is often required when publishing data for clients or organizations. At minimum, always record: source citation, data processing steps, coordinate system used, and scale. Metadata is your professional record of provenance — without it, data becomes untrustworthy over time.

The TIN (Triangulated Irregular Network) is a vector-based surface model representing terrain using triangles of varying size — smaller triangles where terrain is complex, larger where it is flat. TINs are precise at data points but interpolated between them. Used for 3D analysis and visualization of terrain.

Pathnames, Environments & Sharing

▶

Pathnames locate source data files. ArcGIS Pro assigns path types automatically.

Relative Paths

Start from the project file location. Used for data on the local hard drive. Move with the project folder if the entire project is copied correctly.

Absolute Paths

Start from the drive letter. C:\gisclass\mgisdata\Oregon\counties

UNC Paths

\\server\share\folder — for data on a network file server. Used automatically by ArcGIS Pro for network locations.

URL Paths

For Internet data services (ArcGIS Online, WMS, WFS). Data remains on the server; only display/query data is transferred.

When copying a project, copy the ENTIRE project folder — not just the .aprx file or the geodatabase. If data is deleted, moved, or renamed, projects lose track and show a red exclamation point. Projects cannot be renamed because renaming disrupts internal pathname references. "Save As" only copies the .aprx file, not the project data.

Environment Settings — affect tool operation globally or per-run:

  • Environments tab on a tool — settings affect only that single tool run
  • Environments button on the Analysis ribbon — settings affect ALL tools all the time
  • Common settings: output location, output coordinate system, processing extent

Coordinate system reminders when compiling data:

  • Pick one suitable coordinate system and convert all data to it
  • Use NAD83 as the standard datum unless specifically directed otherwise
  • Check Map Properties window to verify all layers share the same coordinate system
