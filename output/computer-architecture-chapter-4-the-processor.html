Introduction & Instruction Execution

▶

CPU performance depends on three factors — instruction count (determined by ISA and compiler), CPI, and cycle time (both determined by CPU hardware). This chapter examines how the hardware actually executes instructions.

Every instruction follows the same basic execution steps:

  • Fetch: PC → instruction memory → fetch the instruction
  • Decode: Register numbers → register file → read registers
  • Execute: ALU computes arithmetic result, memory address, or branch target
  • Memory: Access data memory for loads and stores
  • Write Back: PC ← target address or PC + 4; write result to register

We examine two implementations: a simplified single-cycle version (conceptually clear) and a pipelined version (realistic and fast). The simple subset covers LDUR, STUR, arithmetic/logical R-type instructions, and CBZ.

Logic Design Basics

▶

All CPU hardware is built from two types of logical elements:

Combinational Elements

Output is purely a function of current inputs — no memory. Examples: AND gates, ALUs, adders, multiplexers. Output changes whenever input changes.

Sequential (State) Elements

Store information — output depends on stored state AND current inputs. Examples: registers, memory. Updated only on the clock edge.

Registers are edge-triggered: they capture input D and present it as output Q only when the clock transitions from 0 to 1. Between clock edges, combinational logic transforms data from one set of state elements to the next.

The longest combinational delay between any two state elements determines the minimum clock period. The entire processor must run at the speed of its slowest path — this is the critical path.

Multiplexers are essential when different instructions need different data sources routed to the same destination. You can't just wire two sources together — a mux selects which one to use based on a control signal.

Building the Datapath

▶

The datapath is the collection of elements that process data and addresses: registers, ALUs, multiplexers, and memories. We build it incrementally for each instruction type.

Instruction Fetch

PC → instruction memory → fetch 32-bit instruction. PC + 4 computed by adder for sequential execution. PC is a 32-bit register updated every cycle.

R-Format Instructions

Read two register operands (Rn, Rm) from register file. ALU performs operation. Write result back to destination register (Rd).

Load/Store Instructions

Read base register. Sign-extend 9-bit offset to 64 bits. ALU adds base + offset to get memory address. Load reads memory → register. Store writes register → memory.

Branch Instructions

Read register operands. ALU subtracts to compare (checks Zero output). Sign-extend displacement, shift left 2, add to PC+4 for target address.

Each datapath element can only do one function at a time — this is why we need separate instruction and data memories. If we used one memory, a load instruction would need to fetch an instruction AND access data simultaneously, which is impossible with a single port.

ALU Control & Main Control Unit

▶

The ALU can perform many operations — control logic must select the right one for each instruction. A two-level scheme keeps it manageable:

ALU ControlFunction
0000AND
0001OR
0010Add
0110Subtract
0111Pass input B
1100NOR

Two-level control: The main control unit generates a 2-bit ALUOp from the instruction opcode. A second level of logic uses ALUOp plus the instruction's function bits to generate the final 4-bit ALU control signal.

LDUR and STUR both produce ALUOp = 00 → always add (base + offset). CBZ produces ALUOp = 01 → pass input B (just need to check if register = 0). R-type produces ALUOp = 10 → use function bits to determine the specific operation.

The main control unit also generates: RegWrite, ALUSrc, MemRead, MemWrite, MemToReg, Branch — one signal per datapath multiplexer or enable line.

Pipelining

▶

The single-cycle datapath's critical path is the load instruction: IF → ID → EX → MEM → WB = 800ps. Every instruction must wait this long even if it doesn't need all stages.

Pipelining overlaps execution of multiple instructions — like an assembly line. While one instruction is in MEM, the next is in EX, the next in ID, and so on.

LEGv8 5-stage pipeline:

  • IF: Instruction fetch from memory
  • ID: Instruction decode and register read
  • EX: Execute operation or calculate address
  • MEM: Access memory operand
  • WB: Write result back to register
MetricSingle-CyclePipelined
Clock period800ps (slowest instruction)200ps (slowest stage)
Throughput1 instruction / 800ps~1 instruction / 200ps
Latency per instruction800ps1000ps (5 × 200ps)

Pipelining improves throughput (instructions completed per second), NOT latency (time per individual instruction). Each instruction actually takes longer in a pipeline — but many more complete per unit time. Ideal speedup = number of pipeline stages.

LEGv8's ISA was designed with pipelining in mind: fixed 32-bit instruction size (easy to fetch/decode in one cycle), few regular instruction formats (decode and register read in one step), load/store addressing (address in stage 3, memory in stage 4).

Pipeline Hazards

▶

A hazard is any situation that prevents the next instruction from starting in the next cycle. Three types:

Structural Hazards

A required resource is busy. Example: single memory for both instructions and data. Solution: separate instruction and data memories (or caches).

Data Hazards

An instruction needs data that a previous instruction hasn't finished computing yet. Example: ADD X19, X0, X1 followed immediately by SUB X2, X19, X3.

Control Hazards

The pipeline doesn't know which instruction to fetch next because a branch hasn't been resolved yet. The CPU is still in the ID stage of the branch when it needs to fetch the next instruction.

Forwarding (Bypassing)

Use the result as soon as it's computed — route it directly from the output of EX or MEM back to the input of EX — without waiting for it to be written to the register file.

Load-use hazard: forwarding can't always help. If a LDUR result is needed by the immediately following instruction, the data isn't available until after MEM — but the next instruction needs it at the start of EX. You can't forward backward in time. A one-cycle stall (bubble) is unavoidable.

Code scheduling to avoid stalls: Reorder independent instructions to put non-dependent work between a load and its use:

// With stalls (13 cycles):        // Reordered (11 cycles):
LDUR X1, [X0,#0]                  LDUR X1, [X0,#0]
LDUR X2, [X0,#8]                  LDUR X2, [X0,#8]
ADD  X3, X1, X2   ← stall         LDUR X4, [X0,#16]  ← moved up
STUR X3, [X0,#24]                 ADD  X3, X1, X2    ← no stall
LDUR X4, [X0,#16]                 STUR X3, [X0,#24]
ADD  X5, X1, X4   ← stall         ADD  X5, X1, X4    ← no stall
STUR X5, [X0,#32]                 STUR X5, [X0,#32]

Branch Prediction

▶

Stalling on every branch until the outcome is known is too expensive in deeper pipelines. The solution: predict the branch outcome and continue fetching — only stall if the prediction is wrong.

Static Prediction

Fixed rules based on branch direction. Predict backward branches taken (loops usually repeat). Predict forward branches not taken (if-statements usually skip). No hardware overhead.

Dynamic Prediction

Hardware records recent history of each branch. 2-bit saturating counter: strongly taken, weakly taken, weakly not taken, strongly not taken. Adapts to actual program behavior.

Speculation

Predict and continue issuing instructions — but don't commit results until prediction confirmed. If wrong: flush the pipeline and re-fetch from correct path.

LEGv8 Simple Approach

Predict branches not taken. Fetch instruction after branch with no delay. If branch is taken, flush the incorrectly fetched instruction (one-cycle penalty).

In LEGv8, branch resolution hardware is moved to the ID stage (instead of EX) to minimize the branch penalty to one cycle. This requires a dedicated comparator and adder in the decode stage — extra hardware cost to reduce stall penalty.

Instruction-Level Parallelism & Concluding Remarks

▶

ILP (Instruction-Level Parallelism) goes beyond basic pipelining — issuing multiple instructions per cycle.

Multiple Issue

Issue 2–8 instructions per cycle. Requires multiple functional units (ALUs, FP units, load/store units) operating simultaneously.

Dynamic Scheduling

Hardware reorders instructions at runtime to avoid stalls. Needed because not all stalls are predictable (cache misses) and different ISA implementations have different latencies.

Loop Unrolling

Replicate the loop body multiple times to expose more independent instructions for the pipeline. Reduces loop overhead and enables better scheduling.

SIMD + ILP

Combining unrolled loops with AVX SIMD (4 doubles per instruction) gives massive throughput — 4 C rows × 4 doubles = 16 elements per iteration of the matrix multiply inner loop.

ILP limits: programs have real dependencies that limit parallelism. Pointer aliasing makes it hard to determine if two memory operations are independent. Memory bandwidth and limited instruction windows prevent keeping pipelines full. Speculation helps — but complexity increases power consumption, contributing to the power wall.

Concluding principles:

  • ISA design influences datapath and control — and vice versa
  • Pipelining improves throughput via parallelism, not latency
  • Hazards (structural, data, control) limit pipeline efficiency
  • Compilers can schedule code to reduce stalls — but require knowledge of pipeline structure
  • Poor ISA design (complex instructions, variable lengths, delayed branches) makes pipelining significantly harder
