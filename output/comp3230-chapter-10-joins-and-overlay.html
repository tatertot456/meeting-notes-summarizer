Joins vs. Overlay — Key Difference

▶

Both joins and overlay combine data tables from two sources, but they differ critically in how they handle overlapping features:

Joins (Table Join & Spatial Join)

Features remain WHOLE — geometry is not split. Attribute data from one table is appended to another based on a key or spatial relationship. Features keep their original shape.

Overlay (Intersect & Union)

Features are SPLIT where they overlap — new features are created from the intersection of input geometries. Each new feature gets a unique combination of attributes from both inputs. More complex but avoids one-to-many problems.

Use a JOIN when you want to attach attribute information to existing features. Use OVERLAY when you need to analyze the geographic combination of two datasets — where both sets of conditions exist simultaneously in the same place (e.g., landslide hazard = steep slope AND weak rock).

Spatial Joins

▶

A spatial join matches records using a spatial relationship instead of a common key field. The target feature class is copied to a new dataset containing both tables combined. The same spatial operators as spatial queries apply.

OperatorHow Records Are MatchedExample
IntersectJoin record touches or overlaps target featureAttach the zoning record that intersects each parcel → every parcel gets its zoning info
Within / ContainmentTarget feature falls inside join featureEach septic system gets the geology unit it falls within
Proximity (Closest)Nearest join feature to each target featureEach city gets info from its closest airport + a distance field is added

Intersect vs. Within difference: If a septic system lands exactly ON a geology boundary, Within yields a Null (it's not completely inside either polygon), while Intersect finds two matches (it touches both). Null values in output alert you to these edge cases — they signal exceptions or logical consistency errors in the data.

Cardinality, Join Types & Merge Rules

▶

Cardinality describes how many join features match each target feature. Always anticipate the expected cardinality before setting up a join.

ScenarioExpected CardinalityBest Join TypeWhy
Parcels + ZoningMany-to-one (most parcels in one zone)One-to-oneOne zone per parcel expected. Two matches = data error → Null alerts you.
Cities + AirportsOne-to-one (one closest airport)One-to-oneOnly one airport can be closest. Two equidistant = rare exception → Null alerts you.
Septics + GeologyMany-to-one (septic in one polygon)One-to-oneOne geology unit per septic. On-boundary = exception → Null alerts you.
Highways + SoilsOne-to-many (highway crosses many soils)One-to-one with merge rulesMultiple copies of each highway is not useful. Summarize soils per highway with merge rules instead.

Join types:

One-to-One Join Type

Each target feature produces ONE output copy. Multiple matching join records are summarized using merge rules. Most common choice — cleaner output.

One-to-Many Join Type

Each target feature is DUPLICATED — one copy per matching join record. Used when you need every individual match preserved. Can create unwieldy output for many matches.

Merge rules — how to summarize multiple join records into one (used with one-to-one join on one-to-many cardinality):

  • Mean, Sum, Min, Max, Std Dev — statistical rules for numeric fields (e.g., mean clay content across all soils a highway crosses)
  • First — uses the first matching record's value. Default for text/categorical fields. Often not meaningful.
  • Concatenate — combines all matching values into a list in a single field (e.g., list all hydrologic groups for soils a highway crosses)
  • Join_Count field — automatically added to show how many join records matched each target feature

Do NOT confuse join TYPE with cardinality. A one-to-one JOIN TYPE is commonly used on tables with a one-to-MANY cardinality — you use merge rules to summarize the many join records into one result per target feature. The join type describes the OUTPUT structure, not the input relationship.

Classic Overlay — Intersect & Union

▶

Overlay splits input features where they overlap, creating new features with unique combinations of attributes from all inputs. Each output polygon is a "child" of its "parent" input polygons and inherits all their attributes.

Intersect

Keeps ONLY the areas where both inputs overlap. Output = the geographic intersection of all inputs. Use for: suitability analysis (find areas meeting ALL conditions), combining data where overlap is the area of interest.

Union

Keeps ALL areas from BOTH inputs, whether they overlap or not. Output = full extent of all inputs combined. Use for: combining datasets where you need the complete picture, querying all possible combinations of values.

Geometry rules for overlay:

  • Union: polygon inputs ONLY — requires two polygon layers
  • Intersect: all geometry types supported — polygon, line, or point with polygon
  • Output geometry cannot be HIGHER than the highest input type (line + polygon → line or point, never polygon)
  • Line-in-polygon: assigns geology attributes to streams
  • Point-in-polygon: assigns school district to houses for sale

Intersect vs. Clip comparison:

OperationBoundary UsedFeatures Split?New Attribute Fields?
Clip (Extraction)Only the outer boundary of clip polygonYes — cut at boundaryNO — only target table fields
Intersect (Overlay)All polygon boundaries from both inputsYes — split at all boundariesYES — fields from BOTH input tables

Overlay practical workflow — Snail Habitat example: (1) Query elevation layer → select suitable elevation range. (2) Query geology layer → select limestone areas. (3) Query vegetation layer → select dense conifer areas. (4) Intersect the three selected feature classes → output = only areas meeting ALL three conditions simultaneously. This is the core of GIS suitability analysis.

After overlay, Shape_Length and Shape_Area geodatabase fields update automatically. User-defined area/length fields (e.g., ACRES) must be updated manually using Calculate Geometry Attributes tool.

Data Quality Issues — Slivers & XY Tolerance

▶

Slivers — tiny artifact polygons created when overlaying two datasets whose boundaries should match but don't due to poor logical consistency:

  • Census tract boundaries that don't align with county boundaries produce slivers along the boundary when intersected
  • Slivers are NOT real geographic features — they're errors
  • They inflate feature counts and create meaningless attribute combinations

XY Tolerance Solution

Vertices separated by less than the XY tolerance are collapsed into a single vertex during overlay. Can reduce or eliminate slivers. Set in the geoprocessing environment settings.

XY Tolerance Tradeoff

ALL close vertices get collapsed — not just sliver-causing ones. This degrades accuracy and precision of the entire dataset. Use the smallest tolerance that fixes the problem.

The best solution to slivers is fixing the source data so boundaries are logically consistent before overlaying. XY tolerance is a workaround that sacrifices data quality. Prevention (proper topology in the source data) is always better than correction after the fact.

Setting Up a Spatial Join — Decision Framework

▶

Work through these questions in order to configure a spatial join correctly:

1. What should the final output look like?
   → Defines the target layer (output matches the target's geometry)

2. Which is the target layer?
   → The layer that gets the new attributes appended to it

3. Which spatial operator is appropriate?
   → Intersect? Within? Proximity (closest)?

4. Which fields do I want in the output table?
   → Can limit output to needed fields only

5. Could a target feature match MULTIPLE join features?
   → If YES: choose one-to-many join type (duplicate targets)
              OR one-to-one with merge rules (summarize matches)
   → If NO: one-to-one join type, no merge rules needed

Worked example — Congressional districts & earthquake deaths:

  • Goal: list of districts with representative names and total earthquake deaths
  • Target: congressional districts (cd114) — output must be a districts layer
  • Join: earthquakes (quakehis)
  • Operator: Intersect — match quakes that touch a district
  • Fields needed: district ID, party, representative name, sum of deaths
  • Cardinality: one district can match MANY quakes (one-to-many)
  • Solution: one-to-one join type with Sum merge rule on deaths field
  • Output: one row per district with total deaths; districts with no quakes show Null
