Setup & Toolchain

▶

The Raspberry Pi runs Raspbian — a Linux-based OS. You can access it directly, remotely via RDP (Remote Desktop Protocol), or over a network connection.

Remote Desktop: Run RDP, enter raspberrypi as the computer. Username: pi, Password: raspberry.

Essential Linux Shell commands:

ls          # List contents of current directory
cd          # Change directory
whoami      # Current user
ps -ux      # Running processes
help        # List available commands
shutdown now  # Graceful shutdown (preferred over pulling the plug)

Three tools you'll use: the assembler (turns assembly into object code), the linker (turns object files into an executable), and the debugger (lets you inspect registers during execution).

Build process:

as -g -mfpu=vfpv2 -o first.o first.s   # Assemble → object file
gcc -o first first.o                    # Link → executable
./first                                 # Run (loader invoked automatically)

The Makefile automates this. Execution runs bottom-to-top in a makefile. Indentation must use tabs — not spaces — or make will fail.

First Programs & Program Structure

▶

Every assembly program needs a starting point declared with .global main. Lines in assembly are one of five types: directives, comments, instructions, parameters, or labels.

/* -- first.s */
.global main        @ 'main' is our entry point — must be global
main:               @ Label marking the start of main
    mov r0, #2      @ Put value 2 into register r0
    bx lr           @ Branch and exchange: return from main

Adding two numbers:

.global main
main:
    mov r1, #3      @ r1 ← 3
    mov r2, #4      @ r2 ← 4
    add r0, r1, r2  @ r0 ← r1 + r2  (result = 7)
    bx lr

Register reuse: Registers are a limited resource — reuse registers that are no longer needed. Using r0 as both an input and the result destination saves a register for other use elsewhere in the program.

Three sections of an assembly program:

.data

Initialized data and constants. Values declared here do NOT change at runtime.

.bss

Uninitialized variables that WILL change at runtime. Allocated but not filled.

.text

The actual code. All instructions go here. Must be 4-byte aligned.

Alignment (.balign 4)

Ensures the next address starts on a 4-byte boundary. Always align variables and code sections. Unused bytes are padding — never reference them.

Memory: Load, Store & Variables

▶

Declaring variables in the .data section and loading them into registers for arithmetic:

/* Data section */
.data
.balign 4
myvar1: .word 3      @ 4-byte integer with value 3
.balign 4
myvar2: .word 4      @ 4-byte integer with value 4

/* Code section */
.text
.balign 4
.global main
main:
    ldr r1, =myvar1  @ r1 ← address of myvar1 (shortcut)
    ldr r1, [r1]     @ r1 ← *r1 (dereference — get the value)
    ldr r2, =myvar2
    ldr r2, [r2]
    add r0, r1, r2   @ r0 = r1 + r2
    bx lr

You can't directly access data from a different section for security/efficiency reasons. ldr r1, =myvar1 loads the address. ldr r1, [r1] dereferences it to get the value. The linker resolves the actual memory location.

Storing values with MOV and STR:

ldr r1, =myvar1    @ r1 ← address of myvar1
mov r3, #3         @ r3 ← 3
str r3, [r1]       @ *r1 ← r3  (write 3 into myvar1)

ldr r2, =myvar2
mov r3, #4
str r3, [r2]       @ *r2 ← r3  (write 4 into myvar2)

Note: STR operand order is source first, destination second — the opposite of what you might expect from LDR.

Branching & Conditional Logic

▶

Normally the PC (Program Counter — register R15) advances by 4 bytes after each instruction. Branches change this to jump to a label.

Unconditional branch:

.global main
main:
    mov r0, #2      @ r0 ← 2
    b end           @ jump to 'end' — skips next instruction
    mov r0, #3      @ this is never executed
end:
    bx lr

Conditional branches use the CPSR (Current Program Status Register) — a special register with four condition flags:

FlagMeaning
NNegative — result was negative
ZZero — result was zero
CCarry — result produced a carry out
VOverflow — result overflowed signed range

Common branch conditions: EQ (Z=1), NE (Z=0), GE, LT, GT, LE, MI (negative), PL (positive/zero), VS (overflow), VC (no overflow), HI, LS, CS/HS, CC/LO.

/* IF r1 == r2 THEN r0=1 ELSE r0=2 */
    mov r1, #2
    mov r2, #2
    cmp r1, r2        @ update CPSR with r1 - r2
    beq case_equal    @ branch if Z=1 (they're equal)
case_different:
    mov r0, #2
    b end
case_equal:
    mov r0, #1
end:
    bx lr

IF/THEN/ELSE template: evaluate condition (updates CPSR) → conditional branch to else → then_part code → unconditional branch to end → else_part code → end label. Every high-level control structure maps to this pattern.

Loops & Control Structures

▶

While loop template:

while_condition:
    /* evaluate E and update CPSR */
    bXX end_of_loop   @ if E is false, leave loop
    /* body of loop */
    b while_condition  @ unconditional branch back
end_of_loop:

For loop — sum of first 22 numbers:

.text
.global main
main:
    mov r1, #0       @ r1 ← 0  (accumulator)
    mov r2, #1       @ r2 ← 1  (counter)
loop:
    cmp r2, #22      @ compare r2 and 22
    bgt end          @ if r2 > 22, exit loop
    add r1, r1, r2   @ r1 ← r1 + r2
    add r2, r2, #1   @ r2 ← r2 + 1
    b loop
end:
    mov r0, r1       @ r0 ← r1  (return result = 253)
    bx lr

You only truly need IF and WHILE — every other control structure (for, do-while, switch) can be derived from these two. The for loop is just a while with an explicit initializer and increment.

Shift operations for fast multiply/divide:

mov r1, r2, lsl #1   @ r1 ← r2 * 2  (shift left 1)
mov r1, r2, lsl #2   @ r1 ← r2 * 4  (shift left 2)
mov r1, r3, asr #3   @ r1 ← r3 / 8  (arithmetic shift right)
add r1, r2, r2, lsl #1  @ r1 ← r2 + r2*2 = r2*3
add r1, r2, r2, lsl #2  @ r1 ← r2 + r2*4 = r2*5

Arrays & Indexing Modes

▶

Arrays require knowing element size, element count, and total byte size. Assembly provides no built-in array type — you manage it manually.

/* Array of 100 integers = 100 * 4 = 400 bytes */
.data
a: .skip 400        @ reserve 400 bytes

.text
.global main
main:
    ldr r1, =a      @ r1 ← &a  (address of array)
    mov r2, #0      @ r2 ← 0  (index)
loop:
    cmp r2, #100
    beq end
    add r3, r1, r2, lsl #2  @ r3 ← r1 + (r2*4)  — element address
    str r2, [r3]             @ *r3 ← r2  — store index value
    add r2, r2, #1
    b loop
end:
    bx lr

.skip 400 reserves 400 bytes without initializing them. The expression r2, lsl #2 multiplies the index by 4 (since each int is 4 bytes) inline — this is the ARM "flexible second operand" shift used in address calculation.

Three non-updating index modes:

str r2, [r1, #12]       @ *(r1 + 12) ← r2  (immediate offset)
str r2, [r1, +r3]       @ *(r1 + r3) ← r2  (register offset)
str r2, [r1, +r2, lsl #2]  @ *(r1 + r2*4) ← r2  (shift offset)

Updating index mode (more efficient for sequential access):

str r2, [r1], #4    @ *r1 ← r2, THEN r1 ← r1 + 4

The updating mode combines the store and the pointer advance into one instruction — saving both a register and an ADD instruction compared to the non-updating approach.

Functions & the Stack

▶

Functions in assembly require careful management of the stack, PC, and Link Register (LR).

PC (R15)

Program Counter — address of the currently executing instruction. Auto-increments by 4 each cycle unless modified.

LR (Link Register)

Stores the return address when BL (Branch with Link) is called. Must be saved before any nested function calls.

R0–R3

Parameter and return value registers. No assumptions about their values unless explicitly set as parameters.

R4+ (Saved Registers)

If a function modifies these, it MUST save the original values on the stack and restore them before returning.

Function rules:

  • Use R0–R3 for parameters — make no assumptions about their values
  • Make no assumptions about the CPSR on entry
  • Save LR on entry, restore before exit
  • Save and restore any R4+ registers you modify
  • Exit via bx lr — or pop {pc} if you pushed LR

Using C-helper functions (printf, scanf, puts) from assembly is possible but requires careful stack alignment. After a C call, push/pop the stack pointer to re-align it — C functions expect 8-byte alignment and will corrupt your stack if it's off.

Using GDB (GNU Debugger)

▶

GDB (GNU Debugger) is part of the Free Software Foundation's open-source ecosystem — the same movement that produced Linux, Python, MySQL, Eclipse, and thousands of other tools. It lets you inspect and modify register values during execution.

Launch: gdb <program> or load inside gdb with file <program>

CommandWhat It Does
startBegin execution and pause at first instruction
quitExit the debugger
disassembleShow machine code with addresses (note 4-byte separation)
info registersShow all register values in hex and decimal
stepiExecute one instruction and pause
p $r0 = 2Print or modify a register value
continueRun to end (or next breakpoint)
break <label>Set breakpoint at a label, function, or address

Assembly is much harder to debug than high-level code. Use GDB heavily — especially stepi to trace through one instruction at a time and info registers after each step to verify values are what you expect. One trick: write the program in C first, run it through the assembler, and study the generated assembly to understand how the compiler handles constructs like loops and function calls.

Structures & Advanced Topics

▶

Unlike arrays (uniform elements), structures have irregular shapes — multiple fields of different types. In C: struct { char f0; int f1; } = 5 bytes logically, but 8 bytes with alignment padding.

/* In assembly — 8 bytes not 5, due to 4-byte alignment */
.data
b: .skip 8    @ char field = 4 bytes padded, int field = 4 bytes

Padding bytes are never referenced — they exist purely to keep fields on 4-byte boundaries. Referencing partial addresses creates extra work for the hardware and can cause alignment faults on some architectures.

Indexing a structure array: Each structure instance is 8 bytes, so moving to the next element requires shifting the address right by 3 bits (8 = 2³). Managing structure arrays manually is one of the most complex tasks in assembly — high-level languages do all of this invisibly.

Collatz Conjecture (advanced exercise): If n is even, divide by 2. If n is odd, multiply by 3 and add 1. Repeat — always reaches 1 for any starting n. Good exercise for combining conditional branches, shifts (divide by 2 = ASR #1), and loops.

Final assignment — Bubble Sort: Input 8 single-digit numbers, sort ascending, print results. Tips: align all data elements, pre-allocate strings, allocate array space with .skip, use printf/scanf for I/O, pseudo-code in a high-level language first, focus on the core sort routine before worrying about I/O.
