Recursive Definitions

▶

A recursive definition defines something in terms of itself — with a formal basis ensuring no circular violation occurs. Particularly useful for defining infinite sets, since exhaustive enumeration is impossible.

Every recursive definition has exactly two parts:

Base Case (Anchor/Ground)

Establishes the basis for all other elements. Provides a concrete result that stops the recursion. Without it: infinite recursion.

Inductive Clause

Rules for creating new elements from existing ones. Reduces the problem toward the base case with each application.

// Natural numbers defined recursively:
// 0 ∈ N                        (base case)
// if n ∈ N, then (n+1) ∈ N    (inductive clause)

// Factorial defined recursively:
// 0! = 1                       (base case)
// n! = n × (n-1)!              (inductive clause)

// 3! = 3 × 2! = 3 × 2 × 1! = 3 × 2 × 1 × 0! = 3 × 2 × 1 × 1 = 6

unsigned int factorial(unsigned int n) {
    if (n == 0)
        return 1;               // base case — stops recursion
    else
        return n * factorial(n - 1);  // inductive clause
}

// x^n defined recursively:
// x^0 = 1                     (base case)
// x^n = x × x^(n-1)           (inductive clause)

double power(double x, unsigned int n) {
    if (n == 0)
        return 1.0;
    else
        return x * power(x, n - 1);
}

Recursive code is often simpler and more faithful to the mathematical definition — but simplicity can hide underlying inefficiency. The translation from recursive specification to code is usually straightforward. The hard part is understanding what happens under the hood (the runtime stack) and recognizing when recursion is problematic.

Function Calls & Recursive Implementation

▶

Every function call requires tracking: parameter values, local variables, return address, and where to resume. This information is stored in an activation record (stack frame) pushed onto the runtime stack.

Activation record contains:

  • Values of the function's parameters
  • Addresses of reference parameters (including arrays)
  • Local variables
  • Return address in the calling function
  • Dynamic link to the calling function's activation record
  • The function's return value (if non-void)
// When main() calls f1(), which calls f2(), which calls f3():
// Runtime stack (top to bottom):
// [f3 activation record]  ← currently executing
// [f2 activation record]  ← suspended, waiting for f3
// [f1 activation record]  ← suspended, waiting for f2
// [main activation record] ← suspended, waiting for f1

// When f3() returns:
// f3 record popped → f2 resumes from its saved return address

// RECURSION: a function calling itself pushes a NEW activation record
// Each recursive call = a new independent instance with its own state

The runtime stack is WHY recursion works — each call gets its own activation record with its own copy of local variables and parameters. This is what prevents the calls from interfering with each other. A recursive call simply suspends the current instance and starts a new one. When the new instance returns, the old one resumes exactly where it left off.

Anatomy of a Recursive Call

▶

Tracing power(5.6, 2) step by step through the runtime stack:

// Call: y = power(5.6, 2)  — from line 136

// Stack evolution:
// 1. power(5.6, 2) pushed. n != 0, so calls power(5.6, 1)
// 2. power(5.6, 1) pushed. n != 0, so calls power(5.6, 0)
// 3. power(5.6, 0) pushed. n == 0, RETURNS 1.0

// Unwinding:
// power(5.6, 0) → returns 1.0, popped
// power(5.6, 1) → resumes: 5.6 × 1.0 = 5.6, returns 5.6, popped
// power(5.6, 2) → resumes: 5.6 × 5.6 = 31.36, returns 31.36, popped
// y = 31.36

// Pattern: wind DOWN to base case, then UNWIND back up
// x^4 unwinds as:
// x^4 = x·x^3 = x·x·x^2 = x·x·x·x^1 = x·x·x·x·x^0 = x·x·x·x·1

All recursive algorithms have the same two-phase structure: the winding phase (recursive calls reducing the problem toward the base case) and the unwinding phase (returning and combining results back up the call chain). Understanding this wind/unwind pattern is key to reading and writing recursive code correctly.

Tail Recursion

▶

Tail recursion: the recursive call is the LAST statement in the function — nothing happens after it returns. No other statements follow the call, and no processing of the return value occurs.

// Tail recursive — recursive call is the final action
void tail(int i) {
    if (i > 0) {
        cout << i << " ";
        tail(i - 1);    // last statement — nothing happens after this
    }
}

// Equivalent iterative version — tail recursion is just a loop
void iterative(int i) {
    for (; i > 0; i--)
        cout << i << " ";
}

Tail recursion is exactly equivalent to a loop — each call simply updates the parameter and repeats. Modern compilers can often optimize tail recursion into iteration (tail call optimization/TCO), eliminating stack growth. However, in C++, this is not guaranteed. Unless you're writing functional/logic code (Prolog, Haskell), convert tail recursion to a loop for guaranteed performance and no stack overflow risk.

Nontail Recursion

▶

Nontail recursion: work happens AFTER the recursive call returns. The function must wait for the recursive call to complete, then do additional processing. This is the "real" power of recursion.

// Reverses a line of input character by character
void reverse() {
    char ch;
    cin.get(ch);              // read a character
    if (ch != '\n') {
        reverse();            // recursive call FIRST
        cout.put(ch);         // THEN output — happens AFTER return!
    }
}
// Input "ABC" → outputs "CBA"
// The stack builds up: A, B, C are read going DOWN
// On the way back UP, C is printed, then B, then A

// Equivalent iterative version — must explicitly use a stack!
void iterativeReverse() {
    char stack[80];
    int top = 0;
    cin.get(stack[top]);
    while (stack[top] != '\n')
        cin.get(stack[++top]);
    for (top -= 2; top >= 0; )
        cout.put(stack[top--]);
}

To convert nontail recursion to iterative, you MUST explicitly implement a stack — the runtime stack can't be eliminated here because post-call processing depends on it. This usually results in longer, less clear code. Key lesson: if converting nontail recursion iteratively doesn't provide a clear performance benefit, keep the recursive version. Clarity and correctness are worth more than micro-optimization in most contexts.

Indirect & Nested Recursion

▶

Direct Recursion

A function calls itself directly: f() → f(). The standard case we've discussed throughout this chapter.

Indirect Recursion

A function calls another function that eventually calls back: f() → g() → f(). Chains can be arbitrarily long: f() → f1() → f2() → ... → fn() → f().

Nested Recursion

A function calls itself AND that call is one of the parameters: f(f(n)). The Ackermann function is the famous example.

Ackermann Function

A(3,m) = 2^(m+3) - 3. A(4,2) has 19,729 digits in its expansion. Grows faster than any primitive recursive function. Elegantly recursive, impossible to define iteratively.

// Indirect recursion example
bool isEven(int n) {
    if (n == 0) return true;
    return isOdd(n - 1);   // calls isOdd
}
bool isOdd(int n) {
    if (n == 0) return false;
    return isEven(n - 1);  // calls isEven — cycle!
}
// isEven → isOdd → isEven → isOdd → ... → base case

// Ackermann function (nested recursion)
int A(int m, int n) {
    if (m == 0) return n + 1;
    if (n == 0) return A(m - 1, 1);
    return A(m - 1, A(m, n - 1));   // A() is a parameter of A()!
}

Excessive Recursion — Fibonacci

▶

Recursion becomes excessive when the same subproblems are solved repeatedly. The Fibonacci sequence is the classic example.

// Naive recursive Fibonacci — deceptively simple, catastrophically slow
unsigned long Fib(unsigned long n) {
    if (n < 2)
        return n;        // base cases: Fib(0)=0, Fib(1)=1
    return Fib(n-2) + Fib(n-1);  // two recursive calls!
}

// Computing Fib(6):
// Fib(6) calls Fib(4) AND Fib(5)
// Fib(5) calls Fib(3) AND Fib(4) — Fib(4) computed AGAIN!
// Total calls to compute Fib(6): 25
// Total calls for Fib(n): 2×Fib(n+1) - 1 (exponential!)
nFib(n)Recursive calls

311,346,269~2.9 million

The problem: no intermediate values are remembered — every Fib(k) is recomputed from scratch each time it's needed. Drawing the call tree (Fig 5.8) reveals massive redundancy. Fix: Dynamic Programming — memoize previously computed values. Or use Binet's formula (closed form). Either approach reduces exponential time to linear or constant time.

// Iterative Fibonacci — O(n) time, O(1) space
unsigned long iterFib(unsigned long n) {
    if (n < 2) return n;
    unsigned long a = 0, b = 1;
    for (unsigned long i = 2; i <= n; i++) {
        unsigned long c = a + b;
        a = b;
        b = c;
    }
    return b;
}

// Binet's formula — O(1) time (for small n)
unsigned long deMoivreFib(unsigned long n) {
    return ceil(exp(n * log(1.6180339897) - log(2.2360679775)) - 0.5);
}
// φ = (1+√5)/2 ≈ 1.6180339897 (golden ratio)

Backtracking

▶

Backtracking is a systematic search approach that explores possible solution pathways. When a path fails, it backs up to the last decision point and tries a different direction — guaranteed to explore all possibilities.

Used in: AI search algorithms, constraint satisfaction problems, compilers (syntax parsing), and optimization. The key requirement: the algorithm must be able to undo its last decision and try alternatives. Recursion makes this natural — returning from a recursive call IS the backtrack.

Classic example: Eight Queens Problem — place 8 queens on a chessboard so no two attack each other.

// Backtracking pseudocode — Eight Queens
putQueen(row):
    for every position col in this row:
        if position (row, col) is safe (no conflicts):
            place queen at (row, col)
            if row < 8:
                putQueen(row + 1)  // try to place next queen
            else:
                success!           // all 8 queens placed
            remove queen from (row, col)  // BACKTRACK — try next col

// C++ structure
bool isSafe(int row, int col); // check no row/col/diagonal conflicts

void putQueen(int board[][8], int row) {
    for (int col = 0; col < 8; col++) {
        if (isSafe(board, row, col)) {
            board[row][col] = 1;      // place queen
            if (row == 7)
                printSolution(board); // found a solution
            else
                putQueen(board, row + 1); // recurse to next row
            board[row][col] = 0;      // remove queen (backtrack)
        }
    }
}

The backtrack step (remove queen from position) is what makes this work — it undoes the last placement so the algorithm can try the next column. Without the undo, there's no backtracking, just forward search. This algorithm finds ALL 92 solutions to the 8-queens problem (some are symmetrical). Backtracking is only practical when the solution space can be pruned aggressively — pure brute force would examine 8^8 = 16M positions.

When to Use Recursion — Guidelines

▶
SituationRecommendation
Tail recursionConvert to loop — tail recursion is just a loop, iterative is clearer and guaranteed stack-safe
Nontail recursion, small dataKeep recursive — clarity and correctness outweigh minor performance cost
Nontail recursion, large dataConsider iterative — explicitly implement a stack if needed
Excessive repeated subproblemsUse Dynamic Programming (memoization) or a closed-form formula
Real-time systems, Big DataPrefer iterative — stringent time requirements demand predictable performance
Backtracking / searchRecursion is natural and usually the right choice
Unclear which is betterImplement both and benchmark on realistic data

The call tree diagnostic: draw the recursive call tree. Deep, narrow trees → stack overflow risk, use iterative. Shallow, wide ("bushy") trees with no repeated nodes → recursion is probably fine. Trees with repeated identical subtrees (like Fibonacci) → dynamic programming. When in doubt, the recursive version is usually clearer and easier to verify correct — optimize only when profiling shows a real problem.
