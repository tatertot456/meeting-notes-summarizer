Introduction & Notation

▶

String matching (stringology) finds patterns within text. The larger the text, the more critical the algorithm's efficiency. Two major categories: exact matching (find identical copies) and approximate matching (find similar substrings).

Notation

|T| = length of text T. Tj = character at position j. T(i...j) = substring from i to j. Positions start at 0. Pattern P is searched for in text T.

Regular expression aⁿ

Represents a string of n copies of character a: "aaa...a" (n times). Used in pattern specification.

Brute Force & Hancart's Algorithm

▶

Compare P and T from their beginnings. On mismatch: shift P one position right, restart. Discards all information from previous attempts.

// Brute force
for i = 0 to |T| - |P|:
    j = 0
    while j < |P| and P[j] == T[i+j]:
        j++
    if j == |P|: return i  // match at position i
// Worst case O(|T|·|P|) — e.g., T = aaa...a, P = aaa...ab

Hancart's Algorithm (1992) — compares P starting at P[1] (second character), ends with P[0]. Achieves shifts of two positions in some cases:

  • If P[0] == P[1]: shift by 2 when P[1] ≠ T[i+1] (since P[0] would also differ)
  • If P[0] ≠ P[1]: shift by 2 after inner loop mismatch (since P[1] == T[i+1] means P[0] ≠ T[i+1])

Brute force is O(|T|·|P|) worst case. Hancart performs better on average than some sophisticated algorithms despite similar worst case. The key problem with brute force: it throws away useful information every time it restarts — we've already compared some characters successfully but we ignore that knowledge.

Knuth-Morris-Pratt (KMP) Algorithm

▶

KMP avoids redundant comparisons by precomputing how far to shift P when a mismatch occurs, based on P's own structure. The text index i is NEVER decremented — each character of T is processed at most once.

// next[j] = length of longest proper suffix of P(0...j-1) // that equals a prefix of P // next[0] = -1 (sentinel: shift entire pattern past mismatch) // Example: P = abababa // next = [-1, 0, 0, 1, 2, 3, 4] // Interpretation: if mismatch at position j, shift P by (j - next[j]) positions // and restart comparison at P[next[j]] // KMP matching: i = j = 0 while i < |T|: if j == -1 or T[i] == P[j]: i++; j++ else: j = next[j] // DON'T increment i — rethink alignment of P if j == |P|: output match at i - |P| j = next[j] // find next possible match // Building next table (KMP applied to P itself): next[0] = -1; i = 1; j = 0 while i < |P|: if j == -1 or P[i-1] == P[j]: next[i] = j + 1; i++; j++ else: j = next[j]

nextS improvement — eliminates redundant comparisons when P[j] == P[next[j]] (same character would mismatch again):

// If P[i] == P[j], use nextS[j] instead of j
// This skips known-futile comparisons
// For P = abababa: nextS = [-1, 0, -1, 0, -1, 0, -1]
// (compared to next = [-1, 0, 0, 1, 2, 3, 4])

KMP runs in O(|T| + |P|) time. The preprocessing (building next) is O(|P|). The key insight: when a mismatch at P[j] occurs after matching P(0...j-1), we know P(0...j-1) was matched. The next table tells us the longest prefix of P that is also a suffix of the matched portion — we can restart from there without re-examining T.

Boyer-Moore Algorithm & Sunday's Variants

▶

Boyer-Moore (BoM) compares P with T right-to-left and skips characters in T (not P) when mismatches occur. Typically faster than KMP in practice — can skip large portions of T.

Two tables for shifting:

delta1 — Character Occurrence

Based on the mismatched character Ti. Three rules: No occurrence → shift P past Ti. Right-side occurrence → shift 1 position. Left-side occurrence → align Ti with closest match in P.

delta2 — Suffix Occurrence

Based on the suffix of P following the mismatch. Full suffix rule: align matched suffix with its occurrence in P (preceded by different char). Partial suffix rule: align with prefix of P.

// delta1 for P = dabacbd:
// delta1['a'] = 3, delta1['b'] = 1, delta1['c'] = 2, delta1['d'] = 0
// All other chars: delta1[ch] = |P| = 7

// delta2 handles the suffix after the mismatch position
// Takes the larger shift from delta1 and delta2

// On each mismatch at P[j] against T[i]:
// shift = max(delta1[T[i]] - (|P| - 1 - j), delta2[j])

// SUNDAY'S QUICKSEARCH (1990) — Left-to-right scan, simpler rules:
// After mismatch, look at character T[i + |P|] (just past current window)
// delta1[ch] = position of RIGHTMOST ch in P, counting from right + 1
// No occurrence: shift P past T[i+|P|] entirely
// Occurrence: align T[i+|P|] with rightmost ch in P

Boyer-Moore-Galil — for multiple occurrences. After first match, shifts by the PERIOD of P (length of shortest repeating unit) and only re-checks the last p characters. Avoids redundant rechecking of overlapping occurrences.

Boyer-Moore is typically the fastest exact-matching algorithm for long patterns and large alphabets. It can skip entire blocks of text — sublinear average case. Sunday's quicksearch() is simpler to implement and often comparable in speed, making it sufficient for most practical uses. The delta2 computation using auxiliary table f runs in O(|P|) instead of the naive O(|P|²) or O(|P|³).

Shift-And (Bit-Oriented) Algorithm

▶

Baeza-Yates & Gonnet (1992) — represents matching state as a bit vector. All prefix matches updated simultaneously using bitwise operations. No text buffering required.

// charactersInP[ch] = bit string where bit j = 1 if P[j] == ch
// state = bit string where bit j = 1 means P(0...j) matches T ending at current position

// Transition:
// state[i] = shiftBits(state[i-1]) AND charactersInP[T[i]]
// shiftBits = left-shift + set bit 0 to 1 (give all prefixes a chance to start)

// Example: P = ababac, match at T position 7 (pattern length 6)
// On processing T[7]:
//   shiftBits(state[6]) shifts all partial matches "forward" by one
//   AND with charactersInP[T[7]] checks if the new character fits
//   If bit |P|-1 is set in result: MATCH FOUND

// PSEUDOCODE:
// Preprocess:
for each char ch: charactersInP[ch] = 0
for j = 0 to |P|-1: set bit j of charactersInP[P[j]]
state = 0
// Match:
for i = 0 to |T|-1:
    state = ((state << 1) | 1) & charactersInP[T[i]]
    if bit |P|-1 of state is set: output match at i-|P|+1

The pattern length must not exceed the word size (typically 64 bits for modern systems). A dynamic version handles longer patterns by splitting into multiple 64-bit chunks and propagating carry bits between chunks. This algorithm processes each text character exactly once with a constant number of bit operations — O(|T|·|P|/w) where w is word size, effectively O(|T|) for short patterns.

Multi-Pattern Matching — Aho-Corasick

▶

Find all occurrences of multiple patterns simultaneously in one pass through text. Used in UNIX fgrep. Builds a trie + failure function — analogous to KMP for multiple patterns.

// Three components:
// goto(state, ch): trie transitions (or fail)
// failure(state): KMP-like fallback on mismatch
// output(state): keywords that end at this state

// BUILD: Insert all keywords into a trie
// keywords = {inner, input, in, outer, output, out, put, outing, tint}
// Root loops back for all unrecognized chars (no fail from root)

// FAILURE FUNCTION (breadth-first construction):
// For each state s reachable via ch from parent r:
//   follow failure links from failure(r) until nonfail ch-transition found
//   f(s) = that nonfail destination
// This creates KMP-style fallback across ALL patterns simultaneously

// OUTPUT FUNCTION expansion:
// output(s) += output(failure(s))
// Captures keywords that are SUFFIXES of the current match

// MATCHING:
state = 0
for each char Ti:
    while goto(state, Ti) == fail:
        state = failure(state)
    state = goto(state, Ti)
    if output(state) not empty:
        report all words in output(state) ending at i

Aho-Corasick runs in O(|T| + total pattern length + number of matches) — optimal. The failure function handles cases where the current partial match fails: it finds the longest proper suffix of the current match that is also a prefix of some keyword, and continues from there. The output function expansion ensures that when "input" is matched, "in" and "put" (if they're substrings) are also reported.

Regular Expression Matching (Thompson, 1968)

▶

Regular expressions specify patterns: r|s (r or s), rs (concatenation), r* (zero or more r), (r) (grouping). Thompson's algorithm builds a Non-Deterministic Finite Automaton (NDFA) from the regex.

// Regular expression components → NDFA fragments:
// Single char c:  ○ →(c)→ ◎
// r|s:  ○ →(ε)→ [r-NDFA] →(ε)→ ◎
//       ↘(ε)→ [s-NDFA] →(ε)→ ↗
// rs:   [r-NDFA] → [s-NDFA]
// r*:   ○ →(ε)→ [r-NDFA] →(ε)→ ↺ (loop back) and skip edge

// MATCHING with epsilon-closure:
// epsilon(S) = all states reachable from S via ε-edges
// goto(S, ch) = all states reachable from S via ch-edge

// Algorithm:
states = epsilon({initState})
for each Ti:
    states = epsilon(goto(states, Ti))
    if acceptState in states: record match

// For T = aabbcdeffaefc and regex = a(b|cd)*ef:
// Matches: T(0...6) = abbcdef, T(8...10) = aef

Suffix Tries, Suffix Trees & Suffix Arrays

▶

Preprocess the text once to enable efficient repeated queries.

Suffix Trie

Trie containing all suffixes of T. Each root-to-leaf path = one suffix. Space: O(|T|²) — quadratic due to shared node elimination. Simple to build but impractical for large texts.

Suffix Tree

Compressed suffix trie — merge chains of single-descendant nodes into single edges labeled with substrings (stored as index pairs). Space: O(|T|). Ukkonen's algorithm builds it in O(|T|) time using suffix links.

Suffix Array (Manber & Myers, 1993)

Array of suffix starting positions in lexicographic order. Only O(|T|) space. Pattern search via binary search: O(|P| log |T|). More cache-friendly than suffix trees.

Suffix Links

Link from suffix trie node representing string aX to node representing X. Used in Ukkonen's O(|T|) construction algorithm. Analogous to failure function in Aho-Corasick.

// Suffix array for T = "proposition": // Lexicographically sorted suffixes: // 8: "ion" // 6: "ition" // 10: "n" // 9: "on" // 2: "oposition" // 4: "osittion" // ... // pos = [8, 6, 10, 9, 2, 4, 3, 0, 1, 5, 7] // Search for pattern P using binary search on pos: // All occurrences of P are in a contiguous range of pos // Binary search finds left and right boundaries of that range // O(|P| log |T|) per query after O(|T|) or O(|T| log |T|) construction

Approximate String Matching — Edit Distance

▶

Exact matching is all-or-nothing. Approximate matching finds substrings within a similarity threshold using Levenshtein distance (Levenshtein, 1965) — minimum edit operations to transform one string into another.

Insertion (I)

Add a character to the string. Cost: 1. D(i,j) from D(i,j-1)+1.

Deletion (D)

Remove a character. Cost: 1. D(i,j) from D(i-1,j)+1.

Substitution (S)

Replace one character with another. Cost: 1 (if different), 0 (if same = match). D(i,j) from D(i-1,j-1)+c(i,j).

// Wagner-Fischer Algorithm (1974) — O(|Q|·|R|) dynamic programming
// D(i,j) = edit distance between Q(0...i-1) and R(0...j-1)

// Recurrence:
// D(i,j) = min(D(i-1,j) + 1,       // delete Qi
//              D(i,j-1) + 1,       // insert Rj
//              D(i-1,j-1) + c(i,j)) // substitute/match
// where c(i,j) = 0 if Qi == Rj, else 1

// Base cases: D(i,0) = i, D(0,j) = j

// Example: Q = "capital", R = "apple"
// Edit table (7×5):
//    ""  a  p  p  l  e
// "" [0  1  2  3  4  5]
// c  [1  1  2  3  4  5]
// a  [2  1  2  3  4  5]
// p  [3  2  1  2  3  4]
// i  [4  3  2  2  3  4]
// t  [5  4  3  3  3  4]
// a  [6  5  4  4  4  4]
// l  [7  6  5  5  4  5]
// D("capital","apple") = 5

// Longest Common Subsequence:
// d2(Q,R) = |Q| + |R| - 2·lcs(Q,R)
// lcs finds longest sequence of characters common to both strings
// (need not be consecutive)

The edit table is filled left-to-right, top-to-bottom. Each cell requires only its left, top, and top-left neighbors. To find an alignment (trace through edits), backtrack from the lower-right corner following the minimum path. The diff command in UNIX uses the longest common subsequence algorithm to show file differences.

Approximate Matching with k Errors (Shift-And Extension)

▶

Find all substrings of T matching P with at most k errors (insertions, deletions, substitutions). Wu & Manber's agrep (1992) extends shift-and for approximate matching.

// Use k+1 state tables: state0, state1, ..., statek
// statek[j] = 1 means P(0...j) matches a substring ending at Ti
//             with at most e errors

// Recurrence (for each error level e):
// statee,i+1 = (shiftBits(statee,i) AND charactersInP[Ti])   // exact match continues
//              OR shiftBits(statee-1,i)                        // insertion of Ti
//              OR shiftBits(statee-1,i+1)                      // deletion from P
//              OR statee-1,i                                   // substitution

// Each statee is derived from statee-1 using bitwise operations
// A match is reported when bit |P|-1 is set in statek

// PARTITION OPTIMIZATION for small k:
// Divide P into k+1 blocks of size r = |P|/(k+1)
// If P matches T with ≤k errors, at least one block matches EXACTLY
// Find exact block matches using Aho-Corasick (simultaneously)
// For each exact block match: search for approximate P match in neighborhood
// This is much faster than the basic algorithm when k << |P|
