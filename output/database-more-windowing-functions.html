What Are Windowing Functions?

▶

Introduced in SQL Server 2005 (SQL 2003 Standard), windowing functions perform operations on subsets of data without collapsing rows like GROUP BY does. Before windowing functions, these operations required liberal use of temp tables, CTEs, subqueries, and produced significant performance hits.

Ranking Functions (covered)

ROW_NUMBER, RANK, DENSE_RANK, NTILE — assign position numbers to rows within a window.

Aggregate Functions

SUM, AVG, COUNT applied over a window with ROWS/RANGE framing — running totals, moving averages.

Offset / Analytic Functions

LAG, LEAD, FIRST_VALUE, LAST_VALUE — access values from other rows without self-joins.

Distribution Functions

PERCENT_RANK, CUME_DIST, PERCENTILE_DISC, PERCENTILE_CONT — statistical positioning.

The core component of every windowing function is the OVER clause — it defines the "window" (the specific set of rows) the function operates on. Without OVER, there's no window function.

The OVER Clause & Partitioning

▶

OVER defines the window. PARTITION BY within OVER limits the window to rows sharing the same value in a partitioning column — similar to GROUP BY, but without collapsing rows.

-- Basic OVER: window is ALL rows, ordered by sales
SELECT BusinessEntityID AS SalesID,
    FirstName + ' ' + LastName AS FullName,
    SalesLastYear,
    ROW_NUMBER() OVER(ORDER BY SalesLastYear ASC) AS RowNumber
FROM Sales.vSalesPerson;

-- PARTITION BY: separate window per TerritoryGroup
-- Row numbers restart at 1 for each territory
SELECT BusinessEntityID AS SalesID,
    FirstName + ' ' + LastName AS FullName,
    SalesLastYear,
    TerritoryGroup,
    ROW_NUMBER() OVER(
        PARTITION BY TerritoryGroup
        ORDER BY SalesLastYear ASC) AS RowNumber
FROM Sales.vSalesPerson;

Partitioning is useful for isolating sets with specific characteristics without running a separate query per group. Each partition gets its own independent window — rank 1 in the North partition and rank 1 in the South partition are completely independent.

ROWS & RANGE Framing

▶

ROWS and RANGE further refine which rows within a partition are included in the calculation — based on proximity to the current row. Requires ORDER BY in the OVER clause.

BoundaryMeaningWorks With
CURRENT ROWThe current row being processedROWS and RANGE
UNBOUNDED PRECEDINGStart of the partition (first row)ROWS and RANGE
UNBOUNDED FOLLOWINGEnd of the partition (last row)ROWS and RANGE
n PRECEDINGn rows before the current rowROWS only

-- Running total: start of partition to current row
SUM(SalesLastYear) OVER(
    PARTITION BY TerritoryGroup
    ORDER BY SalesLastYear
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

-- Moving average of last 3 rows including current
AVG(SalesLastYear) OVER(
    ORDER BY SalesLastYear
    ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)

-- Total of entire partition (no frame needed)
SUM(SalesLastYear) OVER(PARTITION BY TerritoryGroup)

ROWS vs RANGE: ROWS treats each physical row individually — precise and predictable. RANGE treats rows with the same ORDER BY value as a group. For running totals, always prefer ROWS — it's more deterministic and generally faster. RANGE is useful when logical grouping of tied values matters.

Offset / Analytic Functions

▶

Offset functions access values from other rows relative to the current row — eliminating the need for self-joins. All four require ORDER BY in the OVER clause to be meaningful.

LAG(col, n, default)

Returns value from n rows BEFORE the current row. If no row exists at the offset, returns the default value (or NULL if not specified).

LEAD(col, n, default)

Returns value from n rows AFTER the current row. If no row exists at the offset, returns the default value (or NULL if not specified).

FIRST_VALUE(col)

Returns the value of the specified column for the FIRST row in the current window frame. Useful for comparing each row to the group's starting value.

LAST_VALUE(col)

Returns the value of the specified column for the LAST row in the current window frame. Requires careful framing — default frame stops at current row, so UNBOUNDED FOLLOWING is usually needed.

-- Compare each salesperson's sales to the previous year
SELECT BusinessEntityID AS SalesID,
    FirstName + ' ' + LastName AS FullName,
    SalesLastYear,
    LAG(SalesLastYear, 1, 0)  OVER(ORDER BY SalesLastYear) AS PrevSales,
    LEAD(SalesLastYear, 1, 0) OVER(ORDER BY SalesLastYear) AS NextSales,
    FIRST_VALUE(SalesLastYear) OVER(ORDER BY SalesLastYear) AS LowestSale,
    LAST_VALUE(SalesLastYear)  OVER(
        ORDER BY SalesLastYear
        ROWS BETWEEN UNBOUNDED PRECEDING
             AND UNBOUNDED FOLLOWING) AS HighestSale
FROM Sales.vSalesPerson;

LAST_VALUE gotcha: the default frame is ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — so LAST_VALUE with the default frame just returns the current row's value. Always specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING when you want the true last value in the partition.

Distribution Functions

▶

Distribution functions perform statistical analysis — showing where a row stands relative to the entire distribution. All require ORDER BY in the OVER clause.

PERCENT_RANK()

Percentile rank as a value between 0 and 1. Formula: (rank - 1) / (total rows - 1). First row = 0.0, last row = 1.0. Useful to see where a row "scores" relative to the group.

CUME_DIST()

Cumulative distribution — what fraction of rows have values ≤ the current row. Between 0 and 1, last row always = 1.0. Shows how far along the distribution the current value falls.

PERCENTILE_DISC(n)

Inverse distribution using a DISCRETE model — returns the first actual value in the dataset whose cumulative distribution is ≥ n. Returns a real data value.

PERCENTILE_CONT(n)

Inverse distribution using a CONTINUOUS model — interpolates between values. Can return a value that doesn't exist in the actual data. More statistically precise.

-- Where does each salesperson rank as a percentage?
SELECT BusinessEntityID AS SalesID,
    FirstName + ' ' + LastName AS FullName,
    SalesLastYear,
    PERCENT_RANK() OVER(ORDER BY SalesLastYear) AS PctRank,
    CUME_DIST()    OVER(ORDER BY SalesLastYear) AS CumeDist
FROM Sales.vSalesPerson;

-- What is the median (50th percentile) sales value?
-- PERCENTILE_DISC: returns an actual value from the data
-- PERCENTILE_CONT: may return an interpolated value
SELECT DISTINCT
    PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY SalesLastYear)
        OVER() AS MedianDisc,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY SalesLastYear)
        OVER() AS MedianCont
FROM Sales.vSalesPerson;

Key distinction: PERCENTILE_DISC returns a value that actually exists in your data (discrete). PERCENTILE_CONT may return an interpolated value between two actual data points (continuous). Use DISC when you need a real data point; use CONT for statistically precise percentile estimation.

Complete Syntax Reference

▶ 
-- Full windowing function syntax
<Window Function> OVER (
    [ PARTITION BY <partition_column> ]   -- optional
    [ ORDER BY <order_column> [ASC|DESC] ] -- required for most
    [ ROWS|RANGE BETWEEN <start> AND <end> ] -- optional
)

-- Start/end options:
-- UNBOUNDED PRECEDING  → start of partition
-- n PRECEDING          → n rows before current (ROWS only)
-- CURRENT ROW          → current row
-- n FOLLOWING          → n rows after current (ROWS only)
-- UNBOUNDED FOLLOWING  → end of partition

-- Complete example combining multiple window functions
SELECT
    BusinessEntityID AS SalesID,
    FirstName + ' ' + LastName AS FullName,
    TerritoryGroup,
    SalesLastYear,
    -- Ranking
    ROW_NUMBER()  OVER(PARTITION BY TerritoryGroup ORDER BY SalesLastYear DESC) AS RowNum,
    RANK()        OVER(PARTITION BY TerritoryGroup ORDER BY SalesLastYear DESC) AS Rnk,
    -- Aggregates
    SUM(SalesLastYear) OVER(PARTITION BY TerritoryGroup
                            ORDER BY SalesLastYear
                            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS RunningTotal,
    AVG(SalesLastYear) OVER(PARTITION BY TerritoryGroup) AS AvgInTerritory,
    -- Offset
    LAG(SalesLastYear, 1, 0) OVER(PARTITION BY TerritoryGroup ORDER BY SalesLastYear) AS PrevSales,
    -- Distribution
    PERCENT_RANK() OVER(ORDER BY SalesLastYear) AS PctRank
FROM Sales.vSalesPerson
ORDER BY TerritoryGroup, SalesLastYear DESC;

Multiple window functions can appear in a single SELECT — each with its own independent OVER clause, partition, and ordering. The query processes all of them in a single pass over the data, making window functions far more efficient than equivalent self-join or subquery approaches.
