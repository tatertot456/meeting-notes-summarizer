Debugging with Breakpoints

▶

Every language provides debugging tools. Python is no exception. The most important debugging tool is the breakpoint — it stops execution at a specific line of code so you can inspect the program's state.

Breakpoints in Spyder

Use the Debug menu to set breakpoints. Similar to Visual Studio and other IDEs. Click next to a line number to toggle a breakpoint.

Conditional Breakpoints

Breakpoints can be set with conditions — only pause execution when a specific condition is true (e.g., only when i == 5 in a loop).

IPython %debug

After an exception, run %debug in the console to enter the interactive debugger at the point of failure. Step through code, inspect variables.

print() Debugging

The classic approach — inserting print() statements to see variable values at different points. Simple but effective for quick checks.

Types of Errors

▶

Syntax Errors

Arise from incorrect code syntax. Python catches these BEFORE running — the program won't start. Fix: correct the syntax.

Runtime Exceptions

Arise during execution — from function calls, bad user input, division by zero, file not found, etc. Can be "caught" with try/except to prevent a crash.

# Syntax Error — caught before running
int(1 2)
# SyntaxError: invalid syntax

# Runtime Exception — happens WHILE running
int("1 2")
# ValueError: invalid literal for int() with base 10: '1 2'

If a runtime exception is NOT caught, it causes a "fatal" error and the program crashes (stops execution). The whole point of exception handling is to intercept these errors and respond gracefully — log the error, show a friendly message, or try a different approach — rather than letting the program blow up.

try / except

▶ 
# Catch a SPECIFIC exception
try:
    int("1 2")
except ValueError:
    print("1 2")   # handles only ValueError

# Catch ALL exceptions (generic)
# Good for beta testing — catches everything during development
a = 10
b = "zebra"
try:
    print(a + b)          # TypeError: can't add int and str
except Exception as e:
    print("Whoops, check yourself...\n", e)
    # "Whoops, check yourself...
    #  unsupported operand type(s) for +: 'int' and 'str'"

Catching specific exceptions (ValueError, ZeroDivisionError) is better practice than catching all exceptions — it prevents accidentally hiding bugs you didn't intend to catch. Use generic Exception catching (with "as e" to see the message) during development/testing to discover what exceptions your code throws, then narrow it down to specific types for production code.

Multiple except Blocks

▶

Different errors require different responses. Stack multiple except blocks — each handles a specific exception type.

try:
    num1 = int(input("Enter the first number: "))
    num2 = int(input("Enter the second number: "))
    total = num1 / num2
    print("The division is", total)
except ValueError:
    print("Please enter a valid integer.")
except ZeroDivisionError:
    print("Cannot divide by zero!")

Real scenario: opening a file could fail in multiple ways — file doesn't exist (FileNotFoundError), wrong format (ValueError), file is locked (PermissionError). Each needs a different response. Python checks except blocks in order from top to bottom and executes the first one that matches the exception type.

finally Block

▶

The finally block always executes — whether or not an exception occurred. Use it for cleanup code that must run regardless: closing files, releasing database connections, freeing resources.

try:
    print("inside try")
    num1 = int(input("Enter the first number: "))
    num2 = int(input("Enter the second number: "))
    total = num1 / num2
    print("The division is", total)
except ValueError:
    print("inside except — bad input")
finally:
    print("inside finally — ALWAYS runs")
    # close files, release connections here

If your try block throws an exception, normal execution is interrupted — code after the error in the try block doesn't run. Without finally, any cleanup code below the try/except might be skipped. The finally block guarantees cleanup happens no matter what — success, error, or even if the program is being shut down.

Full Structure: try / except / else / finally

▶

Python supports an optional else block that runs only when NO exception was raised. The full structure:

try:
    print("inside try")
    num1 = int(input("Enter the first number: "))
    num2 = int(input("Enter the second number: "))
    total = num1 / num2
    print("The division is", total)
except:
    print("inside except block")
else:
    print("inside else — no exception occurred")
finally:
    print("inside finally — always runs")

# The instructor's point: else is redundant.
# This is equivalent — just put the else code in try:
try:
    print("inside try")
    num1 = int(input("Enter the first number: "))
    num2 = int(input("Enter the second number: "))
    total = num1 / num2
    print("The division is", total)
    print("no exception — this only runs on success")
except:
    print("inside except block")
finally:
    print("inside finally — always runs")

Execution flow summary:

BlockWhen it runs
tryAlways — this is where your main code goes
exceptOnly when the matching exception is raised in try
elseOnly when NO exception was raised (optional, rarely needed)
finallyALWAYS — success, exception, or anything else

The else block runs only if no exception was raised — but you can just put that code at the end of the try block and get the same behavior. The instructor recommends skipping else entirely and using the simpler try/except/finally pattern. The key pattern to master: try the risky code, except specific errors with helpful messages, finally clean up.
