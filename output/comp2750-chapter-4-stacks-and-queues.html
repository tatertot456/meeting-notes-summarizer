Stacks — Concept & Operations

▶

A stack is a restricted access linear data structure — data can only be added and removed from one end (the top). Classic analogy: a stack of cafeteria trays. The last tray added is the first one removed.

LIFO — Last In, First Out. The most recently added element is always the first to be removed. This makes stacks naturally suited to any problem requiring reverse-order processing.

OperationDescription
clear()Remove all elements from the stack
isEmpty()Return true if stack has no elements
push(el)Add element el to the top of the stack
pop()Remove the top element
topEl()Return the top element's value WITHOUT removing it

Applications of stacks:

  • Evaluating expressions and parsing syntax (postfix/infix conversion)
  • Balancing delimiters in program code — {}, (), []
  • Converting numbers between bases
  • Processing financial data
  • AI search and backtracking algorithms
  • Function call management — the call stack itself is a stack

Stack Implementation

▶

Stacks can be implemented with either a vector/array or a linked list. Both give the same ADT behavior — the choice affects memory usage and performance characteristics.

// Stack as doubly linked list
template <class T>
class Stack {
    list<T> lst;
public:
    void clear()      { lst.clear(); }
    bool isEmpty()    { return lst.empty(); }

    void push(T el)   { lst.push_front(el); }  // add to front = top
    T    pop()        { T el = lst.front(); lst.pop_front(); return el; }
    T    topEl()      { return lst.front(); }   // peek without removing
};

// Stack as vector
template <class T>
class Stack {
    vector<T> vec;
public:
    void push(T el)  { vec.push_back(el); }    // back = top
    T    pop()       { T el = vec.back(); vec.pop_back(); return el; }
    T    topEl()     { return vec.back(); }
    bool isEmpty()   { return vec.empty(); }
};

Both implementations give O(1) push and pop. The linked list uses dynamic memory (pointer overhead per node). The vector may occasionally need to resize (O(n) amortized O(1)). For small, bounded stacks use vector. For large or unbounded stacks, a linked list avoids the occasional reallocation cost. The STL stack defaults to deque as its underlying container.

Queues — Concept & Operations

▶

A queue is a restricted access linear data structure where additions happen at the rear and deletions at the front. Items migrate from rear to front before removal.

FIFO — First In, First Out. Items are removed in the same order they were added. Classic analogy: a checkout line at a grocery store. The first person in line is the first to be served.

OperationDescription
clear()Remove all elements
isEmpty()Return true if queue has no elements
enqueue(el)Add element el to the REAR of the queue
dequeue()Remove the element from the FRONT
firstEl()Return the front element without removing it

Unlike a stack, both ends of the queue must be managed simultaneously. Queues appear throughout computer science: OS process scheduling, print spoolers, network packet buffering, breadth-first search, and service simulations (queuing theory).

Queue Implementation

▶

Array implementation — circular array:

As items dequeue, space opens at the front of the array. Rather than shifting everything or wasting that space, treat the array as circular — items can wrap around to the beginning.

// Circular array queue
template <class T>
class Queue {
    T*  arr;
    int capacity, front, rear, size;
public:
    Queue(int cap) : capacity(cap), front(0), rear(-1), size(0) {
        arr = new T[cap];
    }
    void enqueue(T el) {
        rear = (rear + 1) % capacity;  // wrap around!
        arr[rear] = el;
        size++;
    }
    T dequeue() {
        T el = arr[front];
        front = (front + 1) % capacity; // advance front
        size--;
        return el;
    }
    bool isEmpty() { return size == 0; }
    bool isFull()  { return size == capacity; }
};

The queue is full when front and last are adjacent — this can happen in two configurations depending on whether the last element is at the physical end or middle of the array.

Linked list implementation: more flexible, no fixed capacity, no wrapping logic needed.

// Queue as doubly linked list — much simpler
template <class T>
class Queue {
    list<T> lst;
public:
    void enqueue(T el) { lst.push_back(el); }   // add to rear
    T    dequeue()     { T el = lst.front(); lst.pop_front(); return el; }
    T    firstEl()     { return lst.front(); }
    bool isEmpty()     { return lst.empty(); }
};

The circular array approach is cache-friendly but requires a fixed maximum capacity. The linked list approach is flexible but has pointer overhead. The STL queue uses a deque under the hood by default — combining the benefits of both approaches.

Priority Queues

▶

A priority queue overrides normal FIFO ordering — elements are removed based on priority, not arrival order. Elements typically arrive randomly without inherent priority ordering.

Linked List Approach

Maintain sorted order by inserting new elements in priority order. O(n) insertion, O(1) removal. Empirically not useful beyond ~10 elements.

Heap Approach (Best)

A binary tree where parent always has higher priority than children. O(log n) insertion and removal. Used by the STL priority_queue.

Heap-based priority queue operations:

// Enqueue: insert at bottom, heapify UP
// 1. Add new node at the bottom of the heap
// 2. Compare with parent — if new node has higher priority, swap
// 3. Repeat until heap property restored

// Dequeue: remove root, heapify DOWN
// 1. Remove root (highest priority)
// 2. Move last node to root position
// 3. Compare with children — if lower priority than a child, swap with highest child
// 4. Repeat until heap property restored

// Both operations: O(log n) — the height of the heap

The heap ensures the highest priority element is always at the root — O(1) to access the top priority item, O(log n) to insert or remove. This is dramatically better than a sorted linked list (O(n) insertion) or unsorted array (O(n) removal). The heap is the standard implementation for priority queues in production code.

STL Stack

▶

The STL stack is a container adaptor — not a new container, but an adaptation of an existing one to enforce stack behavior. Default underlying container is deque.

#include <stack>

stack<int>              stack1;  // deque by default
stack<int, vector<int>> stack2;  // vector as underlying container
stack<int, list<int>>   stack3;  // list as underlying container

stack1.push(10);         // add to top
stack1.push(20);
stack1.push(30);

stack1.top();            // 30 — peek at top without removing
stack1.pop();            // removes 30 (returns void!)
stack1.top();            // now 20

stack1.empty();          // false
stack1.size();           // 2

STL's pop() does NOT return the removed value — it returns void. This is intentional (exception safety). To get the value before removing it, always call top() first, then pop(). This two-step idiom is standard: int val = s.top(); s.pop();

MethodDescription
push(el)Add element to top
pop()Remove top element (returns void!)
top()Return reference to top element
empty()True if stack is empty
size()Number of elements

STL Queue

▶

The STL queue is also a container adaptor. Default underlying container is deque; list may also be used.

#include <queue>

queue<int> q;

q.push(10);   // enqueue — adds to rear
q.push(20);
q.push(30);

q.front();    // 10 — peek at front (oldest element)
q.back();     // 30 — peek at rear (newest element)

q.pop();      // dequeue — removes front element (returns void!)
q.front();    // now 20

q.empty();    // false
q.size();     // 2

Like stack, queue's pop() returns void. Dequeue is a two-step operation: front() to get the value, pop() to remove it. Enqueue is push() — it goes to the back. front() and back() give you peek access to both ends without removal.

MethodDescription
push(el)Enqueue — add to rear
pop()Dequeue — remove from front (void)
front()Return reference to front element
back()Return reference to rear element
empty()True if queue is empty
size()Number of elements

STL Priority Queue

▶

The STL priority_queue uses a vector (or deque) as its underlying container. The highest-valued element is always at the top by default (max-heap).

#include <queue>

// Max-heap by default (highest value = highest priority)
priority_queue<int> pq;

pq.push(30);
pq.push(10);
pq.push(50);
pq.push(20);

pq.top();    // 50 — always the maximum
pq.pop();    // removes 50
pq.top();    // 30

// Min-heap (lowest value = highest priority)
priority_queue<int, vector<int>, greater<int>> minPQ;
minPQ.push(30);
minPQ.push(10);
minPQ.push(50);
minPQ.top();   // 10 — the minimum

// Custom comparator for objects
struct Task { int priority; string name; };
auto cmp = [](Task a, Task b) { return a.priority < b.priority; };
priority_queue<Task, vector<Task>, decltype(cmp)> taskQueue(cmp);

The default uses operator< so the LARGEST element has highest priority (max-heap). Use greater<T> as the third template argument to get a min-heap. The push() method internally re-heapifies to maintain the heap property — O(log n). top() is O(1), pop() is O(log n).

MethodDescription
push(el)Add element, reorder by priority
pop()Remove highest priority element (void)
top()Return highest priority element
empty()True if empty
size()Number of elements

STL Deque (Double-Ended Queue)

▶

A deque (double-ended queue) allows additions and deletions at BOTH ends. It combines list-like flexibility with vector-like random access.

#include <deque>

deque<int> dq;

// Add to either end
dq.push_front(10);    // add to front
dq.push_back(20);     // add to back
dq.push_front(5);     // [5, 10, 20]

// Remove from either end
dq.pop_front();       // removes 5 → [10, 20]
dq.pop_back();        // removes 20 → [10]

// Random access — like an array!
dq[0];                // direct access by index
dq.front();           // first element
dq.back();            // last element

dq.size();
dq.empty();

The STL deque's internal implementation is NOT a doubly linked list — it's an array of pointers to fixed-size blocks of data (like an ArrayList of arrays). This allows O(1) random access (via pointer math to find the right block and offset), O(1) push/pop at both ends, and avoids the contiguous reallocation cost of vector. The deque is the default underlying container for both stack and queue adaptors.

MethodDescription
push_front(el)Add to front
push_back(el)Add to back
pop_front()Remove from front
pop_back()Remove from back
front()Peek at front
back()Peek at back
operator[]Random access by index
at(i)Bounds-checked random access
