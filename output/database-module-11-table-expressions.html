What Are Table Expressions?

▶

Table expressions are named query expressions that return a virtual relational table. They allow more modular, readable, and maintainable SQL by encapsulating complex logic behind a clean name.

Views

Named table expressions stored permanently in the database. Reusable across sessions and applications. Can provide security by exposing only specific columns/rows.

Inline TVFs

Like parameterized views — stored in database, accept input parameters, return a table. The only table expression type that supports parameters.

Derived Tables

Defined inline within a FROM clause. Exist only for the duration of the query. Not stored in database. Scope is limited to the single outer query.

CTEs (Common Table Expressions)

Named table expressions defined in a WITH clause above the main query. Support multiple definitions, multiple references, and recursion. More readable than nested derived tables.

Views

▶

Views are named table expressions with definitions stored permanently in the database. They behave like tables in SELECT statements but retrieve data live from underlying base tables each time they are queried.

-- Create a simple view
CREATE VIEW HR.EmpPhoneList
AS
SELECT empid, lastname, firstname, phone
FROM HR.Employees;

-- Query the view like a table
SELECT empid, lastname, firstname, phone
FROM HR.EmpPhoneList
ORDER BY lastname;  -- ORDER BY goes in the outer query, not the view

ORDER BY is NOT permitted in a view definition without TOP, OFFSET/FETCH, or FOR XML. A view is a set — sets have no order. Add ORDER BY in the outer query that references the view. Views provide encapsulation, simplification, and security (users can be granted access to the view without direct table permissions).

Inline Table-Valued Functions (TVFs)

▶

Inline TVFs are stored in the database like views, but they accept input parameters — making them "parameterized views." They return a virtual table to the calling query.

-- Create an inline TVF
CREATE FUNCTION Sales.fn_LineTotal (@orderid INT)
RETURNS TABLE
AS
RETURN
    SELECT orderid,
        CAST((qty * unitprice * (1 - discount)) AS DECIMAL(8,2)) AS line_total
    FROM Sales.OrderDetails
    WHERE orderid = @orderid;

-- Call the TVF: use two-part name, pass parameter
SELECT orderid, line_total
FROM Sales.fn_LineTotal(10252) AS LT;

-- TVF in a join: get line totals for all orders by a specific customer
SELECT o.orderid, o.custid, LT.line_total
FROM Sales.Orders AS o
CROSS APPLY Sales.fn_LineTotal(o.orderid) AS LT
WHERE o.custid = 1;

SQL Server supports two types of TVFs: Inline TVFs (single SELECT statement, treated like a parameterized view — better performance) and Multi-statement TVFs (create and populate a table variable, then return it — more flexible but generally slower). Prefer inline TVFs when possible.

Derived Tables

▶

Derived tables are named query expressions created within the FROM clause of an outer SELECT. They are not stored in the database — they exist only for the duration of the query and are unpacked into the underlying query when processed.

-- Basic derived table: count distinct customers per year
SELECT orderyear, COUNT(DISTINCT custid) AS cust_count
FROM (
    SELECT YEAR(orderdate) AS orderyear, custid
    FROM Sales.Orders
) AS derived_year          -- alias is REQUIRED
GROUP BY orderyear;

-- External column aliases (defined in alias list)
SELECT orderyear, COUNT(DISTINCT custid) AS cust_count
FROM (
    SELECT YEAR(orderdate), custid
    FROM Sales.Orders
) AS derived_year(orderyear, custid)  -- column names defined here
GROUP BY orderyear;

-- Derived table with variable argument
DECLARE @emp_id INT = 9;
SELECT orderyear, COUNT(DISTINCT custid) AS cust_count
FROM (
    SELECT YEAR(orderdate) AS orderyear, custid
    FROM Sales.Orders
    WHERE empid = @emp_id   -- pass variable into derived table
) AS derived_year
GROUP BY orderyear;

Derived table rules: MUST have an alias, MUST have named columns (all of them), column names must be unique, ORDER BY is not permitted without TOP/OFFSET-FETCH, and cannot be referenced more than once in the same query. Each additional reference must be separately defined — which is why CTEs are often preferred.

Common Table Expressions (CTEs)

▶

CTEs are named table expressions defined in a WITH clause before the main query. They are similar to derived tables in scope and naming requirements, but more powerful — supporting multiple definitions, multiple references, and recursion.

-- Basic CTE
WITH CTE_year AS (
    SELECT YEAR(orderdate) AS orderyear, custid
    FROM Sales.Orders
)
SELECT orderyear, COUNT(DISTINCT custid) AS cust_count
FROM CTE_year
GROUP BY orderyear;

-- Multiple CTEs in one query
WITH CTE_orders AS (
    SELECT custid, COUNT(*) AS order_count
    FROM Sales.Orders
    GROUP BY custid
),
CTE_customers AS (
    SELECT custid, companyname, country
    FROM Sales.Customers
    WHERE country = N'USA'
)
SELECT c.custid, c.companyname, o.order_count
FROM CTE_customers AS c
JOIN CTE_orders AS o ON c.custid = o.custid
WHERE o.order_count > 5;

-- CTE referenced multiple times (not possible with derived table)
WITH CTE_avg AS (
    SELECT AVG(freight) AS avg_freight
    FROM Sales.Orders
)
SELECT orderid, freight,
    freight - (SELECT avg_freight FROM CTE_avg) AS diff
FROM Sales.Orders
WHERE freight > (SELECT avg_freight FROM CTE_avg);

CTEs are the preferred alternative to nested derived tables. They are more readable, support multiple references to the same expression, and support recursive queries (for hierarchical data like org charts or bill-of-materials). Unlike views and TVFs, CTEs are not stored in the database — they exist only for the statement they precede.

Decision guide: need it permanently and reused across queries → View. Need parameters → Inline TVF. Need it once in a complex query → CTE (preferred over derived table for readability). Avoid nested derived tables — use multiple CTEs instead. CTEs dramatically improve query readability by naming each logical step of a complex computation.
