Introduction & Course Overview

▶

Computer Architecture takes you on a tour of the lesser-known aspects of computer hardware — the whys and hows of hardware design and implementation.

Three primary goals of this course:

  • How computers work at the hardware level
  • How to code in ARM assembly language
  • Various components used to make programs — compilers, call stack, etc.

Key word: Optimization. Everything you learn about low-level hardware applies directly to writing better high-level code.

What you will learn:

  • How programs are translated into machine language and how hardware executes them
  • The hardware/software interface
  • What determines program performance and how to improve it
  • How hardware designers improve performance
  • What parallel processing is

Classes of Computers

▶

Personal Computers

General purpose, variety of software. Subject to cost/performance tradeoff.

Server Computers

Network based. High capacity, performance, and reliability. Range from small servers to building-sized installations.

Supercomputers

High-end scientific and engineering calculations. Highest capability but represent a small fraction of the overall market.

Embedded Computers

Hidden as components of systems. Stringent power/performance/cost constraints.

Personal Mobile Devices

Battery operated, connects to the Internet. Smartphones, tablets, electronic glasses. Hundreds to thousands of dollars.

Cloud Computing

Warehouse Scale Computers (WSC). Software as a Service (SaaS). Portions run on PMD, portions in the Cloud (Amazon, Google).

We are now in the PostPC Era — Personal Mobile Devices and Cloud Computing have replaced the traditional PC as the dominant computing platform.

Understanding Performance

▶

Performance is determined by a layered stack of factors:

  • Algorithm: Determines number of operations executed
  • Programming language, compiler, architecture: Determine number of machine instructions per operation
  • Processor and memory system: Determine how fast instructions execute
  • I/O system (including OS): Determines how fast I/O operations execute

Defining Performance: Performance = 1 / Execution Time. "X is n times faster than Y" means Execution_Time_Y / Execution_Time_X = n.

Response Time

How long it takes to do a task. Also called latency. What a single user cares about most.

Throughput

Total work done per unit time — tasks/transactions per hour. What a data center cares about most.

Elapsed Time

Total response time including processing, I/O, OS overhead, and idle time. Determines system performance.

CPU Time

Time spent processing a given job only — discounts I/O time and other jobs. Split into user CPU time and system CPU time.

Eight Great Ideas in Computer Architecture

▶

1. Design for Moore's Law

ICs double in capacity roughly every 2 years. Design for where technology will be when the product ships, not where it is today.

2. Use Abstraction

Hide lower-level detail to simplify design. Each layer presents a clean interface to the layer above.

3. Make the Common Case Fast

Optimizing the frequent case yields more benefit than optimizing the rare case. Amdahl's Law formalizes this.

4. Performance via Parallelism

Do multiple operations simultaneously to increase throughput.

5. Performance via Pipelining

Overlap execution of multiple instructions — like an assembly line.

6. Performance via Prediction

Guess the likely outcome and start working — correct if wrong. Faster on average than waiting.

7. Hierarchy of Memories

Fastest/smallest/most expensive memory at top, slowest/largest/cheapest at bottom. Cache exploits this.

8. Dependability via Redundancy

Include redundant components so a system can continue working despite failures.

Below Your Program

▶

Programs exist at multiple levels of abstraction:

Application Software

Written in high-level language. Closest to the problem domain. Provides productivity and portability.

System Software

Compiler translates HLL to machine code. OS handles I/O, memory management, task scheduling, and resource sharing.

Assembly Language

Textual representation of machine instructions. Human-readable form of the hardware's native language.

Hardware

Binary digits (bits). Encoded instructions and data. Processor, memory, I/O controllers.

Instruction Set Architecture (ISA) is the hardware/software interface — the contract between hardware and software that allows them to evolve independently.

Memory types:

  • Volatile main memory (DRAM): Loses data when power is off
  • Non-volatile secondary memory: Magnetic disk, flash memory, optical disk — persists without power

CPU Clocking, CPI & Performance Equations

▶

Digital hardware is governed by a constant-rate clock:

  • Clock period: Duration of one clock cycle (e.g., 250ps = 0.25ns)
  • Clock frequency: Cycles per second (e.g., 4.0GHz = 4×10⁹ Hz)

CPU Time = Clock Cycles × Clock Cycle Time = Clock Cycles / Clock Rate

Performance is improved by reducing clock cycles OR increasing clock rate — but hardware designers must trade these off against each other.

CPI (Cycles Per Instruction):

CPU Time = Instruction Count × CPI × Clock Cycle Time

Instruction Count

Determined by the program, ISA, and compiler. Not affected by hardware.

CPI

Determined by CPU hardware. Different instruction classes have different CPI — weighted average depends on instruction mix.

CPI Example: Sequence 1 (IC=5) takes 10 cycles → avg CPI 2.0. Sequence 2 (IC=6) takes 9 cycles → avg CPI 1.5. Sequence 2 is faster despite more instructions!

The Power Wall & Multiprocessors

▶

In CMOS IC technology: Power = Capacitive load × Voltage² × Frequency

As clock rates increased, power consumption became unmanageable:

  • Voltage can't be reduced further without losing reliability
  • Heat can't be removed fast enough
  • Single-core performance hit a wall around 2004

The solution: multicore processors. Instead of one faster core, put multiple cores on one chip. But this requires explicitly parallel programming — the hardware can no longer hide the parallelism from the programmer.

Challenges of parallel programming:

  • Load balancing — keeping all cores busy
  • Optimizing communication between cores
  • Synchronization — avoiding race conditions

Power at idle: Servers rarely run at 100% load. Google data centers mostly operate at 10–50% load. A processor drawing 47% of full power at 10% load is very inefficient — designing power-proportional processors is an active research area.

Amdahl's Law & Pitfalls

▶

Amdahl's Law: The performance improvement from enhancing one part of a system is limited by the fraction of time that part is used.

T_improved = T_affected / improvement_factor + T_unaffected

Example: If multiply accounts for 80s of 100s execution time, making multiply infinitely fast still only gives a 5× speedup — not infinite. The unaffected 20s is the bottleneck. Corollary: make the COMMON case fast.

MIPS is a flawed metric: Millions of Instructions Per Second doesn't account for differences in ISAs or instruction complexity. A program with fewer but more complex instructions can be faster than one with more simple instructions, yet show lower MIPS.

Concluding principles:

  • Execution time is the only reliable performance metric
  • Cost/performance improves due to underlying technology development
  • Hierarchical abstraction exists in both hardware and software
  • Power is a fundamental limiting factor — use parallelism to improve performance
