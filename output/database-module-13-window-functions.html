SQL Windowing

▶

Window functions extend T-SQL's set-based approach by allowing you to specify an order as part of a calculation — without affecting the order of input or final output. They allow partitioning and framing of rows to support powerful analytical calculations.

Window functions can simplify queries that need:

  • Running totals
  • Moving averages
  • Gaps in sequences
  • Rank within a group
  • Comparisons between adjacent rows

Key insight: window functions don't collapse rows like GROUP BY does. Every source row is preserved in the output — the window function adds a calculated column alongside existing data. This is the fundamental difference between window functions and grouped aggregates.

-- Running total of quantity by category and year
SELECT Category, Qty, Orderyear,
    SUM(Qty) OVER (
        PARTITION BY category
        ORDER BY orderyear
        ROWS BETWEEN UNBOUNDED PRECEDING
             AND CURRENT ROW) AS RunningQty
FROM Sales.CategoryQtyYear;

The OVER Clause

▶

OVER defines a window — the set of rows the window function operates on. It has three optional components:

OVER (
    [ PARTITION BY <partition_column> ]
    [ ORDER BY <order_column> ]
    [ ROWS BETWEEN <start> AND <end> ]
)
  • OVER() with nothing: creates a single window of ALL rows — unrestricted
  • OVER(PARTITION BY col): restricts the window to rows sharing the same value in col
  • OVER(ORDER BY col): provides ordering context within the window
  • Multiple OVER clauses: a single query can contain multiple OVER clauses, each with different partitioning and ordering

OVER does NOT sort the final output. To sort the result set you still need a separate ORDER BY clause at the end of the query. OVER's ORDER BY only provides context for the window function calculation.

-- Two different windows in one query
SELECT productid, unitprice,
    RANK() OVER(ORDER BY unitprice DESC) AS pricerank,
    RANK() OVER(PARTITION BY categoryid
                ORDER BY unitprice DESC) AS categoryrank
FROM Production.Products;

Partitioning Windows

▶

Partitioning limits a window to rows with the same value in the partitioning column — similar to GROUP BY but without collapsing rows.

-- Total quantity per customer, alongside each row's detail
SELECT custid, ordermonth, qty,
    SUM(qty) OVER(PARTITION BY custid) AS totalbycust
FROM Sales.CustOrders;

-- Result: every row preserved, total shown for each customer's group
-- custid  ordermonth   qty  totalbycust
-- 1       2007-08      38   174
-- 1       2007-10      41   174    ← same total for custid=1
-- 2       2006-09      6    63
-- 2       2007-08      18   63     ← same total for custid=2

Without PARTITION BY, OVER() operates on ALL rows as a single window. With PARTITION BY, it operates independently within each partition — like running the aggregate separately for each group, but keeping all rows visible.

Ordering & Framing

▶

Window framing defines start and end boundaries within a partition. It requires ORDER BY in the OVER clause to be meaningful — without ordering, "first row" and "last row" have no meaning.

BoundaryMeaning
UNBOUNDED PRECEDINGStart of the partition (first row)
CURRENT ROWThe current row being processed
UNBOUNDED FOLLOWINGEnd of the partition (last row)
N PRECEDINGN rows before the current row
N FOLLOWINGN rows after the current row
-- Running total (cumulative sum): start to current row
SUM(qty) OVER(
    PARTITION BY custid
    ORDER BY ordermonth
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

-- Moving average of last 3 rows including current
AVG(qty) OVER(
    ORDER BY orderdate
    ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)

-- Total of entire partition (no frame — same as no ORDER BY)
SUM(qty) OVER(PARTITION BY custid)

ROWS vs RANGE: ROWS treats each physical row individually. RANGE treats rows with the same ORDER BY value as a group. Use ROWS for running totals — it's more predictable and generally faster.

Window Ranking Functions

▶

Ranking functions assign a rank to each row within a window. They require an ORDER BY in the OVER clause. Partitioning is optional.

FunctionTies?Gaps?Description
RANK()YesYesRanks rows — ties get same rank, next rank skips. 1,1,3,4...
DENSE_RANK()YesNoRanks rows — ties get same rank, no gaps. 1,1,2,3...
ROW_NUMBER()NoNoUnique sequential number regardless of ties. 1,2,3,4...
NTILE(n)N/AN/ADistributes rows into n equal groups, returns group number.
-- Rank products by price, highest first
SELECT productid, productname, unitprice,
    RANK()       OVER(ORDER BY unitprice DESC) AS rnk,
    DENSE_RANK() OVER(ORDER BY unitprice DESC) AS dense_rnk,
    ROW_NUMBER() OVER(ORDER BY unitprice DESC) AS rownum
FROM Production.Products
ORDER BY unitprice DESC;

-- Rank within category partition
SELECT productid, categoryid, unitprice,
    RANK() OVER(PARTITION BY categoryid
                ORDER BY unitprice DESC) AS rank_in_cat
FROM Production.Products;

Use ROW_NUMBER() when you need guaranteed unique numbers (e.g. paging). Use RANK() or DENSE_RANK() when ties matter and you want equal ranks for equal values. Use NTILE(4) to divide data into quartiles for percentile analysis.

Window Aggregate Functions

▶

Window aggregate functions (SUM, MIN, MAX, AVG, COUNT) work like grouped aggregates but operate over a window defined by OVER — preserving all source rows in the output.

-- Running total by customer and month
SELECT custid, ordermonth, qty,
    SUM(qty) OVER(
        PARTITION BY custid
        ORDER BY ordermonth
        ROWS BETWEEN UNBOUNDED PRECEDING
             AND CURRENT ROW) AS running_total,
    AVG(qty) OVER(
        PARTITION BY custid) AS avg_per_cust,
    MAX(qty) OVER(
        PARTITION BY custid) AS max_per_cust
FROM Sales.CustOrders
ORDER BY custid, ordermonth;

The same aggregate function can appear multiple times in a SELECT with different OVER clauses — one for a running total, one for a partition total, one for the overall total — all in the same query without self-joins or subqueries.

Window Offset Functions

▶

Offset functions allow comparisons between rows in a set without needing a self-join. They operate relative to the current row's position in the window.

FunctionDescription
LAG(col, n, default)Returns value from n rows BEFORE current row. Returns default (or NULL) if no row exists.
LEAD(col, n, default)Returns value from n rows AFTER current row. Returns default (or NULL) if no row exists.
FIRST_VALUE(col)Returns first value in the current window frame. Requires ORDER BY to be meaningful.
LAST_VALUE(col)Returns last value in the current window frame. Requires ORDER BY and framing to be meaningful.
-- Compare current year sales to next year's sales per employee
SELECT employee, orderyear,
    totalsales AS currsales,
    LEAD(totalsales, 1, 0) OVER (
        PARTITION BY employee
        ORDER BY orderyear) AS nextsales,
    totalsales - LAG(totalsales, 1, 0) OVER (
        PARTITION BY employee
        ORDER BY orderyear) AS change_from_prev
FROM Sales.OrdersByEmployeeYear
ORDER BY employee, orderyear;

-- Result:
-- employee  orderyear  currsales  nextsales  change_from_prev
-- 1         2006       38789.00   97533.58   38789.00
-- 1         2007       97533.58   65821.13   58744.58
-- 1         2008       65821.13   0.00       -31712.45

LAG and LEAD eliminate the need for self-joins when comparing consecutive rows. The third parameter is the default value returned when no row exists at the offset position (e.g., the last row has no LEAD, the first row has no LAG). Without a default, NULL is returned.

Window Distribution Functions

▶

Distribution functions perform statistical analysis on data. They require a window ORDER BY clause.

PERCENT_RANK()

Returns the relative rank of a row as a percentage between 0 and 1. Formula: (rank - 1) / (total rows - 1). First row = 0.0, last row = 1.0.

CUME_DIST()

Returns the cumulative distribution — the fraction of rows with values less than or equal to the current row. Always between 0 and 1, last row = 1.0.

PERCENTILE_CONT(n)

Calculates a percentile by interpolating between values. Returns a continuous value that may not exist in the data.

PERCENTILE_DISC(n)

Calculates a percentile by returning the first value in the ordered set whose cumulative distribution is >= n. Returns an actual value from the data.

-- Rank distribution: what percentile is each product's price?
SELECT productid, unitprice,
    PERCENT_RANK() OVER(ORDER BY unitprice) AS pct_rank,
    CUME_DIST()    OVER(ORDER BY unitprice) AS cume_dist
FROM Production.Products
ORDER BY unitprice;
