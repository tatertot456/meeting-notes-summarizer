M-Way Search Trees

▶

Trees don't have to be binary — nodes can have multiple children. An m-way search tree generalizes the BST to nodes with up to m children and m-1 keys.

Four characteristics of an m-way search tree:

  • Each node has at most m children and m-1 keys
  • Keys in each node are in ascending order
  • Keys in the first i children are smaller than the ith key
  • Keys in the last m-i children are larger than the ith key

M-way trees offer the same benefits as BSTs (fast retrieval and updates) but suffer the same balance problems. An unbalanced 4-way tree may require different numbers of comparisons to find different keys. Balance is especially critical for disk-based storage — each level traversal potentially means an expensive disk seek.

B-Trees — Disk-Optimized Search Trees

▶

B-Trees (Bayer & McCreight, 1972) are designed specifically for secondary storage. A B-tree node can be as large as a disk block, dramatically reducing the number of disk accesses needed.

Why B-Trees? Binary trees spread nodes across many disk blocks — each pointer follow = another disk seek. Mechanical disk operations are orders of magnitude slower than memory. With B-trees of order m=200 and 2 million keys, the height h < 4 — finding any key requires at most 4 disk seeks. If the root is kept in memory, only 3 disk seeks are needed.

template <class T, int M>
class BTreeNode {
    bool leaf;
    int keyTally;
    T keys[M-1];          // up to M-1 keys per node
    BTreeNode *pointers[M]; // M child pointers
};

// B-Tree properties:
// Root has 2 to M children (or 0 if leaf)
// Non-root nodes have ⌈M/2⌉ to M children
// All leaves at the same level (perfectly balanced)
// Average space utilization: ~69% over time

Insertion — three cases:

Case 1: Leaf has room

Insert key in sorted order in the leaf. No structural change needed — just shift keys to maintain order.

Case 2: Leaf is full — split

Split the full leaf. Distribute keys between old and new node. Promote the median key to the parent. If parent is also full, repeat up to root.

Case 3: Root splits

If split propagates to root, create a new root. Tree grows upward — B-trees are always built bottom-up, staying perfectly balanced.

Presplitting variation

Split full nodes proactively during top-down search, preventing splits from propagating upward later. Trades one extra pass for simpler insertion.

Deletion:

  • If leaf is more than half full: simply remove the key
  • If leaf is less than half full: borrow a key from a sibling
  • If no sibling has an extra key: merge the node with a sibling
  • B-trees must be at least 50% full at all times

B*-Trees — Delayed Splitting

▶

B*-Trees delay splitting by redistributing keys between a node and its sibling when overflow occurs, instead of immediately splitting.

Redistribution (sibling has room)

When a node overflows, redistribute keys between it and its sibling. The median key goes to the parent. No new node created.

Split (both nodes full)

When both the overflowing node AND its sibling are full: create a new node, distribute keys among all three nodes. Two separating keys go to parent. All three nodes end up 2/3 full.

B*-Tree average utilization: ~81% (vs ~69% for standard B-Trees). The fill factor can be generalized: a Bn-tree has fill factor (n+1)/(n+2). B*-tree = B2-tree (2/3 full), B**-tree = B3-tree (3/4 full). The tradeoff: more complex insertion logic for better space utilization.

B+-Trees — Sequential + Random Access

▶

Standard B-trees support random access but sequential access (inorder traversal) requires many page changes. B+-Trees separate random access indexes from sequential data access.

Index Set (Internal Nodes)

Contains keys and child pointers only — used as navigation guides. Keys here may not correspond to actual data records. Optimized for fast top-down search.

Sequence Set (Leaf Nodes)

Contains actual data (keys + record references). Leaves are linked sequentially — scanning all leaves gives data in sorted order without traversing internal nodes.

// B+-tree structure:
// Internal nodes: [key₁ | ptr₁ | key₂ | ptr₂ | ... | keyₙ | ptrₙ₊₁]
// Leaf nodes:     [key₁ | ref₁ | key₂ | ref₂ | ... | keyₙ | refₙ | →next leaf]
//                                                                    ↑ linked list!

// Insertion into a leaf (leaf has room):
// Insert key in sorted order in leaf only — no change to internal nodes

// Insertion into a full leaf (splits):
// Split leaf: distribute keys between old and new leaf
// COPY first key of new node to parent (unlike B-tree which MOVES the key)
// In B+ trees, all data stays in leaves; internal nodes are just separators

// Deletion:
// Key deleted from leaf can REMAIN in internal node — it's just a separator
// Only fix internal nodes if underflow requires merging leaves

B+-trees are the backbone of relational database indexes (MySQL InnoDB, PostgreSQL). Sequential scans (range queries, ORDER BY) can follow the linked leaf list — no tree traversal needed. Random access uses the index set. The key insight: a deleted key in a leaf CAN remain as a separator in an internal node since separators don't need to match actual data.

Simple Prefix B+-Trees — use the shortest prefix that distinguishes adjacent keys as separators, rather than full keys. This fits more separators per node, reducing tree height and improving performance by 25% for trees of 400-800 nodes.

2-4 Trees & Red-Black Trees

▶

A 2-4 tree (also called 2-3-4 tree) is a B-tree of order 4 — each node holds 1, 2, or 3 keys and 2, 3, or 4 children. Unlike disk-based B-trees, 2-4 trees are useful for in-memory structures because they maintain balance automatically.

The problem: implementing 2-4 trees directly in memory wastes space — arrays of 3 keys and 4 pointers per node, where on average only 69% are used. Memory is more precious than disk. Solution: transform 2-4 trees into binary trees using two link colors.

Red-Black Trees — a 2-4 tree represented as a binary tree:

Black links (vertical)

Parent-child links between different 2-4 tree nodes. Represent the tree's actual hierarchical structure.

Red links (horizontal)

Links between keys that belong to the SAME 2-4 tree node. A 3-node becomes two binary nodes connected by a red link.

VH-Trees

Vertical-horizontal trees — an alternative representation. Better for generalizing to B-trees of any order. Properties: equal black-link path length from root to any null; no two consecutive horizontal links.

STL Use

STL set, multiset, map, and multimap are ALL implemented as red-black trees. This gives O(log n) insert, delete, and search guaranteed.

R-Trees & k-d B-Trees — Spatial Data

▶

R-Trees (Guttman, 1984) handle spatial data — objects in 2D or nD space. Each leaf stores (rectangle, object_id) where rectangle is the smallest bounding box containing the object.

// R-Tree leaf entry: (rect, id)
// rect = smallest bounding rectangle containing object id
// Example: Area X at coordinates (10,100),(5,52) → [(10,100),(5,52), X]

// Internal nodes: (rect, child)
// rect = smallest rectangle containing all rectangles in child subtree

// Insertion: like B-tree, with splitting when nodes are full
// Search: may follow "wrong" paths due to overlapping bounding rectangles
//         → can require checking multiple paths

R+-Trees fix R-Tree overlap by ensuring encompassing rectangles don't overlap. Each data rectangle is associated with all rectangles it intersects — a data object may appear in multiple leaves. This eliminates wrong-path searches but adds complexity: an object may need to be deleted from multiple leaves.

k-d B-Trees — multiway version of k-d trees for multidimensional point data. Each leaf holds up to 'a' points, non-terminal nodes hold up to 'b' regions. Insertion may cause cascading splits upward (much more complex than B-tree splits). Splits use a discriminator key to partition points into two groups.

STL Sets, Maps, Multisets & Multimaps

▶

All four STL associative containers are implemented as red-black trees internally.

#include <set>
#include <map>

// SET — unique elements, sorted, O(log n) operations
set<int> s;
s.insert(42);
s.insert(17);
s.insert(42);   // ignored — duplicates not allowed
s.count(42);    // 1
s.find(17);     // iterator to 17
s.erase(17);

// MULTISET — like set but allows duplicates
multiset<int> ms;
ms.insert(42);
ms.insert(42);   // allowed!
ms.count(42);    // 2

// MAP — key→value pairs, unique keys, sorted by key
map<string, int> m;
m["alice"] = 95;
m["bob"]   = 87;
m["alice"] = 99;   // update — keys are unique
m["alice"];        // 99
m.find("bob");     // iterator to ("bob", 87)

// MULTIMAP — like map but allows duplicate keys
multimap<string, int> mm;
mm.insert({"alice", 95});
mm.insert({"alice", 87});   // allowed!

Sets store values; maps store key-value pairs ordered by key only (values can be modified, keys cannot). Ironically, set-theoretic operations (union, intersection) cannot be applied directly to STL sets because of the underlying tree structure — you must first convert to a vector or list. Elements in sets/multisets cannot be modified in place (they're const) because modification would break the sorted ordering.

Tries

▶

A trie (pronounced "try") uses individual characters of keys — not whole keys — for navigation. Each level of the trie processes the ith character of the key. Internal nodes are arrays of pointers indexed by character.

// Searching for a word:
// At level i, check the ith character of the key
// Follow the pointer at that character position
// If null → search fails (or insert here)
// Continue until a leaf containing the full key is found

// Special character '#' handles words that are prefixes of other words:
// "ARE" and "AREA" both in trie
// After processing A-R-E, check '#' link
// If '#' link is non-null → "ARE" is a word (search succeeds)
// Then continue for A-R-E-A

// Trie height = length of longest common prefix + 2
// For 100,000 English words: typically 5-7 comparisons
// Balanced BST for same data: height ≈ 17

Trie advantages: search speed is independent of the number of keys (depends only on key length). Insertion order doesn't affect tree shape (unlike BSTs). Naturally handles prefix searches and autocompletion. The main problem: space waste — each internal node stores pointers for all possible characters (27 for letters + '#'), even if most are null.

Space optimization approaches:

Remove null pointers

Store only non-null pointers. Saves space but loses O(1) character indexing — must search through variable-length lists.

Trie a tergo

Store reversed words to reduce height when words share common suffixes (e.g., "logged", "logging" → share "gnig..." prefix). Tradeoff: common suffixes become common prefixes.

Compressed tries

Interleave node arrays into one large array. Space-efficient but search can yield incorrect results — need verification step. Compression is NP-hard for optimal solutions.

C-tries

Store nodes of each level in consecutive memory. Use bit vectors and offsets. Excellent for static tries — very poor for dynamic updates (insertions require rebuilding).
