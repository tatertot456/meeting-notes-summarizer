OOP Concepts & Terminology

▶

Object-Oriented Programming is one of the most important developments in software — possibly the most significant in the past 50 years. It enables more complex, reusable software with faster development times and fewer bugs.

class

A design document (blueprint) for creating objects. Defines methods, properties, permissions, behaviors, and attributes.

object

An instantiation (creation) of a variable based on a class. Like a house built from a blueprint.

method

A function built into an object. Called with object.method() syntax.

property

A state variable built into an object — stores the object's current state.

constructor

The method called when an object is created. In Python: __init__(). Initializes the object's state.

interface

A behavior specification — guarantees certain methods exist and behave in a defined way.

The key insight of OOP: objects hide complicated procedures behind easy-to-use methods. You don't need to know HOW your leg muscles work to walk to class — you just walk. That's encapsulation. Objects let software work the same way.

Classes, Constructors & Instantiation

▶ 
# Minimal class with documentation
class Employee:
    ''' This is an employee class '''

print(Employee.__doc__)   # prints the docstring
help(Employee)            # full formatted help

# Class with constructor (__init__)
class Employee:
    ''' This is an employee class '''
    def __init__(self, name, age, staffnumber, designation):
        self.name = name
        self.age = age
        self.staffnumber = staffnumber
        self.designation = designation

    def Display(self):
        print(f"The name is: {self.name}")
        print(f"The age is: {self.age}")
        print(f"Staff #: {self.staffnumber}")
        print(f"Designation: {self.designation}")

# Instantiate (create) an object — call the class like a function
EmpKevin = Employee("Kevin", 124, 100329, "Professor Extraordinaire")

# Access attributes with dot notation
print(EmpKevin.name)
print(EmpKevin.age)
print(EmpKevin.designation)

# Call a method
EmpKevin.Display()

# Object dictionary — all attributes stored here
print(EmpKevin.__dict__)
# {'name': 'Kevin', 'age': 124, 'staffnumber': 100329, ...}

__init__ is Python's constructor — it's called automatically when you create an object. It initializes the object's state by assigning values to instance variables using self. The __dict__ attribute gives you a dictionary of ALL instance variables and their current values — very useful for debugging.

self

▶

self is Python's equivalent of C++'s this. It's a reference to the current object instance — necessary to distinguish the class's own variables and methods from local variables.

class Employee:
    def __init__(self, name, age):
        self.name = name    # instance variable — attached to THIS object
        self.age = age

    # self is ALWAYS the first parameter of instance methods
    def Display(self):
        print(f"Name: {self.name}")    # WORKS — uses self
        print(f"Age: {self.age}")

    def DisplayNoSelf(self):
        print(f"Name: {name}")         # NameError — name isn't defined
        print(f"Age: {age}")           # without self, Python looks in
                                       # local scope and can't find it

# __call__ — makes an object callable like a function
class DisplayName:
    def __init__(self, myname):
        self.myname = myname

    def __call__(self):               # invoked when you do: obj()
        print(f"My name is: {self.myname}")

myobj = DisplayName("Kevin")
myobj()   # calls __call__ — prints "My name is: Kevin"

self is not a keyword — it's a convention. You could technically name it anything, but ALWAYS use self. It's the first parameter of every instance method. Without self.variable, Python looks for a local variable with that name and fails. The __call__ dunder method makes an object callable like a function — myobj() instead of myobj.display().

Decorators

▶

A decorator extends an existing function by wrapping it with another function. It's Python's answer to function overloading (which Python doesn't natively support).

# The original function
def greet(myname):
    print("HI!", myname, "! Welcome!")

# A decorator function — takes a function and returns a modified one
def MyDecorator(myfunc):
    def func1(myname):
        if myname == "Michael":
            print("Hello", myname, "! Your functionality is extended!")
        else:
            myfunc(myname)     # call the original function normally
    return func1

# Apply the decorator with @ syntax — binds MyDecorator to greet
@MyDecorator
def greet(myname):
    print("HI!", myname, "! Welcome!")

greet("Tom")       # normal path: "HI! Tom ! Welcome!"
greet("Michael")   # extended path: "Hello Michael ! Your functionality is extended!"

# Decorator chaining — outer runs first, then inner
@MyHash          # outer decorator
@MyAtSymbol      # inner decorator
def MyPrint(mymsg):
    print(mymsg)

# When MyPrint("Hello") is called:
# MyHash wraps MyAtSymbol wraps MyPrint
# Order: MyHash → calls MyAtSymbol → calls MyPrint

# Class decorator example
class DisplayName:
    def __init__(self, myname):
        self.myname = myname

    @SpecialName   # decorators work inside classes too
    def MyDisplay(self):
        print(f"My name is: {self.myname}")

Decorators are functions that take a function as input and return a modified function. The @decorator syntax is shorthand for greet = MyDecorator(greet). In chained decorators, the bottom decorator (@MyAtSymbol) is applied first, then the top one (@MyHash) wraps the result. Outer calls inner which calls the original function.

Variable Types in Classes

▶

Local Variables

Declared inside methods. Only exist within that method's scope. Gone when the method returns.

Class Variables

Declared in the class body (outside __init__). Shared among ALL objects of that class — one copy total.

Instance Variables

Declared in __init__ with self. A separate copy created for EACH object — each object has its own value.

class Employee:
    company = "NNU"    # CLASS variable — shared by all Employee objects

    def __init__(self, name, age):
        self.name = name   # INSTANCE variable — unique per object
        self.age = age

emp1 = Employee("Kevin", 50)
emp2 = Employee("Alice", 30)

print(emp1.company)   # "NNU" — class variable
print(emp2.company)   # "NNU" — same value, shared

print(emp1.name)      # "Kevin" — instance variable
print(emp2.name)      # "Alice" — different per object

Employee.company = "EWU"   # change class variable
print(emp1.company)   # "EWU" — ALL objects see the change
print(emp2.company)   # "EWU"

Use instance variables for attributes that differ per object (name, age, salary). Use class variables for attributes shared by ALL objects of that type (company name, species, constant values). Modifying a class variable changes it for ALL instances — a common gotcha when you accidentally modify a class variable instead of an instance variable.

Getters & Setters

▶

Good OOP design avoids letting external code directly access instance variables. Getters and setters provide controlled access — the setter can validate values before assigning them.

# Problem: direct access allows bad values
john = Person("John", 18)
john.age = 19    # fine
john.age = -1    # WRONG but Python allows it with direct access

# Solution: add validation in a setter
class Person:
    def __init__(self, name, age):
        self.name = name
        self.set_age(age)   # use setter in constructor too!

    def set_age(self, age):
        if age <= 0:
            raise ValueError("Age must be positive")
        self._age = age     # store in _age (convention: _ = "internal")

    def get_age(self):
        return self._age

john = Person("John", 18)
john.get_age()         # 18
john.set_age(20)       # valid
john.set_age(-1)       # ValueError: Age must be positive

# Attribute vs variable
# age is the parameter name (external interface)
# _age is the internal attribute (convention: underscore = internal)
john.age    # AttributeError — no .age without a property
john._age   # 18 — _age IS accessible (Python has no true private)

Python uses the underscore convention (_variable) to signal "internal use" — there's no true private like C++ has. Convention: get_age() and set_age() as getter/setter names. The constructor should always call the setter (not assign directly) so validation runs even at creation time.

Properties — The Pythonic Getter/Setter

▶

Properties are Python's cleaner alternative to getters/setters. They let you use attribute syntax (john.age) while still running validation code behind the scenes.

class Person: def __init__(self, name, age): self.name = name self.age = age # this will call set_age through the property! def set_age(self, age): if age <= 0: raise ValueError("Age must be positive") self._age = age def get_age(self): return self._age # Create the property — connects get/set functions to 'age' age = property(fget=get_age, fset=set_age) # Usage — looks just like direct attribute access! john = Person("John", 18) print(john.age) # calls get_age() → 18 john.age = 20 # calls set_age(20) — validates then stores john.age = -1 # calls set_age(-1) → ValueError! # Full property signature property(fget=None, fset=None, fdel=None, doc=None) # fget: getter function # fset: setter function # fdel: delete function (optional) # doc: docstring (optional)

Properties are the preferred Python pattern over explicit get_/set_ methods. With a property, the calling code uses john.age = 20 (clean, readable) rather than john.set_age(20) — but the validation code still runs transparently. The magic is that age = property(...) at the class level intercepts all reads and writes to the age attribute.

Closures & Functions as Objects

▶

In Python, functions are objects — they can be passed as arguments, returned from functions, and stored in variables. This enables powerful patterns.

# CLOSURE — a nested function that "closes over" variables from its outer scope def outerFn(str1): var1 = " is awesome" def nestedFn(): print(str1 + var1) # accesses str1 and var1 from outer scope return nestedFn # returns the inner function (not its result!) myobj = outerFn("Python") # myobj is now the nestedFn function myobj() # "Python is awesome" # The nested function "remembers" str1 and var1 even after outerFn returns # This "closing over" of the outer scope is what makes it a closure # FUNCTIONS AS PARAMETERS — pass functions just like any other object def myUpper(mystr): return mystr.upper() def myLower(mystr): return mystr.lower() def worldVirus(myfunc): # accepts a function as parameter result = myfunc("Covid-19") print(result) worldVirus(myUpper) # "COVID-19" worldVirus(myLower) # "covid-19"

Closures are the mechanism behind decorators — a decorator is just a function that takes a function, creates a nested function that wraps it, and returns the nested function. Understanding closures explains why decorators work. Functions as parameters (higher-order functions) are central to functional programming in Python — map(), filter(), and sorted(key=...) all use this pattern.
