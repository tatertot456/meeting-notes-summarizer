The Raster Model

▶

A raster stores an n×m array of values in cells (pixels), each representing a square area on the ground. The raster is georeferenced by an x-y corner coordinate and a specified cell size. Each pixel contains exactly ONE numeric value — storing multiple attributes requires multiple rasters (bands).

Continuous Rasters

Store numeric values measurable anywhere: elevation, temperature, precipitation, brightness (imagery). Brightness stored as digital numbers 0–255. Examples: DEM, color aerial photo.

Discrete Rasters

Store categorical data: land use/cover, road type. Often converted from vector data. Can only store one attribute per cell. Join table to raster on cell values, or use multiple bands for more info.

Data Rasters

Store values representing actual measurements or categories. CAN be analyzed. Examples: elevation DEM, aerial imagery, land use, roads.

Picture Rasters

Store color display values — like photographing a map. CANNOT be analyzed. Examples: scanned topo map (DRG), ArcGIS Online basemap tiles.

Why use rasters? Better for continuous data storage and analysis, often faster than vector analysis, imagery is essential for certain maps. Tradeoffs: lower coordinate precision, high resolution = high storage cost, limited to one attribute per cell.

Spatial Resolution & Pyramids

▶

Resolution is the x-y dimension of each pixel (cell size). It defines the precision of the raster — a 10m raster can only represent features ≥10m. Storage grows by the square of resolution.

Storage comparison:
10m resolution raster = 9×9 = 81× more cells than 90m raster
→ 81× more storage space for the same geographic area

Raster Pyramids — pre-built lower-resolution copies to speed display:

  • Built by representing four adjacent cells as one cell (successive halvings)
  • Each pyramid level is efficient for a different display scale
  • Scale ranges on layers control which pyramid level is shown at each scale
  • Building pyramids takes time and increases storage by ~50%
  • ArcGIS Online basemaps use multiple resolution layers with scale ranges to appear fast at all zoom levels

Resampling

▶

Resampling changes the resolution of a raster, producing a new copy. A rule must be chosen to determine the value of each new cell. Required when projecting rasters or analyzing rasters with different resolutions.

MethodHow It WorksBest For
Nearest NeighborGrabs the value from the old cell whose center falls nearest the new cell center. Preserves original values exactly.ALWAYS use for categorical/discrete data. Use when original values must be preserved. Fastest method.
BilinearDistance-weighted average of the 4 nearest old cell centers to the new cell center.Continuous data like elevation. Produces smoother results than nearest neighbor.
Cubic ConvolutionDistance-weighted average of the 16 nearest old cell centers. More complex calculation.Continuous data like elevation. Smoothest results. Most time-consuming method.
Block ResamplingGroups cells and applies a statistic (majority, mean, min, max) to determine new cell value.Used in pyramid building. Majority preserves categorical data during coarsening.

Resampling to a LOWER resolution degrades precision and accuracy — information is permanently lost. Resampling to a HIGHER resolution does NOT improve accuracy — you're just storing the same information in more cells. More cells ≠ more information. Never resample to a finer resolution and assume you've improved the data quality.

Pixel Depth, Bands & Spectral Data

▶

Pixel Depth (Radiometric Resolution) — the number of bits/bytes used per pixel determines the range of storable values:

DepthBytesValue RangeCommon Use
8-bit1 byte0–255 (2⁸ = 256 values)Single-band imagery, land cover categories
16-bit2 bytes0–65,535 (2¹⁶ values)High-precision elevation, multi-range sensors
24-bit3 bytes0–16.7 million (2²⁴)RGB color images (8 bits per channel)
32-bit integer4 bytes0–4.2 billion (2³²)Large integer datasets
32-bit float4 bytesDecimal numbers with sign, mantissa, exponentElevation, continuous analysis outputs

Bands — a multiband raster contains multiple raster arrays with the same dimensions and resolution stored together:

  • Each band = a single raster layer
  • Satellite imagery (Landsat has 7+ bands) — each band captures a different wavelength of light (visible, near-infrared, thermal)
  • Color aerial photos = 3 bands (Red, Green, Blue)
  • Any band can be assigned to R, G, or B display channels for different composite views

Natural Color Composite

Red band → R display, Green band → G display, Blue band → B display. Looks like what the human eye would see. Familiar to non-specialists.

False Color Composite

Non-visible bands (e.g., near-infrared) assigned to visible display channels. Highlights features invisible to the naked eye. Vegetation appears bright red in infrared false color.

Compression & NoData

▶

Run-Length Encoding (Lossless)

For discrete rasters. Stores how many consecutive times each value appears: "0 1 1 1 1 1 3 3 3 3" → "01 15 36". Original data perfectly recoverable. Reversible compression.

Decimal Truncation (Lossy)

Rounds or truncates decimal values: 1.2 3.6 4.2 → 1 3 4. Original data cannot be recovered. Irreversible compression. Reduces storage but permanently loses precision.

NoData — a special value indicating absent data (not zero):

  • Zero is a VALID measurement for many quantities (e.g., zero precipitation = it didn't rain)
  • NoData means "no measurement was taken" or "outside the study area"
  • Rasters are always rectangular — NoData fills the corners and edges outside the actual data extent
  • Example: A DEM of South Dakota is rectangular, but NoData fills cells outside the state boundary

3D Data Formats

▶

3D data includes z values (height/elevation) in addition to x-y coordinates. Many formats exist:

DEM (Digital Elevation Model)

Raster storing one elevation value per cell. Most common 3D surface data. Used for slope, aspect, hillshade, viewshed, watershed analysis. Symbolized low-to-high.

TIN (Triangulated Irregular Network)

Vector-based surface. Stores x,y,z triplets (nodes) forming triangular facets. Efficient: fewer nodes for flat areas, more for steep. Preserves data points exactly.

Point Cloud

Millions of x,y,z triplets from LiDAR or photogrammetry (Structure from Motion). Very high density — can represent individual trees, buildings. Often colored by elevation or return intensity.

Voxel

A raster cell with height — occupies a 3D volume. Voxels stacked together store 3D volumetric data. Used for atmospheric, subsurface, or oceanographic data.

Multipatch

Stores 3D building shapes using oriented surfaces (rectangles, circles, ovals). Represents complex 3D objects. Used for city building models.

Space-Time Cube

Aggregates x,y points into a spatial grid across time slices. Shows patterns in space AND time. Uses netCDF format. Requires a Date/Time field.

2D data can be draped on a DEM surface by aligning x-y locations — creating a 3D visualization without actually having 3D data. 2D features with a base height field (e.g., building footprints with a height attribute) can be extruded to form 3D representations. This is much simpler than true 3D data but effective for visualization.

Georeferencing Rasters

▶

Scanned maps have coordinates in pixels (image space) and must be georeferenced to real-world coordinates before use in GIS.

  • User identifies ground control points (GCPs) — pairs of points visible in both the raster and a reference dataset with known coordinates
  • A transformation calculates real-world coordinates from the GCP pairs and applies them to the raster
  • Affine transformation — adjusts only scale, rotation, and translation (no warping). For maps with consistent distortion.
  • Polynomial transformation (rubbersheeting) — allows differential local adjustment. For maps with varying distortion (old maps, aerial photos).
  • RMSE (Root Mean Square Error) — measures transformation accuracy. Should be recorded in metadata.

Projecting rasters (converting between coordinate systems) requires resampling and is more complex than projecting vector data. It can introduce artifacts and should be done as few times as possible. On-the-fly projection is most efficient when the map coordinate system matches the raster's stored coordinate system.

Storage Formats, Tables & Coordinate Systems

▶

Raster formats analyzable in ArcGIS Pro:

  • File Geodatabase Raster (FGDR) — native ArcGIS format, recommended
  • TIFF (.tif) — widely supported, lossless, can embed georeferencing
  • ERDAS Imagine (.img) — common remote sensing format
  • ArcGRID — old Arc/Info format, still readable

Read/display only (cannot analyze or save in ArcGIS): JPEG, PNG, MrSID, BIL, BSQ, and others.

Raster attribute tables — created automatically for discrete rasters with a limited number of unique values:

  • Value field — each unique cell value
  • Count field — number of cells with that value
  • Additional fields — can include descriptive attributes (e.g., geology unit names from original polygon data)

Rasters should be stored in a projected coordinate system that preserves area or distance to ensure analysis functions work correctly. A DEM in a Geographic Coordinate System (degrees) has different shapes and cell counts than the same DEM in State Plane (meters). Analysis results (slope, area calculations) will be incorrect if performed on unprojected rasters.
