What Is a Greedy Algorithm?

▶

A greedy algorithm makes the locally optimal choice at each step, hoping it leads to a globally optimal solution. It never reconsiders past choices.

Greedy vs DP

DP considers all choices and picks the best. Greedy makes one choice that seems best right now. Greedy is simpler and faster — but only correct for certain problem structures.

When Greedy Works

Must have greedy choice property (a globally optimal solution can be arrived at by making locally optimal choices) and optimal substructure.

Greedy algorithms don't always work — the fractional knapsack problem has a greedy solution but the 0-1 knapsack does not. Knowing which structure applies is the key skill.

Activity Selection Problem

▶

Given n activities with start times sᵢ and finish times fᵢ, select the maximum number of non-overlapping activities.

Greedy choice: Always pick the activity that finishes earliest — it leaves the most time for remaining activities.

GREEDY-ACTIVITY-SELECTOR(s, f)
n = s.length
A = {a₁}              // a₁ has earliest finish time
k = 1
for m = 2 to n
    if s[m] ≥ f[k]    // activity m starts after k finishes
        A = A ∪ {aₘ}
        k = m
return A

Runtime: Θ(n) after sorting by finish time (sorting is Θ(n log n)). The greedy choice is provably optimal: the activity that finishes first is always part of some optimal solution.

The recursive version is conceptually clean but the iterative version is O(n) in time and O(1) in extra space — no recursion stack needed.

Elements of the Greedy Strategy

▶

To prove a greedy algorithm correct, establish two properties:

  • Greedy choice property: There always exists an optimal solution that includes the greedy choice. Prove by exchange argument — take any optimal solution, swap in the greedy choice, show it's still optimal.
  • Optimal substructure: After making the greedy choice, the remaining problem has optimal substructure — its optimal solution extends to the overall optimal solution.

Exchange argument pattern: suppose the greedy choice (e.g., earliest finishing activity) is NOT in an optimal solution. Show you can swap it in without making things worse. Therefore, the greedy choice can always be included — making greedy correct.

The Knapsack Problem

▶

Fractional Knapsack

Can take fractions of items. Greedy works: sort by profit/weight ratio, take greedily from highest ratio down. O(n log n) time.

0-1 Knapsack

Must take whole items or leave them. Greedy FAILS here — counterexamples exist. Requires dynamic programming. O(nW) time.

Why greedy fails for 0-1: taking the highest ratio item might leave a gap in the knapsack that smaller items can't fill efficiently. DP considers all combinations. The distinction between fractional and 0-1 is a classic example of why you must verify greedy choice property.

DP for 0-1 Knapsack: K[i,w] = max value using items 1..i with capacity w. Either exclude item i (K[i-1,w]) or include it if it fits (pᵢ + K[i-1, w-wᵢ]).

Huffman Codes

▶

Huffman coding compresses data by assigning shorter binary codes to more frequent characters. It produces an optimal prefix-free code.

Algorithm:

  • Build a min-priority queue of characters by frequency
  • Repeatedly extract two minimum-frequency nodes
  • Merge them into a new node with combined frequency
  • Insert the merged node back into the queue
  • Repeat until one node remains — that's the root of the Huffman tree

Greedy choice: always merge the two least frequent characters. Proof: if two characters have the lowest frequencies, there exists an optimal code where they are siblings at the deepest level — and greedy achieves exactly that.

Runtime: O(n log n) — n-1 merge operations, each O(log n) with a min-heap. Achieves optimal compression for given character frequencies — no other prefix-free code does better.
