Instance and Static Members

▶

Normally each object gets its own copy of every member variable. A static member variable is shared by ALL objects of the class — there's only one copy regardless of how many objects exist.

// Tree.h
class Tree
{
private:
    static int objectCount;   // ONE shared variable for all Tree objects
public:
    Tree() { objectCount++; } // each constructor call increments it

    // Regular accessor
    int getObjectCount() const { return objectCount; }

    // Static accessor — can be called without an object
    static int getStaticCount() const { return objectCount; }
};

// Must define and initialize OUTSIDE the class (in .cpp)
int Tree::objectCount = 0;

// Usage
Tree t1, t2, t3;

// Call via object (regular)
cout << t1.getObjectCount();      // 3

// Call without any object (static function)
cout << Tree::getStaticCount();    // 3 — called before any object exists

Static member variables must be defined and initialized outside the class — the class declaration only declares them. Static member functions can only access static member variables — they have no "this" pointer because they don't operate on a specific object. Use static members for data that belongs to the class as a whole, not to individual instances.

Friend Functions and Classes

▶

A friend is a function or class that is NOT a member of a class but is granted access to its private members. Declared inside the class with the friend keyword.

class Rectangle
{
private:
    double width, length;
public:
    Rectangle(double w, double l) : width(w), length(l) {}

    // Declare a standalone function as friend
    friend void printDimensions(Rectangle r);

    // Declare a member function of another class as friend
    friend void Inspector::examine(Rectangle r);

    // Declare an entire class as friend
    friend class Validator;
};

// Friend function definition (NOT a member — no :: needed)
void printDimensions(Rectangle r)
{
    // Can access private members directly
    cout << r.width << " x " << r.length << endl;
}

Friendship is granted BY the class, not taken by the friend. It is not mutual — if A is a friend of B, that doesn't make B a friend of A. Friendship is also not inherited. Use friends sparingly — they break encapsulation and should only be used when there's a genuine design reason (like operator overloading).

Memberwise Assignment & Copy Constructors

▶

Memberwise assignment: using = copies each member variable from one object to another. The default copy constructor does the same field-by-field copy.

Rectangle r1(10, 5);
Rectangle r2 = r1;    // copy constructor called — memberwise copy
Rectangle r3;
r3 = r1;              // assignment operator — memberwise copy

The problem with pointers: if a member variable is a pointer, memberwise copy copies the ADDRESS — both objects point to the same memory. Modifying one affects the other.

class SomeClass
{
private:
    int *value;    // pointer member!
public:
    SomeClass(int val = 0) { value = new int; *value = val; }
};

SomeClass obj1(5);
SomeClass obj2 = obj1;   // both obj1 and obj2 point to SAME int!
obj2.setVal(13);
cout << obj1.getVal();   // prints 13 — obj1 was changed too! Bug!

Solution: programmer-defined copy constructor that allocates separate memory:

SomeClass::SomeClass(const SomeClass &obj)
{
    value = new int;     // allocate NEW memory
    *value = *obj.value; // copy the VALUE, not the address
}

// Now each object has its own memory
SomeClass obj1(5);
SomeClass obj2 = obj1;
obj2.setVal(13);
cout << obj1.getVal();   // still 5 — independent!

The copy constructor parameter must be a const reference — never by value (that would recursively call the copy constructor forever). The const prevents the copy constructor from accidentally modifying the source object. Rule of Three: if you define a destructor, copy constructor, or copy assignment operator, you should probably define all three.

The this Pointer

▶

Every non-static member function has access to a predefined pointer called this that always points to the object that called the function.

class Student
{
private:
    string name;
public:
    string getName() const { return name; }

    // this is implicitly available in every member function
    void setName(string name)
    {
        // "name" is ambiguous — parameter or member?
        // Use this-> to explicitly mean the member variable
        this->name = name;
    }
};

Student s1, s2;

s1.getName();   // inside getName, this points to s1
s2.getName();   // inside getName, this points to s2

// Returning *this enables chaining
SomeClass& SomeClass::setValue(int v)
{
    value = v;
    return *this;   // return the current object
}
obj.setValue(5).setName("test");  // chaining works!

this is passed as a hidden argument to every non-static member function call. Static member functions do NOT have a this pointer — they don't operate on a specific object. The most common uses of this: disambiguating parameter names from member variable names, and returning the current object (*this) to enable operator chaining.

Operator Overloading

▶

Operators like =, +, -, [], etc. can be redefined to work with class objects. The function name is operator followed by the symbol.

class SomeClass
{
private:
    int *value;
public:
    // Overloaded assignment operator
    const SomeClass& operator=(const SomeClass &rval)
    {
        *value = *rval.value;   // deep copy the value
        return *this;           // enables chaining: a = b = c
    }

    // Overloaded + operator
    SomeClass operator+(const SomeClass &right)
    {
        SomeClass temp;
        *temp.value = *value + *right.value;
        return temp;
    }
};

// Two ways to call an overloaded operator
object1.operator=(object2);   // explicit function call
object1 = object2;            // conventional notation — same thing
object1 = object2 = object3;  // chaining works because of return *this

// Distance between two 2D points using overloaded -
class Point2d
{
private:
    int x, y;
public:
    double operator-(const Point2d &right)
    {
        return sqrt(pow((x - right.x), 2) + pow((y - right.y), 2));
    }
};
Point2d p1(2,2), p2(4,4);
cout << p2 - p1;   // displays 2.82843

Cannot be overloaded: ?: . .* :: sizeof. Cannot change the number of operands. Overloaded relational operators should return bool. Overloaded << and >> must return reference to ostream/istream and take them as parameters (must be friend functions, not member functions). Prefix and postfix ++ / -- are overloaded differently.

Object Conversion

▶

You can define how an object converts to a built-in type by writing an operator function with no return type declaration.

class FeetInches
{
private:
    int feet;
    int inches;
public:
    // Convert FeetInches to int — returns just the feet
    operator int() { return feet; }

    // Convert FeetInches to double
    operator double() { return feet + inches / 12.0; }
};

FeetInches distance(6, 3);

// Automatic conversion
int d = distance;         // calls operator int() — d = 6
double d2 = distance;     // calls operator double() — d2 = 6.25

// Works in expressions too
cout << distance + 2;    // converts distance to int, adds 2

Aggregation

▶

Aggregation: a class has another class as a member variable. Models the "has a" relationship between classes. Same notation as nested structs.

class StudentInfo
{
private:
    string firstName, lastName;
    string address, city, state, zip;
public:
    // constructor, accessors, mutators...
};

class Student
{
private:
    StudentInfo personalData;   // Student HAS A StudentInfo
    double gpa;
    int yearInSchool;
public:
    // Access nested object through dot operator chaining
    void setName(string first, string last)
    {
        personalData.setFirstName(first);
        personalData.setLastName(last);
    }
};

// Real-world example: Course HAS an Instructor and a TextBook
class Instructor { ... };
class TextBook { ... };
class Course
{
private:
    Instructor teacher;    // Course HAS an Instructor
    TextBook book;         // Course HAS a TextBook
    string courseName;
};

Aggregation models "has a" relationships. Inheritance (coming in Chapter 15) models "is a" relationships. A Course HAS an Instructor (aggregation). A SavingsAccount IS A BankAccount (inheritance). Choosing the right relationship is a fundamental OOP design decision.

Lvalues, Rvalues & Move Semantics

▶

Lvalue

Persists beyond the statement that created it. Has a name that makes it accessible to other statements. Example: a variable (int x = 5 — x is an lvalue).

Rvalue

Temporary, cannot be accessed beyond the statement that created it. Example: the result of 2 * 6, or the return value of a function that's immediately used.

int x;
x = 2 * 6;   // 2*6 produces a temporary rvalue (12)
              // then 12 is assigned to x (lvalue)
              // then the temporary is discarded

int square(int a) { return a * a; }
x = square(5);  // square(5) returns rvalue 25
                // 25 is copied to x, then discarded

Rvalue References (&&) allow binding to temporary objects that would otherwise have no name:

// Move constructor — transfers ownership instead of copying
// Much more efficient when the source is a temporary
class Person
{
private:
    string *name;   // pointer member
public:
    // Copy constructor: allocates new memory, copies data
    Person(const Person &other)
    {
        name = new string(*other.name);
    }

    // Move constructor: steals the pointer from the temporary
    Person(Person &&other)   // && = rvalue reference
    {
        name = other.name;    // steal the pointer
        other.name = nullptr; // leave source in valid empty state
    }
};

Move semantics (C++11) dramatically improve performance when a class manages dynamic memory. Instead of copying all data to a new allocation (expensive), a move constructor "steals" the pointer from a temporary object — O(1) instead of O(n). Anytime a class has a pointer member, implement both copy AND move constructors/assignment operators (Rule of Five in modern C++).
