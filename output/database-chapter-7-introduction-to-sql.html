T-SQL Basics & Language Elements

▶

SQL was developed by IBM in the 1970s and adopted by ANSI/ISO standards bodies. Transact-SQL (T-SQL) is Microsoft SQL Server's dialect. SQL is declarative — you describe what you want, not the individual steps to get it.

DML — Data Manipulation Language

Used to query and manipulate data. SELECT, INSERT, UPDATE, DELETE. This is the focus of this course.

DDL — Data Definition Language

Used to define database objects. CREATE, ALTER, DROP. Covered in Chapter 8.

DCL — Data Control Language

Used to manage security permissions. GRANT, REVOKE, DENY. Covered in Chapter 16.

TCL — Transaction Control Language

COMMIT, ROLLBACK, SAVE TRANSACTION. Used to manage data changes atomically.

Key language elements:

  • Predicates: ALL, ANY, BETWEEN, IN, LIKE, OR, SOME
  • Comparison operators: =, >, <, >=, <=, <>, !=
  • Logical operators: AND, OR, NOT
  • Variables: Local variables start with @, system functions use @@. Declared with DECLARE @name datatype = value
  • Comments: Block comments /* ... */ or inline comments --
  • Batch separator: GO keyword separates batches sent to SQL Server

SQL is set-based — it acts on all matching elements at once. Avoid cursors and loops that process rows individually. Tell the engine what you want to retrieve, not how to retrieve it.

The SELECT Statement

▶

The SELECT statement is the main data retrieval command in SQL. Each clause performs a specific function:

ClauseRoleLogical Order
FROMDefines table(s) to query1st evaluated
WHEREFilters rows using a predicate2nd
GROUP BYArranges rows into groups3rd
HAVINGFilters groups by a predicate4th
SELECTDefines which columns to return5th
ORDER BYSorts the final result6th (last)

The order you WRITE a query is not the order SQL Server EVALUATES it. FROM is always evaluated first — this is why WHERE can't reference SELECT aliases (SELECT hasn't run yet when WHERE is evaluated).

Key SELECT options:

SELECT *                          -- all columns (wildcard)
SELECT col1, col2                 -- specific columns
SELECT col1 AS alias              -- column alias
SELECT col1 * col2 AS computed    -- computed column
SELECT DISTINCT col1              -- unique values only
SELECT TOP (10) col1 FROM t       -- first 10 rows
SELECT TOP (10) PERCENT col1      -- top 10% of rows

OFFSET-FETCH for paging through results:

SELECT orderid, orderdate
FROM Sales.Orders
ORDER BY orderdate
OFFSET 10 ROWS
FETCH NEXT 10 ROWS ONLY;   -- rows 11-20

FROM Clause & Joins

▶

The FROM clause determines source tables and produces a virtual table consumed by subsequent clauses. Joins combine rows from multiple tables based on related columns.

A Cartesian product combines every row from one table with every row from another — usually NOT desired. Inner and outer joins start with the Cartesian product then filter it. Always specify join conditions to avoid accidental Cartesian products.

INNER JOIN

Returns only rows where the join condition matches in both tables. Most common join type.

LEFT OUTER JOIN

Returns all rows from the left table plus matching rows from the right. Non-matching right rows get NULL.

RIGHT OUTER JOIN

Returns all rows from the right table plus matching rows from the left. Non-matching left rows get NULL.

CROSS JOIN

Returns all possible combinations — the full Cartesian product. No ON clause. Useful for test data generation.

SELF JOIN

Joins a table to itself using two aliases. Used to compare rows in the same table — e.g. employees and their managers.

FULL OUTER JOIN

Returns all rows from both tables. Non-matching rows from either side get NULL placeholders.

-- INNER JOIN
SELECT o.orderid, o.orderdate, od.productid
FROM Sales.Orders AS o
JOIN Sales.OrderDetails AS od ON o.orderid = od.orderid;

-- LEFT OUTER JOIN: all customers, orders if they exist
SELECT c.custid, c.contactname, o.orderid
FROM Sales.Customers AS c
LEFT OUTER JOIN Sales.Orders AS o ON c.custid = o.custid;

-- Customers with NO orders
SELECT c.custid, c.contactname
FROM Sales.Customers AS c
LEFT OUTER JOIN Sales.Orders AS o ON c.custid = o.custid
WHERE o.orderid IS NULL;

-- SELF JOIN: employees and their managers
SELECT e.empid, e.lastname, m.lastname AS manager
FROM HR.Employees AS e
JOIN HR.Employees AS m ON e.mgrid = m.empid;

WHERE Clause & Filtering

▶

The WHERE clause filters rows based on predicates — logical conditions that evaluate to TRUE, FALSE, or UNKNOWN. Only rows where the predicate is TRUE are returned.

-- Comparison operators
WHERE country = N'Spain'
WHERE orderdate > '20070101'
WHERE orderdate >= '20070101' AND orderdate < '20080101'

-- Special operators
WHERE custid BETWEEN 1 AND 50       -- inclusive range
WHERE country IN ('Spain', 'France', 'Germany')
WHERE lastname LIKE 'S%'            -- starts with S
WHERE region IS NULL                -- test for NULL
WHERE region IS NOT NULL

Three-valued logic: SQL uses NULL to mark missing values. Comparisons involving NULL return UNKNOWN, not TRUE or FALSE. Always use IS NULL or IS NOT NULL — never = NULL or <> NULL, which will always return UNKNOWN.

Logical operators: AND, OR, NOT combine multiple conditions. AND requires both sides to be TRUE. OR requires at least one side to be TRUE. Parentheses control evaluation order.

LIKE wildcards: % matches any sequence of characters. _ matches exactly one character. Example: LIKE 'S%th' matches "Smith", "South".

ORDER BY Clause

▶

ORDER BY sorts the final query result. Without ORDER BY, SQL Server makes no guarantee about row order. It is the last clause logically processed.

-- Sort by column name
SELECT orderid, orderdate FROM Sales.Orders
ORDER BY orderdate;

-- Sort descending
SELECT orderid, orderdate FROM Sales.Orders
ORDER BY orderdate DESC;

-- Sort by alias
SELECT orderid, YEAR(orderdate) AS orderyear
FROM Sales.Orders
ORDER BY orderyear;

-- Cascading sort: last name, then first name
SELECT lastname, firstname FROM HR.Employees
ORDER BY lastname ASC, firstname ASC;

ORDER BY can reference column names, aliases, or ordinal position (not recommended). It can reference columns not in the SELECT list (unless DISTINCT is used). NULLs are sorted together — either first or last depending on sort direction.

Aggregate Functions & GROUP BY

▶

Aggregate functions collapse a collection of rows into a single scalar value. They ignore NULLs except for COUNT(*).

FunctionReturns
COUNT(*)Total number of rows (including NULLs)
COUNT(col)Number of non-NULL values in column
MIN(col)Minimum value
MAX(col)Maximum value
SUM(col)Sum of all non-NULL values
AVG(col)Arithmetic mean (ignores NULLs!)
-- Basic aggregates
SELECT AVG(unitprice) AS avg_price,
       MIN(qty) AS min_qty,
       MAX(discount) AS max_discount
FROM Sales.OrderDetails;

-- GROUP BY: aggregate per group
SELECT custid, COUNT(*) AS order_count
FROM Sales.Orders
GROUP BY custid;

-- HAVING: filter groups (like WHERE but for groups)
SELECT custid, COUNT(*) AS order_count
FROM Sales.Orders
GROUP BY custid
HAVING COUNT(*) > 10;

-- DISTINCT with aggregates: count unique values
SELECT empid, COUNT(DISTINCT custid) AS unique_custs
FROM Sales.Orders
GROUP BY empid;

After GROUP BY, all subsequent phases (HAVING, SELECT, ORDER BY) operate on groups, not individual rows. Every column in SELECT must either appear in GROUP BY or be inside an aggregate function. Use COALESCE(col, 0) to replace NULLs before aggregating to avoid skewed averages.

WHERE vs HAVING: WHERE filters rows BEFORE grouping. HAVING filters groups AFTER grouping. You can't use aggregate functions in WHERE — use HAVING instead.

Subqueries

▶

A subquery is a SELECT statement nested inside another query. The inner query executes first — its output becomes input for the outer query.

Scalar Subquery

Returns exactly one column and one row. Used in WHERE or SELECT clause to supply a single value.

IN Subquery

Returns one column with multiple rows. Used with IN operator to compare a value against a derived list.

FROM Subquery

Returns a virtual table used as the source in FROM clause. Also called a derived table.

Correlated Subquery

References a column from the outer query. Executes once per row of the outer query — can be slower but very powerful.

-- Scalar subquery in WHERE
SELECT orderid, custid
FROM Sales.Orders
WHERE custid = (SELECT custid FROM Sales.Customers
                WHERE contactname = 'Maria Anders');

-- IN subquery
SELECT productid, productname
FROM Production.Products
WHERE categoryid IN (SELECT categoryid
                     FROM Production.Categories
                     WHERE categoryname = 'Beverages');

-- Correlated subquery with EXISTS
SELECT custid, contactname
FROM Sales.Customers AS c
WHERE EXISTS (SELECT 1 FROM Sales.Orders AS o
              WHERE o.custid = c.custid);

Correlated subqueries reference the outer query — they re-execute for every row of the outer query. EXISTS returns TRUE if any row matches — it stops as soon as it finds one, making it efficient for existence checks. Use IN for known lists, EXISTS for existence tests.

ALL and ANY operators: ALL requires all values in the subquery list to satisfy the condition. ANY requires at least one value to satisfy it.

Set Operators

▶

Set operators combine the results of two queries. Both queries must return the same number of columns with compatible data types.

UNION

Combines rows from two queries, removing duplicates. Like a mathematical union of two sets.

UNION ALL

Combines rows including duplicates. Faster than UNION since no duplicate elimination step is needed.

INTERSECT

Returns only rows that appear in BOTH result sets. Useful for finding common members.

EXCEPT

Returns rows from the first query that do NOT appear in the second. Useful for finding differences.

-- UNION: all customers and all employees in one list
SELECT custid AS id, contactname AS name FROM Sales.Customers
UNION
SELECT empid, lastname FROM HR.Employees;

-- INTERSECT: cities where both customers and employees live
SELECT city FROM Sales.Customers
INTERSECT
SELECT city FROM HR.Employees;

-- EXCEPT: customers who never ordered
SELECT custid FROM Sales.Customers
EXCEPT
SELECT custid FROM Sales.Orders;

Crafting SELECT Queries

▶

Writing effective SQL queries requires both knowledge of the language and understanding of the data. Build queries one clause at a time in logical evaluation order:

  • 1. FROM — identify which tables you need
  • 2. WHERE — filter rows to only what you need
  • 3. GROUP BY — group if aggregation is needed
  • 4. HAVING — filter groups if needed
  • 5. SELECT — specify which columns to return
  • 6. ORDER BY — sort the final result

Know your data. Real-world databases are messy and remain in service for decades. Understand the data model before writing queries. Know the problem — understand what question you're answering. A query that returns results isn't necessarily correct if you misunderstood the question.

SQL Functions available in T-SQL:

String Functions

SUBSTRING, LEFT, RIGHT, LEN, REPLACE, UPPER, LOWER, LTRIM, RTRIM, STUFF, SOUNDEX

Date/Time Functions

GETDATE, SYSDATETIME, DATEADD, DATEDIFF, YEAR, MONTH, DAY, DATENAME, DATEPART, ISDATE

Aggregate Functions

SUM, MIN, MAX, AVG, COUNT, COUNT_BIG, STDEV, VAR

Conversion Functions

CAST, CONVERT — convert a value from one data type to another. COALESCE — return first non-NULL value.
