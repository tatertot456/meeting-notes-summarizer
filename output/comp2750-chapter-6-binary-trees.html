Trees, Binary Trees & Binary Search Trees

▶

Trees overcome the limitations of linear structures (lists, stacks, queues) by representing hierarchical data. A tree consists of nodes and edges, drawn with the root at the top growing downward.

Key Terminology

Root = topmost node. Leaves (terminal nodes) = bottom nodes with no children. Level = path length from root + 1. Height = maximum level. Path = unique sequence of edges from root to any node.

Binary Tree

Each node has at most two children: left and right (either can be empty). Complete binary tree: all non-leaf nodes have two children, all leaves on the same level. Level i+1 has 2^i nodes.

BST Property

Left subtree values < node value. Right subtree values > node value. Values are unique. This ordering enables efficient search — navigate left or right based on comparison.

Why Trees?

Applying consistent ordering to nodes dramatically reduces searches — similar to a skip list. A balanced BST finds any element in O(log n) comparisons vs O(n) for sequential search.

// BST Node definition
template <class T>
class BSTNode {
public:
    T info;
    BSTNode *left, *right;
    BSTNode(T el, BSTNode *l = nullptr, BSTNode *r = nullptr)
        : info(el), left(l), right(r) { }
};

// BST Search
BSTNode<T>* search(T el) {
    BSTNode<T> *p = root;
    while (p != nullptr) {
        if (el == p->info) return p;      // found!
        else if (el < p->info) p = p->left;   // go left
        else p = p->right;                // go right
    }
    return nullptr;  // not found
}

Complete binary tree facts: an (i+1)-level complete tree has 2^i leaves, 2^i - 1 nonterminal nodes, totaling 2^(i+1) - 1 nodes total. For any binary tree where all nonterminal nodes have exactly two non-empty children: number of leaves = number of nonterminal nodes + 1. Searching is efficient only for randomly created or balanced trees — highly skewed trees degrade to O(n) sequential search.

Tree Traversal

▶

Tree traversal visits each node exactly once. With n nodes there are n! possible traversals, but a handful are especially useful.

Breadth-First Search (BFS) — level by level:

void BFS() {
    queue<BSTNode*> q;
    q.push(root);
    while (!q.empty()) {
        BSTNode* p = q.front(); q.pop();
        visit(p);                          // process node
        if (p->left)  q.push(p->left);    // enqueue children
        if (p->right) q.push(p->right);
    }
}
// Top-down left-to-right BFS of tree with root 13, children 10,25:
// Output: 13 10 25 2 12 20 31 29

Depth-First Traversals — three orderings (V=Visit, L=Left, R=Right):

TraversalOrderProperty
PreorderV → L → RRoot first, then subtrees. Used in prefix notation.
InorderL → V → RVisits BST nodes in SORTED ORDER. Used in infix notation.
PostorderL → R → VRoot last. Used in postfix notation, evaluating expressions.
// Recursive implementations — simple and elegant
void inorder(BSTNode* p) {
    if (p != nullptr) {
        inorder(p->left);   // L — traverse left subtree
        visit(p);           // V — process this node
        inorder(p->right);  // R — traverse right subtree
    }
}

void preorder(BSTNode* p) {
    if (p != nullptr) {
        visit(p);
        preorder(p->left);
        preorder(p->right);
    }
}

void postorder(BSTNode* p) {
    if (p != nullptr) {
        postorder(p->left);
        postorder(p->right);
        visit(p);
    }
}

Inorder traversal of a BST always produces nodes in sorted order — this is a key property. The runtime stack tracks the backtrack points: going left pushes the current node's state, so after returning from the left subtree, the node is visited, then the right subtree is explored. Non-recursive versions require explicitly managing a stack — more code, avoids activation record overhead, but not always faster.

Threaded Trees & Morris Traversal

▶

Threaded Trees — incorporate the "stack" into the tree structure itself using threads (pointers to inorder predecessors/successors). Null pointers are repurposed as threads — left pointer → predecessor, right pointer → successor.

// Threaded tree traversal — no stack needed!
// Right-thread only (successor threads)
BSTNode* p = leftmost node;
while (p != nullptr) {
    visit(p);
    if (p->rightThread)   // right pointer is a thread
        p = p->right;     // follow thread to successor
    else
        p = leftmost of p->right subtree;
}

Morris Inorder Traversal — stackless, no threads, temporarily modifies tree structure:

void MorrisInorder() {
    BSTNode *p = root, *tmp;
    while (p != nullptr) {
        if (p->left == nullptr) {
            visit(p);          // no left child — visit and go right
            p = p->right;
        } else {
            tmp = p->left;
            while (tmp->right != nullptr && tmp->right != p)
                tmp = tmp->right;  // find rightmost of left subtree

            if (tmp->right == nullptr) {
                tmp->right = p;    // add temp link (thread)
                p = p->left;
            } else {
                visit(p);          // temp link found — visit node
                tmp->right = nullptr; // remove temp link
                p = p->right;
            }
        }
    }
}

Morris traversal achieves O(1) extra space (no stack, no threads) by temporarily modifying the tree during traversal and restoring it. The key insight: inorder traversal is trivial for trees with no left children (just VR). Morris makes every node temporarily have no left child by threading the rightmost node of the left subtree back to the current node. Tree is fully restored when traversal completes.

Insertion

▶

BST insertion follows the same logic as search — navigate to where the new node belongs, then attach it as a leaf.

void insert(T el) {
    BSTNode *p = root, *prev = nullptr;
    while (p != nullptr) {        // find insertion point
        prev = p;
        if (el < p->info) p = p->left;
        else p = p->right;
    }
    // prev is now the future parent
    if (prev == nullptr)          // tree was empty
        root = new BSTNode(el);
    else if (el < prev->info)
        prev->left = new BSTNode(el);
    else
        prev->right = new BSTNode(el);
}

Insertion always adds a new leaf — never restructures existing nodes (unless the tree is self-balancing). The new node's position is determined entirely by the BST ordering property. This means the shape of the tree depends heavily on insertion order — sorted input creates a completely skewed (degenerate) tree, degrading all operations to O(n).

Deletion

▶

Three cases depending on the node's number of children:

Case 1: Leaf Node

Simplest. Set parent's pointer to nullptr and delete the node. No restructuring needed.

Case 2: One Child

Set parent's pointer to the node's only child (bypass the deleted node). Still simple — just relink.

Case 3a: Two Children — Merging

Find the rightmost node in the left subtree (predecessor). Attach the right subtree there. Promote the left subtree to replace the deleted node. Can increase tree height.

Case 3b: Two Children — Copying

Copy the predecessor's value into the deleted node, then delete the predecessor (which has at most one child). Avoids height increase but can create right-heavy bias.

// Deletion by copying — preferred approach
void deleteByCopying(BSTNode*& node) {
    BSTNode *prev, *tmp = node;

    if (node->right == nullptr)      // no right child
        node = node->left;
    else if (node->left == nullptr)  // no left child
        node = node->right;
    else {
        // find rightmost node in left subtree (predecessor)
        tmp = node->left;
        prev = node;
        while (tmp->right != nullptr) {
            prev = tmp;
            tmp = tmp->right;
        }
        node->info = tmp->info;      // copy predecessor's value
        if (prev == node)
            prev->left = tmp->left;  // predecessor was left child
        else
            prev->right = tmp->left; // relink around predecessor
    }
    delete tmp;
}

// Improvement: alternate between deleting predecessor (left) 
// and successor (right) to avoid right-heavy bias

Deletion by merging can increase tree height (bad for performance). Deletion by copying avoids height increase but creates asymmetry — always deleting from the left subtree makes the right subtree grow relatively larger over time. The symmetric fix: alternate between predecessor and successor deletion. This significantly improves balance without the overhead of formal rebalancing algorithms.

Balancing — DSW Algorithm & AVL Trees

▶

A binary tree is height-balanced if the height difference of subtrees at every node is 0 or 1. Skewed trees search no better than linked lists.

Simple rebalancing (sorted array → balanced tree):

void balance(T data[], int first, int last) {
    if (first <= last) {
        int middle = (first + last) / 2;
        insert(data[middle]);         // insert middle as root
        balance(data, first, middle - 1);   // left half
        balance(data, middle + 1, last);    // right half
    }
}
// Drawback: requires all data sorted in array first
// Can't be used while tree is being built

DSW Algorithm (Day-Stout-Warren) — balances any BST in-place using rotations:

// Phase 1: Transform BST into a "backbone" (right-skewed vine) snagify(root): tmp = root while tmp != nullptr: if tmp has left child: rotate left child about tmp // right rotation set tmp to child that just became parent else: set tmp to its right child // Phase 2: Transform backbone into perfectly balanced tree // Apply left rotations: promote every second node createPerfectTree(): n = number of nodes m = 2^floor(log₂(n+1)) - 1 // closest perfect tree size make (n-m) rotations from top // handle overflow while m > 1: m = m / 2 make m rotations from top

AVL Trees — maintain balance after every insertion/deletion using local rotations:

// Each node stores a balance factor = height(right) - height(left)
// Must be -1, 0, or +1 for AVL property

// Case 1: Right-right imbalance → single left rotation of parent about right child
// Case 2: Left-right imbalance → double rotation: left child's right subtree
//         First: rotate right child about left child
//         Then:  rotate that child about parent

// After deletion: update balance factors from deleted node up to root
// Rotate at every node where balance factor becomes ±2
// May need multiple rotations up the tree (unlike insertion, which needs at most 1)

AVL trees maintain height ≤ 1.44 log₂(n) — searches are always O(log n). The tradeoff: insertion and deletion require more work to maintain balance. DSW is ideal for batch rebalancing existing trees. AVL is ideal for trees that need to stay balanced as elements are added/removed individually. The minimum AVL tree formula: AVL_h = AVL_(h-1) + AVL_(h-2) + 1 (similar to Fibonacci numbers).

Self-Adjusting Trees & Splaying

▶

Rather than maintaining perfect balance, self-adjusting trees move frequently accessed elements closer to the root — optimizing for actual access patterns.

Move-to-Root

Each access rotates the accessed node to the root. Assumption: recently accessed elements will be accessed again soon. Works well for "hot" items.

Single Rotation

Each access rotates the node one level up (toward root). More conservative than move-to-root. Slowly improves access for frequently used items.

Splaying (Tarjan & Sleator, 1985)

Applies rotations in PAIRS based on the node-parent-grandparent relationship. Simultaneously moves the node toward root AND flattens the tree. Amortized O(log n) per operation.

Semisplaying

A modification of splaying that continues with the parent of the node rather than the node itself after homogeneous splays. Produces more balanced results in some cases.

// Splaying cases:
// Case 1: Node R's parent is root → single rotation (R about parent)
// Case 2: Homogeneous (both same direction: LL or RR)
//   → rotate parent Q about grandparent P FIRST
//   → then rotate R about Q
//   (This flattens the tree more than move-to-root)
// Case 3: Heterogeneous (different directions: LR or RL)
//   → rotate R about Q FIRST
//   → then rotate R about P

Splaying(P, Q, R):
    while R is not root:
        if R's parent is root:
            rotate R about its parent
        else if homogeneous configuration:
            rotate Q about P, then R about Q
        else:    // heterogeneous
            rotate R about Q, then R about P

Empirical results: AVL trees generally outperform self-adjusting trees. Even regular BSTs sometimes outperform splay trees in practice. Self-adjusting trees shine when access patterns are highly non-uniform (some items accessed much more frequently than others). For uniform access, maintain structural balance instead.

Heaps

▶

A heap is a binary tree with two properties: (1) parent value ≥ children values (max-heap), and (2) perfectly balanced with leftmost leaves on the last level. Ideal for priority queues.

// Heap stored efficiently as an array:
// Parent of node at index i: (i-1)/2
// Left child: 2i+1, Right child: 2i+2
// heap[i] >= heap[2i+1] and heap[i] >= heap[2i+2]

// ENQUEUE — add to end, heapify UP
void heapEnqueue(T el) {
    heap[++last] = el;           // add at end
    int i = last;
    while (i > 0 && heap[i] > heap[(i-1)/2]) {
        swap(heap[i], heap[(i-1)/2]);  // swap with parent
        i = (i - 1) / 2;
    }
}

// DEQUEUE — remove root, heapify DOWN
T heapDequeue() {
    T el = heap[0];              // save root (max value)
    heap[0] = heap[last--];      // move last leaf to root
    int p = 0;
    while (true) {
        int left = 2*p+1, right = 2*p+2;
        int largest = p;
        if (left <= last && heap[left] > heap[largest])
            largest = left;
        if (right <= last && heap[right] > heap[largest])
            largest = right;
        if (largest == p) break;
        swap(heap[p], heap[largest]);  // swap with larger child
        p = largest;
    }
    return el;
}

// Floyd's bottom-up heapify — build heap from array O(n)
// Start from last non-leaf (index n/2-1) and heapify down
for (int i = n/2 - 1; i >= 0; i--)
    moveDown(heap, i, n-1);

Floyd's algorithm builds a heap from an unsorted array in O(n) time — better than inserting n elements one by one (O(n log n)). The trick: start from the bottom, treating each subtree as already heapified, and just fix the root of each subtree. Most nodes are near the leaves where little work is needed. This is the foundation of heapsort.

Treaps & k-d Trees

▶

Treap = BST + Heap. Each node has both a key (BST ordering) and a priority (heap ordering). Combines BST's efficient search with heap's priority structure.

// Treap node
struct TreapNode {
    T key;
    int priority;  // randomly assigned
    TreapNode *left, *right;
};

// Insertion: insert as leaf (BST position), then rotate up if priority > parent
// Deletion: rotate node with higher-priority child until leaf, then delete

With randomly assigned priorities, a treap has the same expected structure as a randomly built BST — O(log n) expected height. The random priorities prevent the worst-case O(n) degradation of standard BSTs with sorted input.

k-d Trees — binary search trees using multiple keys, alternating which key is used at each level:

// 2-d tree: alternates x and y coordinate at each level // Level 1 (root): split on x coordinate // Left subtree: points with x < root.x // Right subtree: points with x > root.x // Level 2: split on y coordinate // Left subtree: points with y < node.y // Right subtree: points with y > node.y // Level 3: back to x, etc. // Level = depth % k determines which key to use // Applications: // Exact match search: same as BST but use appropriate key per level // Range query: check if node is in range, then decide which subtrees to explore // Nearest neighbor search

k-d tree deletion is complex: finding the predecessor or successor requires searching BOTH subtrees (not just one) because the key used to order nodes changes at every level. In general, for nodes on every kth level, only the left subtree need be searched for finding a minimum — otherwise both subtrees may need examination.

Polish Notation & Expression Trees

▶

Expression trees represent arithmetic/logical expressions as binary trees where leaves are operands and internal nodes are operators. They eliminate ambiguity without parentheses.

// Process left-to-right. Operand → push. Operator → pop 2, compute, push result.

// Expression tree node
class ExprTreeNode {
    char *key;           // operator (+,-,*,/) or operand (number/variable)
    ExprTreeNode *left, *right;
};

// Code generation from expression tree (compiler use):
// Postorder traversal generates instructions in correct order
// Left subtree → right subtree → operator instruction
// This is why compilers use expression trees internally

Inorder traversal of an expression tree produces infix notation — the same as the original expression. But ALL THREE trees in Fig 6.64 produce the same infix output despite having different structures, proving infix is ambiguous without parentheses. Preorder (prefix) and postorder (postfix) are unambiguous — operator position relative to operands fully determines evaluation order. This is why compilers and calculators prefer postfix (RPN).
