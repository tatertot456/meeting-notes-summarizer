What Is a Thread?

▶

Up to this point, all programs you've written are single-threaded — one instruction executes at a time, in sequence. A thread is an independent path of execution within a program. Multi-threading allows a program to run multiple tasks simultaneously.

Single-Threaded

One sequence of instructions. Task A must finish before Task B starts. Simple, predictable, but can be slow for independent tasks.

Multi-Threaded

Multiple sequences run concurrently. Task A and Task B run at the same time. Faster for independent work, but introduces new complexity.

Requires #include <thread> (C++11). Threads are not magic — they introduce real problems like race conditions and deadlocks that you must manage carefully. The concepts here build directly on everything you've learned about functions, pointers, and objects.

Creating Threads

▶

A thread is created by constructing a std::thread object and passing it a callable — a function, functor, or lambda. The thread starts executing immediately upon construction.

#include <thread>
#include <iostream>

// A function to run in a thread
void countUp(int start, int end)
{
    for (int i = start; i <= end; i++)
        std::cout << i << " ";
}

int main()
{
    // Create two threads — both start immediately
    std::thread t1(countUp, 1, 50);    // thread 1: counts 1-50
    std::thread t2(countUp, 51, 100);  // thread 2: counts 51-100

    // Wait for both to finish before main() exits
    t1.join();
    t2.join();

    return 0;
}

The thread constructor takes the function name followed by the arguments to pass to that function — just like calling the function, but it runs on a new thread. The thread starts running the moment it's constructed. Without join() or detach(), the program will crash when main() exits while threads are still running.

Joining & Detaching Threads

▶

Every thread must be either joined or detached before the thread object is destroyed — failing to do so terminates the program.

std::thread t(myFunction);

// join(): parent waits here until t finishes
t.join();
// After join(), execution continues — t is done

// detach(): thread runs independently in background
// Parent does NOT wait — thread cleans itself up when done
t.detach();
// After detach(), you can no longer access t

// Check if a thread is still joinable before joining
if (t.joinable())
    t.join();

// Get the ID of the current thread
std::thread::id myId = std::this_thread::get_id();

Detached threads are sometimes called "daemon threads" or "background threads." Once detached, you lose all contact with the thread — you can't join it, check its status, or get its result. Only detach when you genuinely don't need to know when or whether the thread finishes. For most cases, join() is the right choice.

RAII wrapper pattern — ensures threads are always joined/detached even if an exception is thrown:

class ThreadGuard
{
    std::thread &t;
public:
    explicit ThreadGuard(std::thread &t_) : t(t_) { }
    ~ThreadGuard()
    {
        if (t.joinable())
            t.detach();   // auto-cleanup on destruction
    }
};

Race Conditions

▶

A race condition occurs when two or more threads access shared data simultaneously and at least one is writing. The result depends on which thread "wins" — producing inconsistent, unpredictable behavior.

// BROKEN — race condition!
int counter = 0;   // shared between threads

void increment(int n)
{
    for (int i = 0; i < n; i++)
        counter++;   // NOT atomic — read, modify, write = 3 steps
}

std::thread t1(increment, 1000000);
std::thread t2(increment, 1000000);
t1.join();
t2.join();

// Expected: 2,000,000
// Actual: some unpredictable smaller number — race condition!

counter++ looks like one operation but is actually three: READ the value, ADD 1, WRITE back. If two threads both read the same value before either writes, one increment is lost. This is a critical section — a block of code that must not be executed by more than one thread at the same time. The solution is mutual exclusion.

Mutexes & Mutual Exclusion

▶

A mutex (mutual exclusion object) ensures only one thread can execute a critical section at a time. Requires #include <mutex>.

#include <mutex>

int counter = 0;
std::mutex m;      // declared globally, shared by all threads

void increment(int n)
{
    for (int i = 0; i < n; i++)
    {
        m.lock();      // acquire lock — other threads wait here
        counter++;     // critical section — only one thread at a time
        m.unlock();    // release lock — next thread can enter
    }
}

// Better: use lock_guard for automatic unlock (even on exception)
void saferIncrement(int n)
{
    for (int i = 0; i < n; i++)
    {
        std::lock_guard<std::mutex> guard(m);  // locks on creation
        counter++;
    }   // guard goes out of scope here — automatically unlocks
}

Always prefer lock_guard or unique_lock over manual lock()/unlock(). Manual unlock is easy to forget, and if an exception is thrown between lock() and unlock(), the mutex is never released — deadlocking every other thread that tries to acquire it. lock_guard uses RAII to guarantee the mutex is always released.

Performance tip: locking/unlocking has overhead, and other threads must wait. Minimize what's inside the critical section. For a counter, consider using a thread-local variable to accumulate results, then add to the shared counter only once at the end:

void fasterIncrement(int n)
{
    int localCount = 0;       // private to this thread — no lock needed
    for (int i = 0; i < n; i++)
        localCount++;          // no contention

    std::lock_guard<std::mutex> guard(m);
    counter += localCount;    // one lock, one write
}

Thread Callbacks: Functions, Functors & Lambdas

▶

A thread can be started with any callable — a function pointer, a function object (functor), or a lambda expression.

// 1. Function pointer (classic)
void myTask(int x) { std::cout << x; }
std::thread t1(myTask, 42);

// 2. Function object (functor)
class Worker
{
public:
    void operator()(int start, int end)
    {
        for (int i = start; i <= end; i++)
            std::cout << i << " ";
    }
};
Worker w;
std::thread t2(w, 1, 100);          // pass functor object
std::thread t3(Worker(), 1, 100);   // pass temporary functor

// 3. Lambda (most flexible, captures local variables)
int multiplier = 5;
std::thread t4([multiplier](int x)
{
    std::cout << x * multiplier << "\n";
}, 10);

t1.join(); t2.join(); t3.join(); t4.join();

Lambdas are the most convenient for threads — they can capture local variables and define the thread's work inline without creating a separate named function. Be careful what you capture by reference in a lambda — if the variable goes out of scope before the thread finishes, you have a dangling reference.

Passing Parameters to Threads

▶

Arguments passed to a thread are copied by default. Passing pointers or references requires extra care — the original data must still exist when the thread accesses it.

// Passing by value — safe, thread gets its own copy
void processValue(int x) { ... }
std::thread t1(processValue, 42);   // 42 is copied

// Passing a pointer — DANGEROUS if local variable goes out of scope
void processPtr(int *ptr) { *ptr += 1; }
int val = 10;
std::thread t2(processPtr, &val);   // make sure val outlives t2!
t2.join();   // must join before val goes out of scope

// Passing by reference — use std::ref() wrapper
void processRef(int &x) { x += 1; }
int num = 5;
std::thread t3(processRef, std::ref(num));  // std::ref needed!
t3.join();
std::cout << num;   // 6 — thread modified the original

// With dynamically allocated memory — safe as long as
// the memory isn't deleted before the thread finishes
int *heapVal = new int(100);
std::thread t4(processPtr, heapVal);
t4.join();
delete heapVal;

std::thread always copies arguments by default — even if the function takes a reference parameter. To actually pass by reference, you must wrap it in std::ref(). Without std::ref(), the thread gets a copy and your function's reference parameter points to that copy, not the original. This is one of the most common threading bugs for beginners.
