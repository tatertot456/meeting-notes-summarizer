Tables & Database Management Systems

▶

GIS tables come in two types: attribute tables (data associated with a spatial feature class) and standalone tables (tabular data from any source — Excel, CSV, text files, database files).

Table structure: rows = records, columns = fields (attributes). The Feature ID (FID/OID) is a unique integer managed by the software to identify each record.

Flat File

Simple ASCII text file. One record per line. Attributes separated by commas, spaces, or tabs. Simple but very inefficient for queries — no relationships between tables.

Hierarchical DBMS

Multiple tables with predefined parent-child relationships. Very efficient for specific queries but inflexible — only predefined queries are possible. Limits what relationships can be explored.

Relational DBMS

Multiple tables where relationships are defined as needed. Very flexible — any table can be linked to any other on matching fields. Ideal for open-ended applications. MOST COMMON type in GIS.

Table Joins & Cardinality

▶

A join links two tables using a common field (key), allowing them to be used as a single table. The target table receives additional information from the join table.

  • Primary key — column(s) that uniquely identify a row in its own table
  • Foreign key — column(s) that link a row to a row in another table
  • The key field can have different names in each table as long as values match

Cardinality — how many join records match each target record:

TypeRelationshipExampleHow to Handle
One-to-One1 join record = 1 target recordStates to Governors, Countries to CapitalsStandard JOIN — works perfectly
Many-to-One1 join record = many target recordsCities to States, Schools to DistrictsStandard JOIN — join record repeated as needed
One-to-ManyMany join records = 1 target recordStates to Cities, Districts to SchoolsUSE RELATE — tables stay separate but linked
Many-to-ManyMany join records = many target recordsStudents to Classes, Stores to CustomersRELATE — rarely handled well in GIS

JOIN RULE: Each record in the target table must match ONE AND ONLY ONE record in the join table. One-to-one and many-to-one satisfy this rule. One-to-many does NOT — use a Relate instead. In a Relate, tables remain separate but selections in one table highlight linked records in the other. Always put the target table first when evaluating cardinality.

Unmatched records — if a target record has no match in the join table, its join fields show <Null>. Use "Keep All Target Features" (Outer Join) to retain unmatched records. Uncheck for Inner Join (only matched records).

Multiple joins — more than two tables can be joined to one target. A bridge table (with shared fields from both) can link tables that have no direct common field. A table cannot be both a target and a join table simultaneously.

Statistics, Charts & Critical Thinking

▶

Statistics and charts are data exploration tools — use them to understand your data before analysis and to identify problems.

Summary Statistics Tool

Calculates statistics (sum, mean, min, max, count, std dev) for one or more fields. Results stored in a new table. Use a CASE FIELD to group results by category (e.g., sum earthquake damage by state instead of for all earthquakes).

Charts (Linked to Map)

Scatterplots, bar charts, histograms linked to map selections. Features selected on the chart are simultaneously selected on the map. Useful for detecting outliers and correlations (e.g., temperature vs. latitude).

Apply the Straight Face Test to your statistics. Zero-magnitude earthquakes represent missing values — they'll distort your mean and std dev. Earthquake datasets screened to show only large events will look nothing like the natural distribution (which has exponentially more small quakes). Always understand what the data represents before interpreting statistics.

Data Storage — Binary, ASCII & Field Types

▶

Computers store data as binary (base 2) numbers — sequences of 0s and 1s. 1 bit = 1 binary digit. 8 bits = 1 byte = stores values 0–255.

ASCII storage: Text stored as character codes "CAT" = {67, 65, 84} = 3 bytes (one byte per character) "148" stored as ASCII = {49, 52, 56} = 3 bytes (text!) "148" stored as binary = 10010100 = 1 byte (efficient!) Binary is generally more efficient for numbers. Many GIS analysis tools REQUIRE numeric fields — not text.

Integer storage limits:

BytesUnsigned RangeSigned Range
1 byte0 to 255-127 to +127
2 bytes0 to 65,535-32,767 to +32,767
4 bytes0 to 4.29 billion-2.15B to +2.15B

Floating point — stores decimals as scientific notation (mantissa × exponent). Single precision (32-bit) has ~7 significant digits. Double precision (64-bit) has ~15 significant digits. Large numbers lose precision in single precision.

ArcGIS Pro field types:

TypeStorageRange / Notes
Short Integer2 bytes binary-32,767 to +32,767
Long Integer4 bytes binary-2.15B to +2.15B
Float4 bytes (32-bit float)Decimal, ~7 significant digits
Double8 bytes (64-bit float)Decimal, ~15 significant digits

Match the field type to the data: don't use Double when Short Integer will do (wastes storage). Don't store a number in a Text field if you need to do math on it (many tools won't work). Field type CANNOT be changed after creation — plan ahead. Field naming rules: max 13 characters, letters and numbers only, must start with a letter.

Fields, Schema & Geometry Fields

▶

The schema is the complete structure of a table — its fields, their definitions, domains, and defaults. A schema can be saved as a template to create empty tables with identical structure.

Geometry fields — special fields in geodatabase attribute tables:

  • Shape_Length and Shape_Area — automatically updated when feature shape changes (generalize, dissolve operations)
  • User-defined geometry fields must be updated manually with the Calculate Attribute Geometry tool
  • Shapefiles have NO automatic geometry fields — all must be updated manually

Fields view — used to view and modify table structure:

Layer Mode (Cosmetic Changes)

Changes affect only the TABLE VIEW — the underlying data is unchanged. Can modify: display on/off, read-only, display width, number formatting, field highlighting, alias (one table view).

Data Source Mode (Permanent Changes)

Changes affect the SOURCE TABLE permanently. Can modify: add/delete fields, rename fields, change storage width, create domains/subtypes, change alias (all maps and projects). Be careful — cannot be undone!

Domains

▶

Domains are rules about what values can be placed in a field. They enforce data integrity and speed data entry.

Coded Value Domains

Provide a pick list of allowed values. Store a short code but display a readable description. Examples: Conifer/Deciduous, Solid/Dashed, True/False, Good/Fair/Poor. Reduces typing errors.

Range Domains

Specify the minimum and maximum numeric values allowed. Example: a Percent field constrained to 0–100. Prevents invalid data entry.

Domains are created at the geodatabase level (not the feature class level) so they can be reused across many fields. The domain's field type must match the field it's assigned to. Split and merge policies control domain behavior during feature editing operations.

Domains are especially valuable for field data collection on GPS devices and mobile apps where users must choose from a list rather than type freehand. Once a domain is assigned, the field will only accept values in the domain — this protects data quality at the point of entry.

Table Views, Calculating Fields & Table View Properties

▶

Table views display and manipulate attribute data. Right-click a field name for field-level actions. Table Options button provides additional table functions. Status bar shows selection counts.

Table view properties (cosmetic — do not affect source table):

  • Field aliases — display a friendly name instead of the actual field name
  • Numeric formatting — consistent decimal places or significant figures
  • Show/hide fields — display only relevant fields
  • Field order — reorder columns for easier reading

Calculate Field tool — populate or update field values using expressions:

  • Reference existing fields in the expression
  • Use built-in numeric and string functions
  • Python code snippets for complex calculations
  • Use List By Editing panel in Contents pane to lock layers against accidental editing

Importing Excel & Text Data

▶

Excel import requirements:

  • First row must contain legal field names
  • Every column must contain one consistent data type
  • Do NOT use n/a or x to indicate missing values in numeric columns
  • No formulas, blank lines, or merged cells
  • A "dummy row" with example data types (ddd=text, 999=integer, 9.9=float) helps ArcGIS determine field types correctly
Example dummy row format:
NAME, NUM, STREET, CITY, STATE, ZIP, PERCENT
ddd, 999, 999, ddd, ddd, 99, 99.99       ← dummy row
Smith, 527, W. Main St., Peoria, IL, 45098, 38.3

Text file formats:

  • Tab-delimited — fields separated by tab characters
  • CSV (comma-delimited) — fields separated by commas. Often most reliable format.
  • Fixed-column — fields at fixed character positions. ArcGIS Pro CANNOT read this format directly — use Excel to convert to CSV first.

Use the Excel to Table tool to load spreadsheets into a geodatabase table. A CSV file may work better than Excel in some cases. The XY Table to Point tool converts coordinate tables to point feature classes.
