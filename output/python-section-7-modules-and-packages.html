What Is a Module?

▶

Modules are how we organize code for portability, reusability, ease of maintenance, and organization. At its most basic, a module is just a Python file containing variables, functions, and classes that can be imported into other programs.

Function / Method

A group of statements in a code block.

Module

A group of functions, variables, and classes in a single .py file.

Package

A group of modules organized in a folder (with an __init__.py file).

Library

A group of packages in one or more folders. NumPy, Pandas, etc. are libraries.

Always set your working directory in code at the start of a script — don't rely on whatever directory happens to be active. This makes scripts portable and predictable regardless of where they're launched from.

import os
os.getcwd()                           # see current directory
os.chdir(r"C:\MyProjects\Python")     # set working directory
# r"..." = raw string — backslashes treated literally

Creating a Module

▶

Creating a module is simple — write code in a .py file. Any .py file IS a module. Save it and import it elsewhere.

# Multiply.py  ← this IS the module
myVal = 3

def MyMultiply(num1, num2):
    """ Takes 2 numbers, multiplies them and returns the result """
    return num1 * num2

class Multiply1:
    pass

The triple-quoted string immediately after a function or class definition is called a docstring. Python treats it as documentation for that object. It appears in help() output and in IDE tooltips (Spyder's Object Explorer / intellisense). Always write docstrings — they make your code dramatically easier for others (and future you) to understand.

# Access the docstring
print(MyMultiply.__doc__)
# "Takes 2 numbers, multiplies them and returns the result"

help(MyMultiply)   # formatted help output

Importing Modules

▶ 
# Basic import — use module name as prefix
import Multiply
print(Multiply.myVal)             # 3
print(Multiply.MyMultiply(3, 4))  # 12
print(Multiply.MyMultiply.__doc__)

# Import with alias — shorten or clarify the name
import Multiply as Mult
print(Mult.myVal)
print(Mult.MyMultiply(3, 4))

# Import everything (generally NOT recommended)
from Modules import *
# Pollutes namespace — unclear where names come from
# OK for quick scripts, bad for production code

# Import from a subfolder
from Modules import Multiply   # relative path: Modules/ subfolder
print(Multiply.myVal)

Using an alias (import numpy as np, import pandas as pd) is standard practice for commonly used libraries — it reduces typing while keeping names clear. Avoid from module import * in production code because it floods the namespace with names whose origin becomes unclear, making debugging harder.

Import Paths

▶

Python first looks in the current active directory for the module file. For other locations, use sys.path to tell Python where to search.

LocationHow to Import
Same folderimport Multiply (just works)
Subfolderfrom Modules import Multiply
Parent foldersys.path.append("..") then import
Sibling of parentsys.path.append(r"..\ParentModules") then import
Absolute pathsys.path.append(r"C:\full\path") then import
import sys

# Parent folder
sys.path.append("..")
import MultiplyParent as Multiply

# Subfolder of parent
sys.path.append(r"..\ParentModules")
import Multiply

# Full absolute path
myPath = r"G:\My Drive\NNU\Python\ParentModules"
sys.path.append(myPath)
import Multiply

print(Multiply.myVal)
print(Multiply.MyMultiply(3, 4))
print(Multiply.MyMultiply.__doc__)

__name__ and __main__

▶

Python sets a special variable __name__ in every script. Its value depends on HOW the script is being run — directly, or imported by something else.

Run directly

__name__ == "__main__". Python treats this as the entry point.

Imported as module

__name__ == the module's filename (e.g., "Multiply"). Not "__main__".

def myfunc():
    print("Inside function")

if __name__ == "__main__":
    # This block only runs when this script is run DIRECTLY
    print("Program execution is direct")
    print("__name__ is:", __name__)   # "__main__"
    myfunc()
else:
    # This block runs when this script is IMPORTED elsewhere
    print("Running as a module from another program")
    print("__name__ is:", __name__)   # "scriptname"

This pattern is the Python equivalent of C's main() function — it gives your script a defined entry point. Without it, every statement at the top level runs whenever the file is imported, which can cause unwanted side effects. Always wrap your "run this script" code in if __name__ == "__main__": for any module you intend to also import elsewhere.

Logging

▶

The logging module tracks events as they occur — invaluable for debugging, auditing, and maintaining a historical record without watching the program run live. It's thread-safe and used by many third-party packages.

LevelWhen to use
DEBUGDetailed diagnostic info, only useful during development
INFOConfirmation things are working as expected
WARNINGSomething unexpected — program still works (default level)
ERRORA serious problem — function couldn't complete
CRITICALA very serious error — program may not continue
import logging

# Simple warning to screen
logging.warning("Watch out!")

# Log to a file with timestamps
logging.basicConfig(
    filename="mywarninglog.txt",
    level=logging.WARNING   # only WARNING and above get logged
)

logging.debug("Debug message")     # NOT logged (below WARNING)
logging.info("Info message")       # NOT logged (below WARNING)
logging.warning("Warning message") # logged
logging.error("Error message")     # logged
logging.critical("Critical message") # logged

# Capture the full stack trace on exception
import logging
num1 = 3
num2 = 0
try:
    result = num1 / num2
except Exception as e:
    # exc_info=True includes the full stack trace in the log
    logging.error("Displaying Exception", exc_info=True)

exc_info=True is critical for production debugging — it captures the full call stack (state of the program at the point of failure, including function calls and variable values). Without it, you only know that an error occurred but not where or why. Always use exc_info=True when logging exceptions in except blocks.

Assertions

▶

Assertions let you check conditions during development without adding temporary print statements you'll need to remove later. They can stay in the code permanently — they only trigger when the condition fails.

def mySum(myElements):
    assert len(myElements) != 0, "Empty list"   # condition, message
    return sum(myElements)

# Works fine
list1 = [1, 2, 3, 4, 5, 6]
print("Sum:", mySum(list1))   # 21

# Fails — triggers AssertionError
list2 = []
print("Sum:", mySum(list2))   # AssertionError: Empty list

# Square root with assertions
import math

def isNumber(n):
    try:
        float(n)
        return True
    except ValueError:
        return False

def safeSqrt(n):
    assert isNumber(n), "Input must be numeric"
    assert float(n) >= 0, "Input must be non-negative"
    return math.sqrt(float(n))

user_input = input("Enter a number: ")
print(safeSqrt(user_input))

Assertions differ from exceptions: assert is for developer errors (things that should never happen if the code is correct), while try/except is for runtime errors (things that might reasonably happen). Assertions can be disabled globally with the -O (optimize) flag when running Python — so never use them for input validation in production. Use them to document and verify your assumptions about your own code's internal state.

Packages

▶

A package is a folder containing related modules. Python requires an __init__.py file at the package root to recognize it as a package (it can be empty).

# Package structure example:
# Games/
#   __init__.py
#   Cricket/
#     __init__.py
#     othercountrybatsman.py
#     India/
#       __init__.py
#       indianbatsman.py
#   Football/
#     __init__.py
#     goalkeeper.py
#     forward.py

# Importing from packages
import Games.Cricket.othercountrybatsman
Games.Cricket.othercountrybatsman.name_othercountrybatsman()

import Games.Cricket.India.indianbatsman
Games.Cricket.India.indianbatsman.name_indianbatsman()

import Games.Football.goalkeeper
Games.Football.goalkeeper.name_goalkeeper()

# With alias for cleaner calls
import Games.Football.forward as fwd
fwd.name_forward()

The __init__.py file is what tells Python "this folder is a package, not just a directory." Without it, imports across subdirectories won't work. It can be completely empty — its presence is what matters. Set sys.path appropriately so Python can find the package root directory before importing.
