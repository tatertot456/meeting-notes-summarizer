What Is a View?

▶

A view is a stored query expression. It behaves like a table — it has a name, can be referenced by other queries, and returns rows and columns — but the data itself is stored in the underlying base tables, not in the view.

Views are used to:

  • Simplify complex relationships between tables into a clean, queryable surface
  • Prevent unauthorized access to data — users see only the columns and rows they need
  • Create a stable interface between underlying data structures and external applications
  • Simplify reporting by presenting data in the correct format
  • Filter records by restricting columns or rows returned

Views are the primary mechanism for hiding complexity and enforcing security boundaries at the database layer. If underlying table structures change, views can be updated to maintain the same interface for consumers — without requiring changes to application code.

Types of Views

▶

Standard Views

User-defined views that store only the query definition. Data is always retrieved live from base tables when the view is queried. No physical storage of data.

Indexed Views

Materialized views — the result set is computed and stored physically on disk. Faster for repeated complex queries but requires maintenance overhead on INSERT/UPDATE/DELETE.

Partitioned Views

A view onto a horizontally partitioned table — a large table split across multiple tables with identical structure. Can be local or distributed across servers.

System Catalog Views

Views onto internal SQL Server metadata. sys.views, sys.tables, sys.columns, etc. Organized by category: object views, schema views, linked server views.

Dynamic Management Views (DMVs)

Return internal SQL Server state information. Used to monitor health, diagnose problems, and tune performance. DMFs (Dynamic Management Functions) take parameters.

Information Schema Views

ISO-standard views for querying metadata. INFORMATION_SCHEMA.TABLES, INFORMATION_SCHEMA.COLUMNS, etc. More portable across database platforms than sys.* views.

System Views & DMVs

▶

SQL Server provides several categories of system views for querying metadata and monitoring the server:

-- List all views in a database
SELECT name, object_id, schema_id
FROM sys.views;

-- Get the definition of a view
SELECT OBJECT_DEFINITION(OBJECT_ID('Sales.vOrderDetails'));

-- List objects that depend on a view
SELECT referencing_entity_name
FROM sys.sql_expression_dependencies
WHERE referenced_entity_name = 'vOrderDetails';

-- DMV: check active connections
SELECT session_id, login_name, status
FROM sys.dm_exec_sessions;

-- Information schema (ISO standard)
SELECT TABLE_NAME, TABLE_TYPE
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_SCHEMA = 'Sales';

Compatibility views (backward compatible with SQL Server 2000 system tables) should NOT be used for new development. Use sys.* catalog views or INFORMATION_SCHEMA views instead. DMVs and DMFs together are called DMOs (Dynamic Management Objects) — they are invaluable for performance monitoring and troubleshooting.

Creating Views

▶

Views are created with CREATE VIEW. Several optional attributes control behavior:

-- Basic view
CREATE VIEW Sales.vCustomerOrders AS
SELECT c.custid, c.contactname, o.orderid, o.orderdate
FROM Sales.Customers AS c
JOIN Sales.Orders AS o ON c.custid = o.custid;

-- WITH ENCRYPTION: obfuscates the view definition
CREATE VIEW Sales.vSensitiveData
WITH ENCRYPTION AS
SELECT custid, creditlimit
FROM Sales.Customers;

-- WITH SCHEMABINDING: prevents changes to underlying tables
-- that would break the view (required for indexed views)
CREATE VIEW Sales.vOrderTotals
WITH SCHEMABINDING AS
SELECT o.custid, SUM(od.qty * od.unitprice) AS total
FROM Sales.Orders AS o
JOIN Sales.OrderDetails AS od ON o.orderid = od.orderid
GROUP BY o.custid;

-- WITH CHECK OPTION: prevents inserting rows that don't
-- match the view's WHERE clause
CREATE VIEW Sales.vActiveCustomers
WITH CHECK OPTION AS
SELECT custid, contactname, isactive
FROM Sales.Customers
WHERE isactive = 1;
OptionEffect
WITH ENCRYPTIONObfuscates view definition in system tables — limited protection
WITH SCHEMABINDINGBinds view to base table schema — prevents breaking changes. Required for indexed views.
WITH VIEW_METADATAReturns view metadata instead of base table metadata to client tools
WITH CHECK OPTIONEnsures INSERTs and UPDATEs through the view conform to its WHERE clause

Managing Views: Alter, Drop & Ownership

▶ 
-- Alter an existing view (preserves permissions)
ALTER VIEW Sales.vCustomerOrders
WITH ENCRYPTION AS
SELECT c.custid, c.contactname, o.orderid, o.orderdate, o.freight
FROM Sales.Customers AS c
JOIN Sales.Orders AS o ON c.custid = o.custid;

-- Drop a view (also drops permissions)
DROP VIEW Sales.vCustomerOrders;

-- Drop multiple views at once
DROP VIEW Sales.vView1, Sales.vView2, Sales.vView3;

ALTER VIEW preserves existing permissions on the view. DROP VIEW removes the view AND all associated permissions — use ALTER VIEW when you want to change the definition without affecting security settings.

Ownership chains matter for security:

  • If view and underlying tables have the same owner: a user granted access to the view can query it even without direct table permissions — ownership chain is intact
  • If view and tables have different owners: the ownership chain is broken — users need explicit permissions on the underlying tables too

This is why views are a powerful security tool: grant users access to the view only, and the underlying tables remain protected.

Updateable Views

▶

Data can be modified (INSERT, UPDATE, DELETE) through a view — but only under strict conditions:

  • The view must include columns from only one base table
  • Columns must directly reference base table columns — no expressions or computations
  • No aggregations — GROUP BY, HAVING, DISTINCT, or aggregate functions disqualify a view
  • The modifications must comply with all base table constraints (NULL/NOT NULL, PKs, FKs)
-- Updateable view: columns from one table, no computations
CREATE VIEW HR.vEmployeeContact AS
SELECT empid, lastname, firstname, phone
FROM HR.Employees;
-- Users can INSERT/UPDATE/DELETE through this view

-- WITH CHECK OPTION enforces view filter on modifications
CREATE VIEW HR.vActiveEmployees
WITH CHECK OPTION AS
SELECT empid, lastname, firstname, isactive
FROM HR.Employees
WHERE isactive = 1;
-- Attempting to INSERT isactive=0 will fail

WITH CHECK OPTION is critical for security — without it, a user could INSERT a row through the view that doesn't match the view's WHERE clause. After the insert, the row would be invisible through the view but would exist in the base table. WITH CHECK OPTION prevents this data leakage.

Performance Considerations

▶

Views themselves don't inherently hurt or help performance — how you design and use them determines their impact.

Dynamic Resolution

SQL Server merges the view definition into the outer query and creates a single optimized query plan. It only retrieves data actually needed — if you query 2 of 10 columns, SQL Server fetches only those 2.

Avoid SELECT *

Never use SELECT * in view definitions. When base tables change, SELECT * doesn't automatically update — and it retrieves columns that may not be needed.

Nested Views

A view calling another view calling another view creates ownership chain problems, poorly performing queries, and debugging nightmares. Avoid deep nesting.

Partitioned Views

Views onto horizontally partitioned tables. All member tables have identical structure. Benefits: faster querying and indexing by limiting the data range accessed.

Indexed Views

▶

An indexed view is materialized — the result set is computed once and stored physically on disk, like a table. This dramatically speeds up reads for complex aggregations at the cost of storage and write overhead.

Requirements for indexed views:

  • Must use WITH SCHEMABINDING
  • Cannot use SELECT *, subqueries, outer joins, or non-deterministic functions
  • Must have a unique clustered index created first
  • All referenced tables must use two-part naming (schema.table)
-- Step 1: Create view with SCHEMABINDING
CREATE VIEW Sales.vOrderTotals
WITH SCHEMABINDING AS
SELECT o.custid,
       COUNT_BIG(*) AS order_count,
       SUM(od.qty * od.unitprice) AS total
FROM Sales.Orders AS o
JOIN Sales.OrderDetails AS od ON o.orderid = od.orderid
GROUP BY o.custid;

-- Step 2: Create unique clustered index to materialize it
CREATE UNIQUE CLUSTERED INDEX IX_vOrderTotals
ON Sales.vOrderTotals(custid);

Indexed views are used two ways: directly (query the view by name and SQL Server uses the stored data) or indirectly (query the base tables and SQL Server's optimizer automatically uses the indexed view when beneficial). The optimizer can use indexed views even when you don't reference them explicitly.
