Introduction — What is Hashing?

▶

All prior search techniques (sequential, binary, BST) are comparison-based. Hashing takes a completely different approach: calculate the position directly from the key's value.

Hash Function h(K)

Transforms a key K into a table index. Ideal: O(1) access regardless of table size. Real world: approximate O(1) with occasional collisions.

Collision

When two different keys hash to the same position: h("abc") = h("acb") = 294 (ASCII sum). Collisions are inevitable — must be handled.

Perfect Hash Function

Maps every key to a unique position — no collisions. Requires knowing all keys in advance. Rare in practice.

Minimal Perfect Hash

Perfect hash that uses exactly n positions for n keys — no empty cells. Ideal but hard to construct.

The goal: reduce search from O(n) or O(log n) down to O(1). If h can transform any key into a unique index, we access data directly without any search at all. In practice, collisions prevent this ideal — but with good hash functions and collision resolution, average O(1) is achievable.

The division method is most common when key distribution is unknown. For string keys, XOR individual characters or XOR chunks of bytes (size = integer width) for better distribution than XORing individual bytes (limited to 0-127 range). The key tradeoff: more complex hash functions distribute better but take longer to compute.

Collision Resolution

▶

Open Addressing — find another empty slot in the same table when a collision occurs:

// Probing sequence: h(K), h(K)+p(1), h(K)+p(2), ...
// Each probed position = norm(h(K) + p(i)) = (h(K) + p(i)) mod TSize

// LINEAR PROBING — p(i) = i
// Try: h(K), h(K)+1, h(K)+2, ...
// Problem: PRIMARY CLUSTERING — filled cells near clusters fill faster,
// causing clusters to grow, dramatically reducing efficiency.

// QUADRATIC PROBING — p(i) = ±i²
// Try: h(K), h(K)+1, h(K)-1, h(K)+4, h(K)-4, ...
// Reduces primary clustering BUT creates SECONDARY CLUSTERING:
// Same probe sequence for any key that hashes to same position.
// TSize should be prime of form 4j+3 to ensure all positions are checked.

// DOUBLE HASHING — use two hash functions
// Probe sequence: h(K), h(K)+hp(K), h(K)+2·hp(K), ...
// Different hp(K) for each key → different probe sequences → no secondary clustering
// TSize must be prime (all positions reachable)
// hp(K) = i·h(K) + 1 is a common choice

Linear probing is simplest but creates primary clusters — once a cluster forms it tends to grow. Quadratic probing eliminates primary clustering but secondary clustering still occurs (same starting position → same probe sequence). Double hashing eliminates both by making the probe step key-dependent. As the table fills past ~70% occupancy, all open-addressing methods degrade rapidly.

Separate Chaining — store colliding keys in linked lists:

// Each table slot holds a pointer to a linked list // Collisions: append to the list at that slot // Table NEVER overflows (lists grow dynamically) // Fast for short chains; degrades with long chains // Use ordered lists or self-organizing lists for better performance // Space overhead: one pointer per table slot + node pointers // COALESCED HASHING — hybrid of chaining + open addressing // Colliding key goes in LAST AVAILABLE TABLE SLOT // index stored with existing key to link the chain // Less space than pure chaining // Overflow keys use a "cellar" — dedicated overflow area at end of table

Bucket Addressing — each table slot holds multiple items:

// Each slot = a bucket capable of storing b items
// Collision within bucket: bucket is full → overflow handling
// Options: linear probing to next bucket, OR separate overflow area
// Overflow area can use chaining (pointer stored in bucket)
// Good for disk-based storage where one disk read retrieves whole bucket

Deletion

▶

Deletion in chaining is simple — just remove the node from the linked list. Open addressing is trickier.

// PROBLEM with naive deletion in open addressing:
// Linear probing table: A4 B4 (B4 placed at position 5 due to collision)
// Delete A4 → position 4 is empty
// Search for B4: hash to position 4, find empty → conclude B4 not in table!
// WRONG — B4 is at position 5 but we stopped too early.

// SOLUTION: Mark deleted slots with a "tombstone" (DELETED flag)
// Searches: continue past tombstone (don't stop at DELETED)
// Insertions: can reuse tombstone positions
// Search finds key OR reaches truly EMPTY slot (never inserted)

// PROBLEM with tombstones:
// After many deletions, table fills with tombstones → slow searches
// SOLUTION: Periodic rehashing to purge tombstones
// Move valid keys to freed slots, mark remaining as EMPTY

The key distinction: EMPTY (never used) stops a search; DELETED (tombstone) does not. Without tombstones, deleting a key can make other keys unreachable. With too many tombstones, search performance degrades because every tombstone must be examined. Chaining avoids this entirely — deletion is a standard linked list removal.

Perfect Hash Functions

▶

When data is known in advance (compiler reserved words, dictionaries), a perfect hash function can be constructed. Only 1 in a million random functions is perfect for 50 elements in a 100-slot table.

Cichelli's Method (1980) — for small sets of reserved words:

// Form: h(word) = (length(word) + g(firstLetter) + g(lastLetter)) mod TSize
// Goal: find values for g() such that h() produces unique values for all words
// g() values don't need to be unique — just h() values must be

// Algorithm:
// 1. Count first and last letter occurrences across all words
// 2. Order words by letter frequency (most frequent letters first)
// 3. Search: assign g-values 0..Max to letters, backtrack on collision

// Example (Nine Muses):
// Letter counts: A(3), C(2), E(6), M(1), O(2), P(1), T(2), U(1)
// Order: Euterpe, Calliope, Erato, Terpsichore, Melpomene, Thalia, Clio, Polyhymnia, Urania
// After backtracking: A=C=E=O=M=T=0, P=2, U=4 → minimal perfect hash!

Cichelli's method is exponential in worst case and doesn't guarantee finding a perfect hash. It works well for small word sets (<~50 words) like compiler keywords. Extensions by Sebesta-Taylor (1986) and Haggard-Karplus (1986) use more sophisticated formulas. The FHCD algorithm (Fox, Heath, Chen, Daoud, 1992) is more powerful, using three hash functions and a dependency graph to construct g() — but g must be stored as a lookup table, which can be large for big datasets.

Rehashing & Cuckoo Hashing

▶

Rehashing — when a table becomes full or too slow, allocate a larger table and re-insert all items:

  • New table size: doubled, nearest prime to doubled, etc.
  • New hash function uses new TSize
  • All items from old table rehashed to new table
  • Old table discarded
  • Drawback: entire system halts during rehash — can be unacceptable for real-time systems

Cuckoo Hashing (Pagh & Rodler, 2001) — uses TWO tables and TWO hash functions:

insert(K):
    if K already in T1[h1(K)] or T2[h2(K)]: return   // already present
    for i = 0 to maxLoop-1:
        swap(K, T1[h1(K)])   // K goes in, displaced key comes out
        if K is null: return  // empty slot found — done!
        swap(K, T2[h2(K)])   // try second table
        if K is null: return
    rehash()                  // loop detected — expand and retry
    insert(K)

// Search: at most TWO lookups — check T1[h1(K)] and T2[h2(K)]
// Guaranteed O(1) worst-case lookup!

// Example:
// Insert 2 → T1[h1(2)] = 2
// Insert 7 → h1(7) = h1(2), displace 2 to T2[h2(2)], place 7 in T1
// Insert 12 → h1(12) = h1(7), displace 7 to T2[h2(7)] = T2[h2(2)],
//             displace 2 back to T1... potential loop → rehash!

Cuckoo hashing's key insight: like the cuckoo bird evicting eggs from other nests, inserting a key evicts the current occupant who must find a new home. Search is guaranteed O(1) — check exactly two positions. Insertion is amortized O(1) but may trigger rehash. The name comes from the cuckoo bird's habit of laying eggs in other birds' nests, displacing the original eggs.

Extendible Hashing — Dynamic File Growth

▶

Standard rehashing stops all processing while rebuilding. Extendible hashing (Fagin et al., 1979) allows local expansion with no full reorganization — ideal for disk-based files.

// Structure:
// - Directory: expandable table of pointers to buckets
// - Buckets: fixed-size pages on disk
// - Each bucket has a LOCAL DEPTH (# of bits used to route to it)
// - Directory has a GLOBAL DEPTH d (# of bits used to index directory)

// Hash produces PSEUDOKEY (bit string): h(K) = 01011...
// Use leftmost d bits to index directory → find bucket pointer

// INSERTION — bucket has room: insert normally
// INSERTION — bucket is full:
//   if local_depth < global_depth:
//     Split bucket, redistribute keys
//     Update HALF the directory pointers (only those pointing to old bucket)
//   if local_depth == global_depth:
//     DOUBLE the directory (depth++)
//     Split bucket, redistribute keys
//     newdir[2i] = olddir[i], newdir[2i+1] = olddir[i] for all i
//     Update pointers for the split bucket

// Example with depth 2:
// Directory: 00→b00, 01→b01, 10→b10, 11→b10 (b10 has local depth 1)
// Key with h-value 11001 arrives, first 2 bits = 11 → bucket b10
// b10 overflows → split into b10 (local depth 2) and b11 (local depth 2)
// Only directory[11] pointer needs updating → fast!

Extendible hashing's key advantage: only the directory changes on overflow — not the entire file. Since directories are usually kept in main memory, expansion is cheap. Drawback: directory can double in size suddenly (when a bucket with local_depth = global_depth splits). With many small buckets, the directory can grow large enough to need disk storage, slowing access significantly.

Linear Hashing — No Directory Needed

▶

Linear hashing (Litwin, 1980) achieves dynamic expansion WITHOUT any directory or index. Buckets split in a predetermined linear order, tracked by a single pointer.

// TWO hash functions at each level:
// h_level(K)   = K mod (TSize × 2^level)    // for unsplit buckets
// h_level+1(K) = K mod (TSize × 2^(level+1)) // for already-split buckets

// Split pointer 'split' indicates WHICH bucket splits NEXT (always linear)
// Splitting is triggered by load factor exceeding a threshold (e.g., 80%)

// INSERT algorithm:
// 1. Compute h_level(K)
// 2. If result < split: use h_level+1(K) instead (bucket already split)
// 3. Insert into computed bucket (use overflow area if full)
// 4. If load factor > threshold: split bucket[split], increment split
//    If split reaches end of level: split = 0, level++, define h_level+2

// Example (TSize=3, threshold=80%):
// Level 0: h0(K) = K mod 3, buckets: 0,1,2
// Load exceeds 80% → split bucket 0 → new bucket 3
//   h1(K) = K mod 6: keys with K mod 6 = 0 stay, K mod 6 = 3 go to bucket 3
// split = 1, next overflow → split bucket 1 → new bucket 4, etc.
// When split reaches 3: level=1, split=0, new h2(K) = K mod 12

Linear hashing vs. extendible hashing: linear hashing is simpler (no directory = less memory, faster) and splits happen predictably in order. But splitting is predetermined — a lightly loaded bucket may split before a heavily loaded one. An overflow area is required because of this fixed split order. For very large files, linear hashing's lack of directory overhead makes it significantly faster than extendible hashing.
