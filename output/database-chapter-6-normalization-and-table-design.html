Tables, Keys & Schemas

▶

Relational databases store data in tables (relations) — collections of named columns containing zero or more rows. Each row represents a single instance of an entity. Rows have no guaranteed order.

Primary Key

Uniquely identifies each row. Must be unique, cannot be NULL, should not change. Can be composite (multiple columns). Natural keys use real data; surrogate keys are system-generated numbers.

Foreign Key

Holds the primary key from another table — creates relationships. Self-references permitted. Referenced rows cannot be deleted without cascading options. Multiple FKs allowed per table.

Surrogate Keys

IDENTITY auto-increments. SCOPE_IDENTITY() returns last value in current session. @@IDENTITY returns last value across all sessions. SEQUENCE creates a shared counter across multiple tables.

Schemas

Containers for database objects (tables, views, procedures, functions). Security boundaries — permissions granted at schema level apply to all objects within. Always reference as Schema.Object.

GUID (uniqueidentifier) is highly selective but costly on updates/deletes. Never store GUIDs as strings — use the uniqueidentifier type. Generate new values with NEWID(). IDENTITY cannot be used with uniqueidentifier.

Data Integrity

▶

Data integrity is enforced across multiple application layers — UI tier, middle tier, and data tier. All three layers may enforce rules, requiring synchronization.

Domain Integrity

Defines the allowed values in columns. Enforced by data types, nullability, CHECK constraints, and DEFAULT constraints.

Entity Integrity

Primary key uniquely identifies each row. Enforced by PRIMARY KEY constraint — values must be unique and NOT NULL.

Referential Integrity

Defines relationships between tables. Enforced by FOREIGN KEY constraints. Cascading options control what happens on UPDATE or DELETE.

Triggers

Code executed automatically when data is modified. Used for complex business rules that can't be expressed as simple constraints.

Cascading referential integrity options:

OptionBehavior on Parent Delete/Update
NO ACTION (default)Return error and rollback — child must be removed first
CASCADEAutomatically delete/update matching child rows
SET NULLSet child FK columns to NULL
SET DEFAULTSet child FK columns to their default values

Constraints

▶ 
-- PRIMARY KEY
CREATE TABLE Orders (
    OrderID   INT IDENTITY(1,1) NOT NULL,
    CustomerID INT NOT NULL,
    OrderDate  DATE NOT NULL,
    CONSTRAINT PK_Orders PRIMARY KEY (OrderID)
);

-- FOREIGN KEY with CASCADE
ALTER TABLE Orders
ADD CONSTRAINT FK_Orders_Customers
    FOREIGN KEY (CustomerID)
    REFERENCES Customers(CustomerID)
    ON DELETE CASCADE;

-- UNIQUE constraint (allows one NULL)
ALTER TABLE Employees
ADD CONSTRAINT UQ_Email UNIQUE (Email);

-- DEFAULT constraint
ALTER TABLE Orders
ADD CONSTRAINT DF_OrderDate DEFAULT GETDATE() FOR OrderDate;

-- CHECK constraint (only rejects FALSE — NULL passes!)
ALTER TABLE Products
ADD CONSTRAINT CK_Price CHECK (Price > 0);

-- Composite CHECK (table-level, references multiple columns)
ALTER TABLE Scheduling
ADD CONSTRAINT CK_Dates CHECK (EndDate >= StartDate);

CHECK constraints only reject FALSE. NULL evaluates to UNKNOWN — not FALSE — so NULL passes a CHECK constraint. Always consider nullability alongside CHECK constraints. Give constraints meaningful names — they must be named to be suspended, altered, or dropped later.

Constraints can be suspended to improve bulk load performance or to add constraints to tables with existing valid data. UNIQUE constraints allow one NULL value (unlike PRIMARY KEY which allows none).

Normalization — Why & How

▶

Normalization evaluates and corrects table structures to minimize data redundancy and prevent data anomalies. Invented by Edgar F. Codd.

A well-normalized table:

  • Represents a single subject
  • Has only one value per row/column intersection (atomic values)
  • Has no data stored unnecessarily in more than one table
  • Has all non-key attributes dependent on the primary key
  • Has no insertion, update, or deletion anomalies

Normalized data is smaller in size, faster to insert/update, and more flexible to modify. The normalization process works one relation at a time, identifying dependencies and progressively decomposing tables until 3NF is achieved. Higher forms (BCNF, 4NF) are rarely needed in business environments.

Key dependency concepts:

  • Functional dependence: A determines B if each value of A determines exactly one value of B
  • Partial dependency: A non-key attribute depends on only PART of a composite primary key — violates 2NF
  • Transitive dependency: A non-key attribute depends on another non-key attribute — violates 3NF

Normal Forms: 1NF through 4NF

▶
Normal FormRequirementFixes
1NFPrimary key defined, no repeating groups, atomic values onlyRepeating groups, multi-valued cells
2NFIn 1NF + no partial dependenciesAttributes that depend on only part of a composite key
3NFIn 2NF + no transitive dependenciesNon-key attributes that depend on other non-key attributes
BCNFEvery determinant is a candidate keySpecial case when table has multiple overlapping candidate keys
4NFIn 3NF + no independent multivalued dependenciesRows with multiple independent multi-valued facts about an entity

How to convert:

1NF → 2NF

Only needed when PK is composite. Create new tables for attributes that depend on only PART of the PK. Link with foreign key. If PK is single-column, table is already in 2NF.

2NF → 3NF

Find non-key attributes that determine other non-key attributes. Move them to a new table. The determining attribute becomes the PK of the new table, linked back by foreign key.

BCNF

A special case of 3NF — violated only when the table has more than one candidate key and a non-candidate-key attribute determines part of a candidate key.

4NF

No row may contain two or more independent multivalued facts. Split into separate tables — one for each independent multivalued relationship.

If a 1NF table has a SINGLE-column primary key, it's automatically in 2NF (there can be no partial dependencies on a single-column key). Focus 2NF analysis on tables with composite keys.

Denormalization

▶

Denormalization intentionally introduces redundancy to improve read performance. It produces a lower normal form — a deliberate trade-off between data redundancy and query speed.

Why normalize first, then denormalize? Normalization creates the correct logical design. Denormalization is then applied surgically where performance measurement proves it's needed — not as a default approach.

Redundant Data

Store ZIP and CITY together even though ZIP determines CITY. Avoids an extra join. Application validates city based on zip via dropdown.

Derived Data

Store student classification (Freshman/Sophomore etc.) even though it's derivable from credit hours. Avoids recalculating. Application keeps it updated.

Pre-aggregated Data

Store GPA in the STUDENT table even though it's calculable from ENROLL/COURSE. Computed once on grade entry, not on every report query.

Temporary Tables

Use a temp denormalized table to hold report data when a tabular report requires column pivots impossible in plain SQL. Table deleted after report generation.

Data warehouse tables should be denormalized — no frequent modifications, so redundancy costs less. OLTP tables should be normalized — frequent inserts/updates/deletes, where redundancy causes anomalies and locking problems.

Indexes

▶

Without an index, SQL Server performs a table scan — reading every page. A table with no indexes is called a heap. Indexes use a B-tree structure to dramatically reduce rows read for queries.

Heap

No defined order. Inserts go to first available space. SELECTs must scan the entire table. Good for: bulk loads with no concurrent queries.

Clustered Index

Logically orders the rows by the clustering key. Only ONE per table. Pages are linked in logical sequence. Best for range queries on the clustering key.

Nonclustered Index

Separate structure pointing to data rows. Up to 999 per table. Consumes extra space. Speeds up SELECTs for covered columns but slows INSERT/UPDATE/DELETE.

Composite Index

Index on multiple columns. Index on (A,B) is NOT the same as (B,A). More selective than single-column indexes. Each component can be ASC or DESC.

Index data type considerations:

Data TypeBenefitsNegatives
NumericSmall size, more values per page, fastSmall types = high density
CharacterSearched frequently, better than heapSlower than numeric, prone to fragmentation
DateSmall size, fastSmall types = high density
GUIDHighly selective, fast readsPoor performance on updates/deletes
BITExtremely smallVery dense — low selectivity

Selectivity (few rows returned / total rows) = good index. Density (many duplicates) = poor index. Fragmentation occurs when page splits happen on INSERT — detect with sys.dm_db_index_physical_stats. SQL Server automatically creates and updates statistics on indexes — leave auto-creation and auto-update enabled.

Data-Modeling Checklist

▶

Use this checklist to verify an ERD meets minimum design requirements:

Business Rules

Document and verify all rules with end users. Each rule should be written precisely, dated, and signed off by an approving authority. Rules identify entities, attributes, relationships, and constraints.

Naming Conventions

Entity names: nouns, short, meaningful, unique in model. Attribute names: unique within entity, use entity abbreviation as prefix, use _ID/_NUM/_CODE suffix for PKs, no reserved words, no spaces or special characters.

Entities

Each entity represents a single subject. All entities should be in 3NF or higher (justify anything below). PK clearly defined and supports the selected data granularity.

Attributes

Simple and single-valued (atomic). Document defaults, constraints, synonyms. Derived attributes clearly identified. No redundancy unless required for performance or history. All non-key attributes fully dependent on PK.

Relationships

Clearly identify participants. Define participation, connectivity, and document cardinality. Relationship names should be active or passive verbs.

ER Model

Validate against inserts, updates, and deletions. Minimize redundancy (single-place updates). Conform to the minimal data rule: all that is needed is there, and all that is there is needed.

Design principles to remember: Use best practices. Stay consistent. Plan for change. Normalization is a tool — understand when to apply it and when denormalization is justified by real measurement, not assumption.
