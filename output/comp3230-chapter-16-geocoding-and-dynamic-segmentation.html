Geocoding & Reference Databases

▶

Geocoding converts text-based postal address data into geographic coordinates (longitude/latitude pairs). Address geocoding — plotting street addresses as point features — is the most common type. Locations are determined by linear interpolation within the address range of a matched street segment.

Geocoding reference database — required for address matching. Contains:

  • Street name for each segment
  • Address ranges (left and right side, low and high numbers)
  • ZIP codes

Historically derived from TIGER/Line files (US Census Bureau). Now also available from commercial providers (TomTom, HERE/NAVTEQ, Sanborn CitySets). Commercial databases typically have better positional accuracy than TIGER/Line files.

Linear interpolation assumes addresses are evenly distributed within a segment's address range. If a segment runs from 100–200, address 150 is placed at the midpoint. An alternative is a "location of address" database where actual x,y coordinates of building centroids or footprints are stored — far more accurate but requires more data collection effort.

The Address Matching Process

▶

Address matching consists of three phases:

1. Preprocessing

Parsing the input address into components (number, street name, type, directional, ZIP). Address standardization — normalize abbreviations, spelling variations, and formatting to match the reference database format.

2. Matching

The geocoding engine compares the parsed address against the reference database. A scoring system quantifies match quality. Matching conditions can be relaxed to improve hit rate — expect to run the process more than once.

3. Plotting

Matched addresses are plotted as point features using linear interpolation within the matched street segment's address range. Side offset and end offset options refine point placement.

Quality, Scoring & Offset Options

▶

Hit rate — percentage of input addresses successfully matched. The primary quality metric for geocoding:

Hit RateAssessment
60%Unacceptable for crime mapping and analysis
85%Statistically derived minimum acceptable rate
95%+Required for competitive location-based services

To achieve high hit rates: use a current, accurate reference database AND validate/standardize input addresses before geocoding. Positional accuracy (how close geocoded points are to the true location) is a secondary quality measure beyond hit rate.

Offset plotting options — refine point placement to avoid all points stacking on the street centerline:

Side Offset

Moves the geocoded point left or right away from the street centerline — places points on the correct side of the street. Improves visual clarity.

End Offset

Moves the geocoded point away from the endpoint of a street segment — prevents points from stacking at intersections.

Applications of Geocoding

▶

Location-Based Services (LBS)

Mobile apps using your current location to find nearby services, navigate, or deliver location-targeted content. Requires 95%+ accuracy for competitive applications.

Business Applications

Customer mapping, market analysis, service territory design, site selection, delivery routing. Geocoding customer databases enables spatial analysis of customer distribution.

Wireless Emergency Services (E911)

Automatically locating mobile callers during emergencies. Accuracy is life-critical. Requires precise reference databases and real-time positioning.

Crime Mapping & Analysis

Plotting crime incidents to analyze spatial patterns, identify hotspots, allocate resources. Minimum 85% hit rate required; 60% is considered unacceptable.

Public Health

Mapping disease cases, environmental health hazards, healthcare access. Spatial clustering analysis requires accurate geocoding to identify true patterns vs. geocoding artifacts.

Dynamic Segmentation

▶

Dynamic segmentation computes the location of events along a route dynamically using a linear measurement system — without physically splitting the underlying line features.

Route

A linear feature with a linear measurement system (measure values — "m" values) stored with its geometry. A route links a series of line segments together and has a unique Route ID. Measure values increase continuously along the route.

Events

Linearly referenced data that occur along routes. Stored in event tables referencing Route ID and measure values. Do NOT require physical splitting of the route geometry.

Point Events

Occur at a single measure value (one location). Examples: accidents, stop signs, bridges, mile markers. Stored with one measure value per event.

Line Events

Cover a portion of a route between two measure values. Examples: pavement condition, speed limit zones, road surface type. Stored with from-measure and to-measure values.

The power of dynamic segmentation: the SAME route geometry can have many different event tables overlaid simultaneously — speed limits, pavement conditions, accident history, rest area locations — without creating separate copies of the road network for each attribute. Attributes reference the route by measure value, not by splitting the geometry.

Creating Routes & Events

▶

Creating routes — two methods:

Interactive Method

Digitize or select existing line segments to build the route. User manually chooses which segments constitute each route. Good for custom or non-standard routes.

Conversion Method

Create routes from all linear features or from features selected by a query. Efficient for converting existing road networks (e.g., all Interstate highway features → route feature class).

Types of routes:

  • Simple route — follows one direction, no complications
  • Combined route — joined with another route (e.g., two highways share a segment)
  • Split route — subdivides into two routes (e.g., I-84 splitting at Emigrant Hill)
  • Looping route — intersects itself (a circuit in graph theory). Must be divided into parts for proper measure assignment — a loop cannot have unique measure values without subdivision.

Creating event tables — two methods:

  • Create from an existing table that already has Route ID and measure values
  • Create by locating point or polygon features along a route (e.g., overlay a polygon layer with a route layer → linear event table with from/to measures for each polygon segment)

Applications of Dynamic Segmentation

▶

Data Management

Transportation departments manage speed limits, rest areas, bridges, pavement conditions along highway routes — all referencing the same route geometry without duplication.

Data Display

Once an event table is linked to a route it is georeferenced and can be used as a feature layer. Example: display highway segments with speed limit 70 mph in a distinct symbol.

Data Query

Attribute AND spatial queries on event tables and their layers. Point query on a route returns Route ID, x,y coordinates, and the measure (m) value at that location plus route beginning and ending measures.

Data Analysis

Routes and events can be used as inputs for analysis such as overlay. Example: overlay a slope raster with a stream route to analyze slope conditions along the route. Combine accident events with pavement condition events to study relationships.
