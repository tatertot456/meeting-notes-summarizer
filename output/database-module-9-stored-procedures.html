What Is a Stored Procedure?

▶

When applications interact with SQL Server, they have two ways to execute T-SQL code:

  • Direct execution: every statement issued individually by the application
  • Stored procedures: groups of statements stored on the server and called by name

Stored procedures are similar to procedures or methods in other programming languages:

  • Can accept input parameters
  • Can produce output parameters
  • Can return sets of rows
  • Executed using the EXECUTE statement
  • Can be written in T-SQL or managed code (.NET)

Two categories of system stored procedures: System SPs (T-SQL code in master database, used for administration) and Extended SPs (references to external DLLs that extend SQL Server functionality). Always use two- or three-part naming when executing stored procedures to avoid unnecessary object resolution searches.

Benefits of Stored Procedures

▶

Enhanced Security

Users can be granted permission to EXECUTE a stored procedure without having direct permissions on the underlying tables. The SP acts as a security boundary — users can only do what the procedure allows.

Modular Programming

Create once, call many times from many applications. Logic is centralized — change it in one place and all callers benefit immediately.

Performance

A single EXECUTE statement sent over the network can run hundreds of lines of T-SQL. Execution plans are cached and reused — avoiding recompilation on each call.

Delayed Binding

A stored procedure can reference objects that don't exist yet. This allows flexible object creation order and avoids circular dependency issues during deployment.

Creating, Altering & Dropping Stored Procedures

▶ 
-- Create a stored procedure
CREATE PROCEDURE Reports.GetCustomerOrders
AS
BEGIN
    SET NOCOUNT ON;
    SELECT c.custid, c.contactname, o.orderid, o.orderdate
    FROM Sales.Customers AS c
    JOIN Sales.Orders AS o ON c.custid = o.custid;
END;

-- Execute it
EXECUTE Reports.GetCustomerOrders;
-- or shorthand:
EXEC Reports.GetCustomerOrders;

-- Alter (preserves existing permissions)
ALTER PROCEDURE Reports.GetCustomerOrders
AS
BEGIN
    SET NOCOUNT ON;
    SELECT c.custid, c.contactname, o.orderid, o.orderdate, o.freight
    FROM Sales.Customers AS c
    JOIN Sales.Orders AS o ON c.custid = o.custid;
END;

-- SQL Server 2016 SP1+: CREATE OR ALTER
CREATE OR ALTER PROCEDURE Reports.GetCustomerOrders
AS BEGIN ... END;

-- Drop a stored procedure
DROP PROCEDURE Reports.GetCustomerOrders;

-- View all stored procedures in database
SELECT name, object_id, create_date, modify_date
FROM sys.procedures;

CREATE PROCEDURE must be the only statement in its batch. Statements NOT permitted inside stored procedures include: CREATE/ALTER FUNCTION, CREATE/ALTER PROCEDURE, CREATE/ALTER VIEW, CREATE/ALTER TRIGGER, USE databasename, SET SHOWPLAN, SET PARSEONLY, CREATE SCHEMA, CREATE RULE.

Error Handling

▶

Always include error handling in stored procedures using the TRY…CATCH construct:

CREATE PROCEDURE Sales.InsertOrder
    @custid INT,
    @orderdate DATE
AS
BEGIN
    SET NOCOUNT ON;
    BEGIN TRY
        INSERT INTO Sales.Orders (custid, orderdate)
        VALUES (@custid, @orderdate);
    END TRY
    BEGIN CATCH
        -- Capture error details
        SELECT
            ERROR_NUMBER()    AS ErrorNumber,
            ERROR_SEVERITY()  AS ErrorSeverity,
            ERROR_STATE()     AS ErrorState,
            ERROR_PROCEDURE() AS ErrorProcedure,
            ERROR_LINE()      AS ErrorLine,
            ERROR_MESSAGE()   AS ErrorMessage;

        -- Re-throw the error to the caller
        THROW;
    END CATCH
END;
Error FunctionReturns
ERROR_NUMBER()The error number
ERROR_SEVERITY()The severity level
ERROR_STATE()The error state number
ERROR_PROCEDURE()Name of the SP where error occurred
ERROR_LINE()Line number where error occurred
ERROR_MESSAGE()The full error message text

Transaction Handling

▶

Use explicit transactions inside stored procedures to ensure all-or-nothing execution. Always pair with TRY…CATCH so failures trigger a ROLLBACK:

CREATE PROCEDURE Sales.TransferOrder
    @fromCustID INT,
    @toCustID INT,
    @orderID INT
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT ON;   -- auto-rollback on error

    BEGIN TRY
        BEGIN TRANSACTION;

        UPDATE Sales.Orders
        SET custid = @toCustID
        WHERE orderid = @orderID AND custid = @fromCustID;

        -- More related operations...

        COMMIT TRANSACTION;
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;

        THROW;  -- re-raise to caller
    END CATCH
END;

@@TRANCOUNT tracks the nesting level of BEGIN TRANSACTION calls. Each BEGIN increments it; each COMMIT decrements it. ROLLBACK sets it to 0. Always check @@TRANCOUNT before rolling back — rolling back when there's no active transaction throws an error. SET XACT_ABORT ON automatically rolls back the transaction if any statement raises an error.

Parameters: Input & Output

▶

Parameterized stored procedures have three component types: input parameters, output parameters, and return values.

-- Input parameters with defaults
CREATE PROCEDURE Reports.GetOrdersByDate
    @startdate DATE = '2000-01-01',   -- default value
    @enddate   DATE = NULL
AS
BEGIN
    SET NOCOUNT ON;

    -- Validate parameters early
    IF @enddate IS NULL SET @enddate = GETDATE();

    SELECT orderid, custid, orderdate
    FROM Sales.Orders
    WHERE orderdate BETWEEN @startdate AND @enddate;
END;

-- Call with named parameters (order doesn't matter)
EXEC Reports.GetOrdersByDate
    @startdate = '2024-01-01',
    @enddate   = '2024-12-31';

-- Output parameters
CREATE PROCEDURE Reports.GetOrderCount
    @custid     INT,
    @ordercount INT OUTPUT     -- OUTPUT keyword required
AS
BEGIN
    SET NOCOUNT ON;
    SELECT @ordercount = COUNT(*)
    FROM Sales.Orders
    WHERE custid = @custid;
END;

-- Execute with OUTPUT parameter
DECLARE @cnt INT;
EXEC Reports.GetOrderCount
    @custid     = 1,
    @ordercount = @cnt OUTPUT;  -- OUTPUT required here too
SELECT @cnt AS OrderCount;

OUTPUT must be specified in both the procedure definition AND when calling the procedure. Parameters can be passed by position (in order) or by name (using @paramname = value) — named parameters are preferred for clarity and allow you to skip optional parameters.

Parameter Sniffing & Performance

▶

SQL Server generates and caches an execution plan the first time a stored procedure runs — using the parameter values from that first call. Subsequent calls reuse the cached plan even if different parameters would benefit from a different plan. This is called parameter sniffing.

Usually beneficial — but problematic when the optimal plan varies significantly between parameter values (e.g., a query for a common value vs. a rare value).

WITH RECOMPILE

Forces a new execution plan every time the procedure runs. Use when parameter values vary widely and different plans are needed per call.

sp_recompile

Marks a procedure so its plan is discarded next time it runs. Useful for forcing a one-time recompile after statistics change.

EXEC WITH RECOMPILE

Recompiles only for the current execution without permanently changing the procedure.

OPTION (OPTIMIZE FOR)

Hints the optimizer to use a specific parameter value when generating the plan — useful when you know a "typical" value produces the best general plan.

-- Recompile every execution
CREATE PROCEDURE Reports.GetOrders @custid INT
WITH RECOMPILE
AS BEGIN ... END;

-- Optimize for a specific value
SELECT * FROM Sales.Orders
WHERE custid = @custid
OPTION (OPTIMIZE FOR (@custid = 1));

Controlling Execution Context

▶

SQL Server uses two security tokens: a login token (server-level) and a user token (database-level). The EXECUTE AS clause allows a stored procedure to impersonate a different security context.

-- Impersonate a database user
CREATE PROCEDURE Reports.SecureReport
WITH EXECUTE AS 'ReportUser'
AS
BEGIN
    -- Executes as if ReportUser is running this code
    SELECT orderid, custid FROM Sales.Orders;
END;

-- Impersonate a login (server-level)
EXECUTE AS LOGIN = 'sa';
-- ... do work ...
REVERT;  -- return to original context

-- View current security context
SELECT * FROM sys.login_token;  -- login-level tokens
SELECT * FROM sys.user_token;   -- user-level tokens

EXECUTE AS is powerful — it allows granting users access to functionality without granting direct access to underlying objects. A user can EXECUTE the procedure (which runs as a privileged account) without having any permissions on the tables. Always REVERT after impersonation when done manually.

Guidelines & Best Practices

▶
  • Always use SET NOCOUNT ON — suppresses "rows affected" messages that add network traffic and can confuse some clients
  • Qualify all object names with schema prefix inside procedures (Sales.Orders not just Orders)
  • Never use sp_ prefix for user-defined procedures — SQL Server searches master database first for sp_ prefixed names, causing unnecessary overhead
  • One procedure per task — keep stored procedures focused on a single responsibility
  • Validate parameters early in the procedure body before doing any work
  • Use consistent SET options — SET NOCOUNT ON, SET XACT_ABORT ON where appropriate
  • Check @@NESTLEVEL if nesting procedures — maximum nesting is 32 levels
  • Use return codes to communicate execution outcomes to callers (0 = success by convention)
  • Check dependencies using sys.sql_expression_dependencies before dropping objects

WITH ENCRYPTION obfuscates the stored procedure definition stored in SQL Server system tables — providing limited protection for proprietary logic. It is generally not recommended as a security measure since it can be defeated and makes debugging and maintenance harder.
