Static (Class) Variables

▶

Static variables are shared between ALL instances of a class — one copy total, regardless of how many objects exist. Unlike instance variables where each object gets its own copy.

# 5 ways to declare a static variable

# 1. In the class body (most common)
class Student:
    universityName = "NNU"   # class/static variable
    def __init__(self, myName, gradeLevel):
        self.myName = myName
        self.gradeLevel = gradeLevel

# 2. Inside __init__ using class name
Student.universityName = "NNU"

# 3. Inside an instance method using class name
def SetNNU(self):
    Student.universityName = "NNU"

# 4. Inside a class method using cls
@classmethod
def SetNNU(cls):
    cls.universityName = "NNU"

# 5. Inside a static method using class name
@staticmethod
def SetNNU():
    Student.universityName = "NNU"

Shared behavior — changes affect ALL instances:

student1 = Student("Rufus McGee", "Sophomore")
student2 = Student("Alice McGee", "Junior")

print(student1.universityName)   # NNU
print(student2.universityName)   # NNU

Student.universityName = "Northwest Nazarene University"

print(student1.universityName)   # Northwest Nazarene University
print(student2.universityName)   # Northwest Nazarene University — both changed!

# Delete a static variable
del Student.universityName   # using del keyword

Changing a static variable through the CLASS NAME (Student.universityName) updates it for all instances. If you accidentally assign through an INSTANCE (student1.universityName = "X"), Python creates a new INSTANCE variable on that object instead of changing the class variable — a common gotcha. Always use the class name or cls to modify class variables.

class Major: myConcentration = "Data Science" # class variable def __init__(self, myName): self.myName = myName @classmethod def Concentration(cls, degree): # cls refers to the class itself (like self refers to the instance) print(f"{degree} has a concentration in {cls.myConcentration}") # Call on the CLASS itself — no object needed Major.Concentration("Computer Science") # "Computer Science has a concentration in Data Science" # Modify static variable — three equivalent ways class Student: universityName = "NNU" @classmethod def SetNNU1(cls, name): cls.universityName = name # via cls @classmethod def SetNNU2(cls, name): Student.universityName = name # via class name @staticmethod def SetNNU3(name): Student.universityName = name # via class name (no cls available) student1 = Student("Rufus", "Sophomore") student1.SetNNU1("NWU") # via instance Student.SetNNU3("NNU") # via class (static methods only)

Inner Classes & Garbage Collection

▶

Python allows classes to be declared inside other classes — useful for dependent relationships (a University NEEDS Students, not just HAS them).

class University: def __init__(self, myName): self.myName = myName def myDisplay(self): print(f"University: {self.myName}") class Info: # inner class — defined INSIDE University def __init__(self): self.founded = 1913 def MyDisplay1(self): print(f"Founded: {self.founded}") # Access inner class univ = University("NNU") info = University.Info() # create inner class instance info.MyDisplay1()

Inner classes model strong dependency — the inner class conceptually belongs to the outer class and has no meaningful existence without it. Unlike aggregation (HAS A), inner classes represent containment. Don't overuse them — most relationships are better expressed through regular composition.

Garbage Collection & Destructors:

class Demo: def __init__(self): print("Object created — memory allocated") def __del__(self): print("Object destroyed — cleanup performed") obj = Demo() # __init__ called del obj # __del__ called — memory freed # Python's garbage collector runs automatically in the background # recovering memory from objects no longer referenced # GC can be disabled for performance but rarely should be

Inheritance

▶

A child class inherits ALL functionality from a parent and can extend or modify it. This is the "is a" relationship — a Graduate IS A Student.

class Student: def MyMajor(self): print("My major is Computer Science") @classmethod def MyFavoriteFood(cls): print("I love pizza") # Graduate inherits everything from Student class Graduate(Student): def Occupation(self): print("I am a Python developer.") grad = Graduate() grad.MyMajor() # inherited from Student grad.MyFavoriteFood() # inherited from Student grad.Occupation() # defined in Graduate # super() — call the parent's version of a method class Student: def myDisplay(self): print(f"Name: {self.myName}") class Graduate(Student): def myDisplay(self): super().myDisplay() # call parent's myDisplay first print(f"Major: {self.myMajor}") # then add Graduate-specific info # Access other classes — classes are just variables class UpdateMajor: def myUpdate(student, major): # no self — not an instance method student.major = major student.MyDisplay()

super() is essential for extending (not replacing) parent behavior. Without it, overriding a method completely replaces the parent's version. With super().method(), you get the parent's behavior PLUS your additions. Always call super().__init__() in child constructors to properly initialize the parent's instance variables.

Polymorphism & Duck Typing

▶

Polymorphism means changing behavior depending on the situation. Python uses Duck Typing — "if it walks like a duck and quacks like a duck, it's a duck." It assumes behaviors will work at runtime without requiring a shared base class.

class Duck: def Talk(self): print("Quack!") class Bird: def Talk(self): print("Tweet!") class Dog: def Talk(self): print("Woof!") # Works on any object that has a Talk method — no inheritance needed def MakeNoise(animal): animal.Talk() MakeNoise(Duck()) # Quack! MakeNoise(Bird()) # Tweet! MakeNoise(Dog()) # Woof! # Safe duck typing — check before calling def myfunc(obj): if hasattr(obj, "Talk"): obj.Talk() elif hasattr(obj, "Eat"): obj.Eat() # no AttributeError if the method doesn't exist

Duck typing is Python's approach to polymorphism — instead of requiring objects to inherit from a common base class, Python just tries to call the method and sees if it works. Use hasattr() when you're not sure an object has a particular method. This is more flexible than Java/C++ but puts more responsibility on the programmer to ensure objects have the right interface.

Operator Overloading & Method Overriding

▶
# Operator overloading — redefine what operators do for your class # + is tied to __add__, - to __sub__, * to __mul__, etc. class MyCylinder: def __init__(self, myweight): self.myweight = myweight # override + operator def __add__(self, other): return self.myweight + other.myweight c1 = MyCylinder(10) c2 = MyCylinder(20) print(c1 + c2) # 30 — calls __add__ # Common dunder methods for overloading # __str__ → str(obj) and print(obj) # __len__ → len(obj) # __eq__ → obj1 == obj2 # __lt__ → obj1 < obj2 # __add__ → obj1 + obj2 # Method overriding — redefine parent method in child class Parent: def result(self, num1, num2): print(f"Subtraction: {num1} - {num2} = {num1 - num2}") class Child(Parent): def mydisplay(self): print("Hey I am a child class") # Override result — completely replaces parent version def result(self, num1, num2): print(f"Addition: {num1} + {num2} = {num1 + num2}") # Override AND keep parent behavior using super() def result_extended(self, num1, num2): super().result(num1, num2) # parent's subtraction print(f"Also addition: {num1 + num2}") # plus child's addition

Access Modifiers

▶

Access modifiers control who can access variables and methods. Python uses naming conventions rather than enforced keywords.

ModifierSyntaxAccessible From
PublicmyVar (no underscore)Everywhere — default
Protected_myVar (single underscore)Class + subclasses (convention only — not enforced)
Private__myVar (double underscore)Within the class only (Python name-mangles it)
class Student: myPublicName = None # public — accessible everywhere _myProtectedId = None # protected — flag: use getters/setters __myPrivateSSN = None # private — name-mangled to _Student__myPrivateSSN s = Student() print(s.myPublicName) # OK print(s._myProtectedId) # OK but you shouldn't — use getter print(s.__myPrivateSSN) # AttributeError! print(s._Student__myPrivateSSN) # works but very bad practice

Python doesn't truly enforce access modifiers like C++ or Java do. Single underscore (_var) is a convention meaning "please don't touch this directly." Double underscore (__var) triggers name-mangling — Python renames it to _ClassName__var — making it harder (but not impossible) to access from outside. The real enforcement comes from code review and team discipline, not the language.

Abstract Classes & Interfaces

▶

Abstract classes define methods that MUST be implemented by any subclass. You cannot directly instantiate an abstract class — it's a template, not a usable object.

from abc import ABC, abstractmethod # Abstract class — inherits from ABC class Person(ABC): @abstractmethod def MyMethod1(self): pass # no implementation — must be overridden def MyMethod2(self): print("I am a concrete method") # has implementation # Concrete subclass — MUST implement all abstract methods class Student(Person): def MyMethod1(self): print("I am overriding the abstract method") p = Person() # TypeError — cannot instantiate abstract class s = Student() # OK s.MyMethod1() # "I am overriding the abstract method" s.MyMethod2() # "I am a concrete method" — inherited from Person # Simulating an interface — abstract class with ONLY abstract methods class Myengineer(ABC): @abstractmethod def mybranch(self): pass @abstractmethod def myskills(self): pass # Any class that "implements" Myengineer must define BOTH methods class MyEEE(Myengineer): def mybranch(self): print("Electrical and Electronics Engineer") def myskills(self): print("Circuit design, signal processing")

Python doesn't have a true interface keyword like Java — it simulates interfaces with abstract classes containing only abstract methods. Any class inheriting from it must implement ALL abstract methods or it's also abstract. The abc module (Abstract Base Class) is the standard tool. Note: unlike Java/C++, Python won't force you to implement abstract methods at compile time — you'll only find out at runtime when you try to instantiate.
