Introduction & Evaluation Criteria

▶

Sorting improves efficiency of data handling. Evaluating sorting algorithms requires machine-independent metrics applied across different input states.

Number of Comparisons

How many times are two elements compared? The primary metric for comparison-based sorts.

Number of Data Movements

How many times are elements moved or swapped? An algorithm can be fast on comparisons but slow on moves.

Best / Average / Worst Case

Some algorithms behave very differently depending on whether input is sorted, reverse-sorted, or random. All three cases matter.

Practical Considerations

Simpler O(n²) algorithms may outperform O(n log n) for small datasets due to lower constant factors and simpler code. Theory must be tempered by practice.

Elementary Sorting Algorithms

▶

All elementary sorts are O(n²) — fine for small datasets, impractical for large ones.

Insertion Sort — build sorted list one item at a time, inserting each into its correct position:

template<class T>
void insertionSort(T data[], int n) {
    for (int i = 1, j; i < n; i++) {
        T tmp = data[i];
        for (j = i; j > 0 && tmp < data[j-1]; j--)
            data[j] = data[j-1];  // shift larger elements right
        data[j] = tmp;            // insert in correct position
    }
}
// [5 2 3 8 1] → [2 5 3 8 1] → [2 3 5 8 1] → [2 3 5 8 1] → [1 2 3 5 8]
// Best case: O(n) when already sorted — only checks, no moves
// Worst case: O(n²) when reverse sorted

Selection Sort — find minimum, swap to front, repeat:

template<class T>
void selectionSort(T data[], int n) {
    for (int i = 0, j, least; i < n-1; i++) {
        for (j = i+1, least = i; j < n; j++)
            if (data[j] < data[least]) least = j;
        swap(data[least], data[i]);
    }
}
// Always exactly n(n-1)/2 comparisons — no best/worst distinction
// Very few moves: exactly n-1 swaps regardless of input

Bubble Sort — repeatedly swap adjacent out-of-order pairs:

template<class T>
void bubbleSort(T data[], int n) {
    for (int i = 0; i < n-1; i++)
        for (int j = n-1; j > i; --j)
            if (data[j] < data[j-1])
                swap(data[j], data[j-1]);
}
// Small values "bubble" left, large values "bubble" right
// Disadvantage: elements must be swapped one position at a time
// Items that are already in place still get moved (unlike insertion sort)

Comb Sort — bubble sort improvement (Dobosiewicz 1980, popularized 1991). Uses a gap > 1 to eliminate "turtles" (small values near the end that slow bubble sort). Gap starts at n/1.3 and shrinks by dividing by 1.3 each pass until gap = 1 (standard bubble sort finish). Rivals quicksort in practice.

The key insight of comb sort: bubble sort is slow because small values near the end (turtles) must move one position per pass. By comparing elements far apart first, turtles leap toward the front in a single pass. Large values near the start (rabbits) aren't a problem for bubble sort — only turtles are.

Decision Trees — Lower Bound for Sorting

▶

Any comparison-based sort can be modeled as a decision tree — a binary tree where each internal node is a comparison and each leaf is a possible sorted ordering.

  • For n items, there are n! possible orderings → at least n! leaves needed
  • A tree with m leaves has height ≥ ⌈lg m⌉
  • Therefore minimum height ≥ ⌈lg(n!)⌉
  • lg(n!) = Θ(n lg n) by Stirling's approximation

Fundamental lower bound: Any comparison-based sorting algorithm requires Ω(n log n) comparisons in the worst and average cases. This is provably optimal — no comparison sort can do better. This is why we aim for O(n log n) algorithms. Elementary O(n²) sorts are provably suboptimal. Non-comparison sorts (radix, counting) can beat this bound by not using comparisons.

Shell Sort — Diminishing Increment Sort

▶

Shell sort (Donald Shell, 1959) generalizes insertion sort by comparing elements far apart first, then progressively reducing the gap. Far-apart comparisons move elements toward their final position quickly.

// Shell sort with gap sequence: n/2, n/4, ..., 1
void shellSort(T data[], int n) {
    for (int h = n/2; h > 0; h /= 2) {       // gap decreases
        for (int i = h; i < n; i++) {
            T tmp = data[i];
            int j = i;
            while (j >= h && tmp < data[j-h]) {
                data[j] = data[j-h];            // insertion sort with gap h
                j -= h;
            }
            data[j] = tmp;
        }
    }
}
// Example with [10 8 6 20 4 3 22 1 0 15 16], gaps: 5, 3, 1
// Gap 5: compare positions 0&5, 1&6, 2&7, ... — sorts 5 subarrays
// Gap 3: sorts 3 subarrays of the partially-ordered result
// Gap 1: final insertion sort — now nearly sorted, very fast

Shell sort's complexity depends on the gap sequence chosen. The simple n/2 sequence gives O(n²) worst case. Better sequences (Hibbard: 1,3,7,15,...; Sedgewick: 1,5,19,41,...) give O(n^(3/2)) or better. Shell sort is not stable — equal elements may be reordered. In practice, it's often faster than O(n log n) sorts for medium-sized datasets due to simplicity.

Heap Sort

▶

Heap sort (John Williams, 1964) improves selection sort's comparison phase using a heap. Always O(n log n) — better worst-case than quicksort. In-place but not stable.

template<class T>
void heapSort(T data[], int n) {
    // Phase 1: Build max-heap (Floyd's bottom-up algorithm)
    for (int i = n/2 - 1; i >= 0; --i)
        moveDown(data, i, n-1);  // heapify each subtree

    // Phase 2: Extract max repeatedly
    for (int i = n-1; i >= 1; --i) {
        swap(data[0], data[i]);  // move largest to end
        moveDown(data, 0, i-1); // restore heap on remaining elements
    }
}
// Phase 1: O(n) — Floyd's algorithm is linear despite appearances
// Phase 2: n-1 extractions × O(log n) each = O(n log n)
// Total: O(n log n) guaranteed — no worst case degradation

Selection sort is O(n²) because finding the minimum requires scanning all remaining elements each time. Heap sort achieves the same "find and remove the max" pattern but uses the heap property to find the max in O(log n) instead of O(n). The heap is maintained as an array — no extra memory needed. Heap sort is never faster than O(n log n) but also never slower.

Quicksort

▶

Quicksort (C.A.R. Hoare, 1960) — partition around a pivot, then recursively sort each partition. Fastest in practice for most inputs.

// Quicksort — choose pivot, partition, recurse
quicksort(data[], first, last):
    if first < last:
        pivot = data[(first + last) / 2]  // middle element as pivot
        partition data into:
            left:  elements ≤ pivot
            right: elements ≥ pivot
        quicksort(left partition)
        quicksort(right partition)

// Partitioning (Hoare scheme):
// i starts at first, j starts at last
// Move i right while data[i] < pivot
// Move j left while data[j] > pivot
// Swap data[i] and data[j]
// Repeat until i >= j

Pivot Choice Matters

Middle element: good average case. First/last: O(n²) on sorted input. Median-of-three: robust choice. Random: good average, hard to exploit.

Best/Average Case

O(n log n) when pivot splits array roughly in half each time. Average case O(n log n) for random input.

Worst Case

O(n²) when pivot is always smallest or largest element — degenerate partitioning. Happens with sorted/reverse-sorted input and poor pivot choice.

In Practice

Fastest sorting algorithm for most real-world inputs despite O(n²) worst case. Low constant factor, good cache performance, in-place. Not stable.

Quicksort is NOT stable — equal elements may be reordered. Elements equal to the pivot can go to either partition — this affects balance. The algorithm preprocesses to find the largest element and moves it to the end, simplifying the partition step. Once partitioning is complete, the pivot is already in its final position — the key invariant that makes quicksort correct.

Mergesort

▶

Mergesort (John von Neumann, 1945) — split in half, recursively sort each half, merge. Guaranteed O(n log n), stable, but requires O(n) extra memory.

// Mergesort
void mergeSort(T data[], int first, int last) {
    if (first < last) {
        int mid = (first + last) / 2;
        mergeSort(data, first, mid);     // sort left half
        mergeSort(data, mid+1, last);    // sort right half
        merge(data, first, last);        // merge sorted halves
    }
}

// Merge — requires temp array
void merge(T data[], int first, int last) {
    int mid = (first + last) / 2;
    int i1 = 0, i2 = first, i3 = mid + 1;
    T temp[last - first + 1];

    while (i2 <= mid && i3 <= last)
        temp[i1++] = (data[i2] < data[i3]) ? data[i2++] : data[i3++];

    while (i2 <= mid)  temp[i1++] = data[i2++]; // remaining left
    while (i3 <= last) temp[i1++] = data[i3++]; // remaining right

    for (int i = 0; i < i1; i++) data[first + i] = temp[i];
}
// Each merge: exactly 2×(last-first+1) moves, ≤(last-first+1) comparisons
// Total: O(n log n) comparisons and moves — always, no worst case

Mergesort's key advantage over quicksort: guaranteed O(n log n) with no worst-case degradation. It's also stable — equal elements maintain their original order. The key disadvantage: requires O(n) extra memory for the temp array. For external sorting (data on disk), mergesort is almost always the right choice because it accesses data sequentially.

Radix Sort

▶

Radix sort (Herman Hollerith, 1887) — sort by individual digits from least significant to most significant. No comparisons between elements — beats the O(n log n) lower bound by not being comparison-based.

void radixSort(long data[], int n) {
    const int radix = 10;
    const int digits = 10;  // max digits in a long
    Queue<long> queues[radix];

    for (int d = 0, factor = 1; d < digits; factor *= radix, d++) {
        // Distribute into 10 buckets based on digit d
        for (int j = 0; j < n; j++)
            queues[(data[j] / factor) % radix].enqueue(data[j]);

        // Collect from buckets (maintains relative order!)
        for (int j = 0, k = 0; j < radix; j++)
            while (!queues[j].empty())
                data[k++] = queues[j].dequeue();
    }
}
// Example: sort [93, 63, 64, 94] by LSD (least significant digit first)
// Pass 1 (ones): bucket 3=[93,63], bucket 4=[64,94] → [93,63,64,94]
//   wait — ordered within bucket by arrival → [93,63,94,64] then combine
// Pass 2 (tens): bucket 6=[63,64], bucket 9=[93,94] → [63,64,93,94] ✓
// MUST use queues (FIFO) to maintain order within each bucket!

Using queues (not stacks) is critical — queues preserve the relative order established by previous passes. This stability property is what makes LSD radix sort work. Complexity: O(d × n) where d = number of digits — linear in n for fixed-size keys. Limitation: only works for non-negative integers (or data mappable to non-negative integers). Cannot handle negative numbers directly.

Counting Sort

▶

Counting sort — for small integer keys. Count occurrences, compute positions, place elements. O(n + k) where k is the range of key values. Fastest sort when k is small. Often used as a subroutine in radix sort.

void countingSort(int data[], int n, int maxVal) {
    int count[maxVal + 1] = {0};
    int tmp[n];

    // Step 1: Count occurrences
    for (int i = 0; i < n; i++) count[data[i]]++;

    // Step 2: Accumulate counts (count[i] = # elements < i)
    for (int i = 1; i <= maxVal; i++) count[i] += count[i-1];

    // Step 3: Place elements in correct position (reverse for stability)
    for (int i = n-1; i >= 0; i--)
        tmp[--count[data[i]]] = data[i];

    // Step 4: Copy back
    for (int i = 0; i < n; i++) data[i] = tmp[i];
}
// Example: data = [7 1 1 3 0 7 5 5 7 3]
// After count: count[0]=1, count[1]=2, count[3]=2, count[5]=2, count[7]=3
// After accumulate: count gives starting positions for each value
// Place elements right-to-left for stability

Counting sort is the fastest sort in benchmark tests — but only when key values don't significantly exceed the number of items. If sorting 1,000 numbers with values up to 1,000,000, the O(n+k) = O(1,001,000) cost is worse than O(n log n) ≈ O(10,000). Cannot handle negative numbers without offset adjustment. Cannot handle non-integer keys directly.

Sorting in the STL

▶ 
#include <algorithm>

vector<int> v = {5, 2, 8, 1, 9, 3};

// Full sort — introsort (quicksort + heap sort fallback) O(n log n)
sort(v.begin(), v.end());                          // ascending
sort(v.begin(), v.end(), greater<int>());          // descending

// Stable sort — mergesort O(n log n) guaranteed, preserves equal element order
stable_sort(v.begin(), v.end());

// Partial sort — find k smallest and sort them O(n log k)
partial_sort(v.begin(), v.begin()+3, v.end());     // 3 smallest in front
partial_sort(v.begin()+1, v.begin()+4, v.end(), greater<int>()); // 3 largest

// Partial sort copy — copy k sorted elements to another container
vector<int> v2(4);
partial_sort_copy(v.begin(), v.begin()+4, v2.begin(), v2.end());

// Partition — split by predicate (not sorted)
partition(v.begin(), v.end(), [](int x){ return x % 2 == 0; });
// Even elements before odd elements — relative order not preserved
stable_partition(v.begin(), v.end(), [](int x){ return x % 2 == 0; });
// Same but preserves relative order within each group
