What Is a Subquery?

▶

A subquery is a query nested inside another query. The results of the inner query are passed to the outer query — the inner query acts like an expression from the outer query's perspective.

Self-Contained Subquery

Has no dependency on the outer query. Can be executed independently on its own. Easier to write, test, and debug.

Correlated Subquery

Refers to elements (columns) from tables used in the outer query. Cannot be executed independently. Executes once per row of the outer query.

Scalar Subquery

Returns exactly one value (one row, one column). Can be used anywhere a single-valued expression is expected: SELECT list, WHERE clause, HAVING clause.

Multi-Valued Subquery

Returns multiple values as a single column set. Used with the IN predicate. Can often be rewritten as a JOIN — test both for performance.

Subqueries allow complex queries to be expressed in a single statement. Without them, you'd need temp tables, multiple queries, or complex self-joins. The inner query always executes first — its result becomes input for the outer query.

Scalar Subqueries

▶

A scalar subquery returns a single value (one row, one column). If the inner query returns an empty set, the result is converted to NULL — not an error.

-- Find all order details for the most recent order
SELECT orderid, productid, unitprice, qty
FROM Sales.OrderDetails
WHERE orderid = (
    SELECT MAX(orderid) AS lastorder
    FROM Sales.Orders
);

-- Scalar subquery in SELECT clause: show difference from average
SELECT orderid, custid, freight,
    freight - (SELECT AVG(freight) FROM Sales.Orders) AS diff_from_avg
FROM Sales.Orders;

-- Scalar subquery in HAVING clause
SELECT custid, COUNT(*) AS order_count
FROM Sales.Orders
GROUP BY custid
HAVING COUNT(*) > (
    SELECT AVG(cnt)
    FROM (SELECT custid, COUNT(*) AS cnt
          FROM Sales.Orders
          GROUP BY custid) AS counts
);

The outer query's construction determines whether the inner query must return a single value. If you use = in the WHERE clause, the subquery MUST return exactly one value — if it returns multiple rows, SQL Server throws an error. Use MAX(), MIN(), or other aggregates to guarantee a scalar result.

Multi-Valued Subqueries

▶

A multi-valued subquery returns multiple values as a single column. Used with the IN predicate — if any value in the subquery result matches the IN expression, the predicate returns TRUE.

-- Customers in Mexico using subquery
SELECT custid, orderid
FROM Sales.Orders
WHERE custid IN (
    SELECT custid
    FROM Sales.Customers
    WHERE country = N'Mexico'
);

-- Equivalent JOIN (often similar or better performance)
SELECT o.custid, o.orderid
FROM Sales.Orders AS o
JOIN Sales.Customers AS c ON o.custid = c.custid
WHERE c.country = N'Mexico';

-- NOT IN: orders for customers NOT in Mexico
SELECT custid, orderid
FROM Sales.Orders
WHERE custid NOT IN (
    SELECT custid
    FROM Sales.Customers
    WHERE country = N'Mexico'
);

WARNING with NOT IN and NULLs: if the subquery result set contains any NULL values, NOT IN returns no rows at all — because NULL compared to anything yields UNKNOWN, not FALSE or TRUE. Use NOT EXISTS instead of NOT IN when the subquery column might contain NULLs.

Correlated Subqueries

▶

Correlated subqueries refer to columns from the outer query. They cannot be executed independently — they depend on a value from each outer row. The inner query executes once per row of the outer query.

-- Find each employee's most recent order
-- Inner query references O1.empid from the outer query
SELECT orderid, empid, orderdate
FROM Sales.Orders AS O1
WHERE orderdate = (
    SELECT MAX(orderdate)
    FROM Sales.Orders AS O2
    WHERE O2.empid = O1.empid   -- correlation: O1 is outer
)
ORDER BY empid, orderdate;

-- Find each customer's most recent order date
SELECT custid, orderid, orderdate
FROM Sales.Orders AS outerorders
WHERE orderdate = (
    SELECT MAX(orderdate)
    FROM Sales.Orders AS innerorders
    WHERE innerorders.custid = outerorders.custid  -- correlation
)
ORDER BY custid;

Correlated subqueries are harder to test than self-contained ones — you can't just highlight the inner query and run it independently. To test, replace the outer reference with a literal value (e.g., substitute O1.empid with a specific empid) to verify the inner query logic works.

Performance note: because correlated subqueries execute once per outer row, they can be slow on large tables. Consider rewriting as a JOIN or using window functions (like ROW_NUMBER() OVER PARTITION BY) for better performance on large datasets.

The EXISTS Predicate

▶

EXISTS tests whether a subquery returns any rows at all. It returns TRUE or FALSE — never UNKNOWN. No rows are passed back to the outer query; EXISTS stops as soon as it finds the first matching row.

-- Customers who HAVE placed orders
SELECT custid, companyname
FROM Sales.Customers AS c
WHERE EXISTS (
    SELECT *               -- * is conventional — column list doesn't matter
    FROM Sales.Orders AS o
    WHERE c.custid = o.custid  -- correlated to outer query
);

-- Customers who have NOT placed any orders
SELECT custid, companyname
FROM Sales.Customers AS c
WHERE NOT EXISTS (
    SELECT *
    FROM Sales.Orders AS o
    WHERE c.custid = o.custid
);

EXISTS is almost always better than IN for existence tests — especially when the subquery might return NULLs. EXISTS stops at the first match (short-circuit evaluation) making it efficient. NOT EXISTS handles NULLs correctly unlike NOT IN. The SELECT list inside EXISTS doesn't matter — use SELECT * or SELECT 1 by convention.

Self-Contained vs. Correlated: Side by Side

▶ 
-- SELF-CONTAINED: inner query has no reference to outer query
-- Can be run independently
SELECT orderid, productid, unitprice, qty
FROM Sales.OrderDetails
WHERE orderid = (
    SELECT MAX(orderid)        -- no reference to outer tables
    FROM Sales.Orders
);

-- CORRELATED: inner query references outer query's table (O1)
-- Cannot be run independently — depends on each outer row
SELECT orderid, empid, orderdate
FROM Sales.Orders AS O1
WHERE orderdate = (
    SELECT MAX(orderdate)
    FROM Sales.Orders AS O2
    WHERE O2.empid = O1.empid  -- O1 comes from outer query
);

Self-Contained Advantages

Can be tested independently. Easier to debug. SQL Server often optimizes them more aggressively. Execute once — result reused for all outer rows.

Correlated Advantages

Can express per-row logic that self-contained queries cannot. Essential for "most recent per group" type queries. Can be combined with EXISTS for powerful existence tests.

When writing a correlated subquery, always use table aliases to clearly distinguish between inner and outer query references. The pattern of matching O2.empid = O1.empid (inner = outer) is the signature of a correlated subquery — the inner query is "correlated" to each row of the outer query.
