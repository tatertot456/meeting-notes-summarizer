Integer Addition & Subtraction

▶

Integer addition works exactly like binary long addition, but hardware has finite width — results can overflow when they exceed the representable range.

Overflow detection rules for addition:

  • Adding a positive and a negative number: never overflows
  • Adding two positive numbers: overflow if result sign bit is 1
  • Adding two negative numbers: overflow if result sign bit is 0

Subtraction is implemented as adding the negation: 7 – 6 = 7 + (–6)

+7:  0000 0000 … 0000 0111
–6:  1111 1111 … 1111 1010
+1:  0000 0000 … 0000 0001

Multimedia/SIMD uses saturating arithmetic — on overflow, the result clamps to the largest representable value rather than wrapping around. This is correct for audio clipping and video saturation where a hard limit makes more sense than modular wraparound.

Multiplication

▶

Binary multiplication mirrors long multiplication: for each bit of the multiplier, either add a shifted copy of the multiplicand (if bit = 1) or add zero (if bit = 0). The product of two n-bit numbers is up to 2n bits wide.

Basic Hardware

Shift the multiplicand left, check each multiplier bit, add to a running product. Takes one cycle per bit — slow but simple.

Optimized Multiplier

Perform add and shift steps in parallel. One cycle per partial-product addition. Acceptable when multiplication frequency is low.

Faster Multiplier

Uses multiple adders in a tree structure. Can be pipelined — several multiplications in flight simultaneously. High cost/performance ratio.

LEGv8 Instructions

MUL: lower 64 bits of product. SMULH: upper 64 bits (signed). UMULH: upper 64 bits (unsigned). Two instructions needed for full 128-bit result.

Why two instructions for multiply? Two 64-bit numbers can produce a 128-bit result. MUL gives the lower half (sufficient for most code). SMULH/UMULH give the upper half when needed for overflow detection or arbitrary precision.

Division

▶

Division is the inverse of multiplication: given dividend and divisor, produce a quotient and remainder. Always check for division by zero first.

Long division algorithm:

  • If divisor ≤ current dividend bits: quotient bit = 1, subtract divisor
  • Otherwise: quotient bit = 0, bring down next dividend bit
  • Restoring division: if remainder goes negative after subtraction, add the divisor back

Signed Division

Divide using absolute values, then adjust sign of quotient and remainder. Remainder has same sign as dividend.

Hardware

Division hardware looks similar to multiplication hardware. Same datapath can serve both — major cost saving.

Faster Division

Unlike multiplication, division can't easily be parallelized — each subtraction is conditional on the sign of the remainder. SRT division generates multiple quotient bits per step.

LEGv8 Instructions

SDIV (signed), UDIV (unsigned). Both ignore overflow and division-by-zero — software must check.

Floating-Point Representation

▶

Floating-point represents real numbers in scientific notation: ±1.fraction × 2^exponent

This allows representation of both very small numbers (like 10⁻³⁸) and very large numbers (like 10³⁸) — impossible with fixed-point integers of the same width.

Normalized Form

Always ±1.fraction × 2^exp. The leading 1 before the binary point is implicit (hidden bit) — saves one bit of precision.

Denormal Numbers

Exponent = 000...0 means hidden bit is 0. Allows gradual underflow — numbers smaller than normal with diminishing precision rather than an abrupt drop to zero.

Infinity

Exponent = 111...1, Fraction = 0. Represents ±∞. Can propagate through calculations without triggering an immediate overflow check.

NaN (Not a Number)

Exponent = 111...1, Fraction ≠ 0. Results from illegal operations like 0.0/0.0 or √(–1). Propagates through calculations.

Bits have no inherent meaning — the same bit pattern means completely different things depending on whether it's interpreted as integer, unsigned, or floating-point. The instruction determines the interpretation.

IEEE 754 Standard

▶

IEEE 754 (1985) is the universal floating-point standard, developed to fix the chaos of incompatible representations across different systems. Almost universally adopted today.

Format: S | Exponent | Fraction

Value = (–1)^S × (1 + Fraction) × 2^(Exponent – Bias)

FormatTotal bitsExponentFractionBiasRangePrecision
Single328 bits23 bits127±1.2×10⁻³⁸ to ±3.4×10³⁸~6 decimal digits
Double6411 bits52 bits1023±2.2×10⁻³⁰⁸ to ±1.8×10³⁰⁸~16 decimal digits

Biased exponent: the stored exponent = actual exponent + Bias. This ensures the exponent field is always non-negative, making FP comparison possible using integer comparison hardware. Exponents 000...0 and 111...1 are reserved for denormals/zero and infinity/NaN.

Example: –0.75 in single precision

–0.75 = (–1)¹ × 1.1₂ × 2⁻¹
S = 1
Exponent = –1 + 127 = 126 = 01111110₂
Fraction = 10000...00₂
Result: 1 01111110 10000000000000000000000

Floating-Point Arithmetic

▶

FP Addition — 4 steps:

  • 1. Align: Shift the number with the smaller exponent right until exponents match
  • 2. Add: Add the significands
  • 3. Normalize: Shift result to restore 1.fraction form, adjust exponent, check overflow/underflow
  • 4. Round: Round to fit available fraction bits, renormalize if needed

FP Multiplication — 5 steps:

  • 1. Add exponents: New exponent = exp₁ + exp₂ – Bias (subtract bias since both are biased)
  • 2. Multiply significands
  • 3. Normalize
  • 4. Round
  • 5. Determine sign: XOR of the two sign bits

FP arithmetic hardware is much more complex than integer arithmetic and takes several cycles. Running it in one cycle would require such a long critical path that the entire processor clock would have to slow down — penalizing all instructions. FP units are pipelined to hide this latency.

LEGv8 FP registers and instructions:

FP Registers

32 single-precision (S0–S31) and 32 double-precision (D0–D31). Sn stored in lower 32 bits of Dn. Separate from integer registers.

FP Load/Store

LDURS/LDURD, STURS/STURD. Must use FP-specific load/store — integer LDUR won't work with FP registers.

Single Precision

FADDS, FSUBS, FMULS, FDIVS. Example: FADDS S2, S4, S6

Double Precision

FADDD, FSUBD, FMULD, FDIVD. FCMPS/FCMPD for comparison. B.cond for conditional branching on FP results.

SIMD & Subword Parallelism

▶

SIMD (Single Instruction, Multiple Data) performs the same operation on multiple data elements simultaneously — exploiting data-level parallelism in graphics, audio, and scientific workloads.

A 128-bit register can hold:

  • Sixteen 8-bit integers
  • Eight 16-bit integers
  • Four 32-bit floats
  • Two 64-bit doubles

ARMv8 SIMD: 32 × 128-bit V registers (V0–V31)

ADD  V1.16B, V2.16B, V3.16B  // 16 simultaneous 8-bit adds
FADD V1.4S,  V2.4S,  V3.4S   // 4 simultaneous 32-bit FP adds

Matrix multiply performance comparison: Unoptimized scalar code processes one element at a time. AVX-optimized SIMD code (using 256-bit YMM registers) processes 4 doubles simultaneously — achieving roughly 4× throughput on the inner loop. The key instruction is vmulpd (parallel multiply) instead of vmulsd (scalar multiply).

ARMv8 includes 245 SIMD instructions covering: square root, fused multiply-add, saturating arithmetic, and strided vector loads/stores.

Pitfalls & Concluding Remarks

▶

Right Shift ≠ Division for Signed

Arithmetic right shift replicates the sign bit. –5 >> 2 = –2 (rounds toward –∞), not –1. Use with care on signed integers.

FP Associativity Fails

(x+y)+z ≠ x+(y+z) in floating-point due to rounding. Parallel programs that reorder operations can produce different results — validate carefully.

Finite Precision

FP numbers have bounded range and precision. Operations can overflow (too large), underflow (too small), or lose precision through rounding.

The Pentium FDIV Bug

A lookup table error in Pentium's FP divider produced wrong results for certain inputs. Cost Intel $475M in recalls. Accuracy matters even for "everyday" use.

Core takeaway: bits have no inherent meaning. The same 32-bit pattern is a different number depending on whether you apply integer, unsigned, or floating-point instructions. Programs must track types carefully — hardware just manipulates bits.

ISA arithmetic summary: Signed and unsigned integers (exact within range), floating-point (approximate reals with bounded range and precision). Both can overflow — programs must account for finite representation in all numeric computations.
