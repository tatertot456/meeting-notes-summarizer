What Is a Table?

▶

Relational databases store data in tables (relations). A table is defined by a collection of named columns and contains zero or more rows. Tables typically represent a type of object or entity — employees, purchase orders, customers, sales orders.

  • Each row usually represents a single instance of the entity (one employee, one order)
  • Rows have no guaranteed order — use ORDER BY when order matters
  • Tables are a security boundary — permissions can be set at the table level
  • Consistent naming conventions for tables are important for maintainability

Tables in the relational model are based on set theory — they act on all elements at once. Members must be unique (enforced by primary keys). No defined order exists in the underlying set — ORDER BY is required for presentation.

Normalization

▶

Normalization is a process invented by Edgar F. Codd that ensures database structures are appropriate and avoid poor design characteristics. Degrees of normalization are called normal forms.

First Normal Form (1NF)

Eliminate repeating groups in individual tables. Create a separate table for each set of related data. Identify each set with a primary key. Each cell must contain a single atomic value.

Second Normal Form (2NF)

Must be in 1NF. Non-key columns must depend on the ENTIRE primary key — not just part of it. Columns that depend on only part of a composite key belong in a separate table linked by a foreign key.

Third Normal Form (3NF)

Must be in 2NF. Eliminate fields that do not depend on the key — only on the key, the whole key, and nothing but the key. Remove transitive dependencies.

Denormalization

Database designs should initially be normalized. Denormalization might be applied later to improve performance or simplify data warehouse analysis — but only deliberately.

Primary Keys & Foreign Keys

▶

Primary keys uniquely identify each row within a table:

  • Must be unique — no two rows share the same primary key value
  • Cannot be NULL (unknown values can't uniquely identify a row)
  • Can involve multiple columns (composite key)
  • Should not change — updating a primary key cascades complexity
  • Natural key: formed from data related to the entity (e.g. Social Security Number)
  • Surrogate key: system-generated codes or numbers (e.g. IDENTITY column)

Surrogate keys are generally preferred: they never change, are always unique, have no business meaning that could become outdated, and simplify joins. Use IDENTITY or SEQUENCE to generate them automatically.

Foreign keys create references between tables:

  • A foreign key in one table holds the primary key from another table
  • Self-references are permitted (e.g. employee manager is also an employee)
  • Rows that don't exist in the referenced table cannot be inserted
  • Referenced rows cannot be deleted without cascading options
  • Multiple foreign keys can exist in one table

Surrogate key implementation:

-- IDENTITY: auto-incrementing column
CREATE TABLE Employees (
    EmpID INT IDENTITY(1,1) PRIMARY KEY,
    LastName VARCHAR(50) NOT NULL
);

-- @@IDENTITY: last value across ALL sessions (avoid)
-- SCOPE_IDENTITY(): last value in CURRENT session (preferred)

-- SEQUENCE: shared across multiple tables
CREATE SEQUENCE dbo.OrderSeq
    START WITH 1 INCREMENT BY 1
    MINVALUE 1 MAXVALUE 999999;

Designing for Concurrency

▶

SQL Server uses locking to manage concurrent access. Locks can be applied at the row, page, or table level.

Row Locking

Increases concurrency — multiple users can access different rows simultaneously. Higher overhead from maintaining many individual locks.

Table Locking

Lower overhead but blocks all users when one user is writing. Acceptable for batch operations on data warehouses.

OLTP Databases

Should be normalized. Transactions should be small and fast. Smaller tables (fewer columns) are less likely to cause locking conflicts.

Data Warehouses

Should be denormalized. No frequent modifications so locking is less of a concern. Optimized for read-heavy analytical queries.

Data Types

▶

Data types determine what can be stored in a column, variable, expression, or parameter. Choosing appropriate data types assists with query optimization and provides self-documentation.

For NULL vs NOT NULL: NULL indicates the absence of a value — not zero, not empty string. Declare NOT NULL when a value is always required. NULL-able columns should be declared nullable only when "unknown" is a valid state for the data.

Unicode data uses double-byte storage (nchar, nvarchar). Always prefix Unicode string literals with N: WHERE country = N'Spain'. Use LEN() for character count, DATALENGTH() for byte count.

Alias data types are created with CREATE TYPE and behave like system types — useful for standardizing column types across a database.

Schemas

▶

Schemas are containers for database objects — tables, stored procedures, functions, types, views. They also serve as security boundaries.

  • Permissions can be granted at the schema level, applying to all objects within it
  • Simplifies security configuration — grant once, covers everything in the schema
  • Objects are referenced as SchemaName.ObjectName: Production.Product
  • Always specify schema name to avoid ambiguity and improve performance

Name resolution without schema prefix:

  • SQL Server first searches the user's default schema
  • If not found, searches the dbo schema
  • Users with no default schema assigned use dbo as default
-- Create a schema
CREATE SCHEMA Production AUTHORIZATION dbo;

-- Reference objects with schema prefix (always do this)
SELECT ProductID FROM Production.Product;

-- Drop a schema (must be empty first)
DROP SCHEMA Production;

Always reference objects with both schema and object name. Omitting the schema forces SQL Server to search multiple schemas, which adds overhead and can cause unexpected behavior when two schemas contain objects with the same name.

Creating, Altering & Dropping Tables

▶ 
-- CREATE TABLE
CREATE TABLE Production.Products (
    ProductID   INT IDENTITY(1,1) NOT NULL,
    ProductName NVARCHAR(100)     NOT NULL,
    Price       DECIMAL(10,2)     NOT NULL,
    CategoryID  INT               NULL,
    CONSTRAINT PK_Products PRIMARY KEY (ProductID),
    CONSTRAINT FK_Products_Categories
        FOREIGN KEY (CategoryID)
        REFERENCES Production.Categories(CategoryID)
);

-- ALTER TABLE: add a column
ALTER TABLE Production.Products
ADD Description NVARCHAR(500) NULL;

-- ALTER TABLE: drop a column
ALTER TABLE Production.Products
DROP COLUMN Description;

-- DROP TABLE (can't drop if referenced by foreign key)
DROP TABLE Production.Products;

ALTER TABLE retains existing data and permissions. DROP TABLE permanently removes the table AND all its data, constraints, indexes, and triggers — but NOT stored procedures or views that reference it (those become invalid). Reference tables with foreign keys pointing to them cannot be dropped until the foreign key is removed first.

Temporary Tables

▶

Temporary tables exist only for the duration of a session or scope, stored in the tempdb system database.

Local Temporary Tables (#)

Visible only to the creator in the same session and scope. Automatically dropped when out of scope or when the session ends. Prefix with single #.

Global Temporary Tables (##)

Visible to ALL users across all sessions. Deleted when all users referencing the table disconnect. Prefix with double ##.

-- Local temp table
CREATE TABLE #TempOrders (
    OrderID INT,
    TotalAmt DECIMAL(10,2)
);

-- Often created with SELECT INTO
SELECT OrderID, TotalAmt
INTO #TempOrders
FROM Sales.Orders
WHERE OrderDate > '2024-01-01';

-- Global temp table (visible to all)
CREATE TABLE ##SharedData (
    ID INT,
    Value VARCHAR(100)
);

-- Always drop temp tables explicitly in code
DROP TABLE IF EXISTS #TempOrders;

Don't rely on automatic cleanup of temporary tables — drop them explicitly in code when done. This prevents tempdb bloat in long-running sessions and makes the code's intent clear.

Computed Columns

▶

Computed columns are derived from other columns or functions. They provide easier access to data without actually denormalizing the table structure.

-- Computed column: calculated on the fly
CREATE TABLE Sales.OrderDetails (
    OrderID     INT NOT NULL,
    Quantity    INT NOT NULL,
    UnitPrice   DECIMAL(10,2) NOT NULL,
    LineTotal   AS (Quantity * UnitPrice)  -- computed
);

-- PERSISTED: value stored physically, improves SELECT performance
CREATE TABLE Production.Products (
    ProductID   INT NOT NULL,
    Price       DECIMAL(10,2) NOT NULL,
    TaxRate     DECIMAL(5,2) NOT NULL,
    PriceWithTax AS (Price * (1 + TaxRate)) PERSISTED
);

Non-persisted computed columns are recalculated every time they're accessed. PERSISTED computed columns store the value physically — improving SELECT performance at the cost of slightly more storage and compute during INSERT/UPDATE. Use PERSISTED when the computed column is frequently queried.
