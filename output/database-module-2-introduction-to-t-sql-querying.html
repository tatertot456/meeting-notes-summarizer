About T-SQL

▶

Structured Query Language (SQL) was developed by IBM in the 1970s and adopted by ANSI and ISO standards bodies. Transact-SQL (T-SQL) is Microsoft SQL Server's dialect — the querying language of SQL Server.

SQL is declarative — you describe WHAT you want, not the individual steps to get it. You tell the engine "give me customers in Spain" and it figures out HOW to retrieve them. This is fundamentally different from procedural languages where you write step-by-step instructions.

Other dialects: PL/SQL (Oracle), SQL Procedural Language (IBM). The ANSI standard provides a common foundation, but each vendor adds proprietary extensions.

Categories of T-SQL Statements

▶

DML — Data Manipulation Language

Used to query and manipulate data. SELECT, INSERT, UPDATE, DELETE. SELECT is the primary focus of this course.

DDL — Data Definition Language

Used to define and modify database objects and their structure. CREATE, ALTER, DROP.

DCL — Data Control Language

Used to manage security permissions on database objects. GRANT, REVOKE, DENY.

TCL — Transaction Control Language

Used to manage transactions. BEGIN TRANSACTION, COMMIT, ROLLBACK, SAVE TRANSACTION.

Variables: Local variables store values temporarily. Names begin with @. @@ is reserved for system functions. Must be declared and used within the same batch.

-- Declare and initialize in one statement (SQL Server 2016+)
DECLARE @search varchar(30) = 'Match%';
SELECT custid, companyname FROM Sales.Customers
WHERE companyname LIKE @search;

Expressions: combinations of identifiers, values, and operators that evaluate to a single result. Used in SELECT, WHERE, and other clauses.

Batch Separators: Batches are sets of commands sent to SQL Server as a unit. The GO keyword separates batches in SQL Server tools. GO is NOT a T-SQL command — it's a tool directive. Variable scope and name resolution are bounded by batches. GO [count] executes the preceding batch that many times.

Set Theory & SQL Server

▶

The relational database model is mathematically grounded in set theory. Understanding sets explains why SQL behaves the way it does.

Set Theory CharacteristicSQL Server Application
Acts on all elements at onceQueries operate on whole tables — set-based processing
Tell the engine what to retrieveDeclarative SQL — describe results, not steps
Avoid processing items individuallyAvoid cursors and loops — use set-based queries
Elements must be uniqueDefine unique keys in tables (PRIMARY KEY, UNIQUE)
No defined order in a setUse ORDER BY when sorted results are required

A SQL result set is NOT truly relational by default — rows are NOT guaranteed to be unique and have NO guaranteed order. DISTINCT enforces uniqueness in results. ORDER BY enforces ordering. Without these, SQL Server may return results in any order it finds efficient.

Predicate Logic

▶

Predicate logic is the second mathematical foundation of the relational model. A predicate is a property or expression that evaluates to TRUE or FALSE (also called a Boolean expression).

Predicates are used throughout SQL:

Filtering Data

WHERE clause predicates determine which rows survive. Only rows where the predicate evaluates to TRUE are returned.

CASE Expressions

Each WHEN clause contains a predicate. First predicate that evaluates to TRUE determines the returned value.

Joining Tables

The ON clause predicate determines which rows from two tables are matched and combined.

Subqueries & Integrity

EXISTS predicate tests whether a subquery returns rows. CHECK constraints use predicates to validate data.

Three-valued logic: SQL predicates can evaluate to TRUE, FALSE, or UNKNOWN (when NULL is involved). WHERE and HAVING clauses filter out both FALSE and UNKNOWN — only TRUE rows pass. This is why NULL comparisons behave unexpectedly: NULL = NULL evaluates to UNKNOWN, not TRUE. Always use IS NULL or IS NOT NULL to test for NULL values.

Logical Order of Operations in SELECT

▶

The order you write a SELECT query is NOT the order SQL Server evaluates it. Understanding logical processing order explains many "why doesn't this work?" moments.

StepClauseWhat It Does
1FROMIdentifies source tables, applies joins, produces virtual table
2WHEREFilters rows from the FROM virtual table using predicates
3GROUP BYGroups remaining rows by specified columns
4HAVINGFilters groups using predicates (aggregate conditions)
5SELECTDefines which columns to return, applies aliases
6ORDER BYSorts the final result set
-- Full example showing all clauses in logical order
USE TSQL;
SELECT EmployeeId, YEAR(OrderDate) AS OrderYear
FROM Sales.Orders
WHERE CustomerId = 71
GROUP BY EmployeeId, YEAR(OrderDate)
HAVING COUNT(*) > 1
ORDER BY EmployeeId, OrderYear;

Critical consequence: SELECT aliases (like OrderYear above) are NOT visible to WHERE or HAVING — those clauses execute BEFORE SELECT. Only ORDER BY can see SELECT aliases because it executes last. To filter on a computed value, either repeat the expression in WHERE/HAVING, or wrap the query in a CTE or subquery.
