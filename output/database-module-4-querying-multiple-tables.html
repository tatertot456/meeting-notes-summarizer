Understanding Joins

▶

The FROM clause determines the source tables for a SELECT statement. Its result is a virtual table consumed by subsequent clauses (WHERE, GROUP BY, SELECT, ORDER BY). Joins combine rows from multiple tables into that virtual table.

Join TypeDescription
CROSSCombines all rows from both tables — creates a Cartesian product. No ON clause.
INNERStarts with Cartesian product, applies filter to match rows based on predicate. Only matching rows returned.
OUTER (LEFT/RIGHT/FULL)All rows from designated preserved table returned. Matching rows from other table retrieved. NULLs inserted where no match exists.

Always use ANSI SQL-92 syntax with the JOIN keyword and ON clause. The old SQL-89 syntax joins tables with commas in the FROM clause and filters in WHERE — this makes Cartesian products easy to accidentally create if you forget the WHERE condition. SQL-89 is NOT recommended.

-- ANSI SQL-92 (preferred)
SELECT o.orderid, o.orderdate, od.productid
FROM Sales.Orders AS o
JOIN Sales.OrderDetails AS od
ON o.orderid = od.orderid;

-- ANSI SQL-89 (avoid — accidental Cartesian product risk)
SELECT o.orderid, od.productid
FROM Sales.Orders AS o, Sales.OrderDetails AS od
WHERE o.orderid = od.orderid;

Cartesian Product

▶

A Cartesian product combines every row from one table with every row from another table — producing all possible combinations. It is the mathematical foundation from which all join types start.

Example: 3 names × 3 products = 9 rows in the Cartesian product:

-- 3 employees × 3 products = 9 result rows
Names: Davis, Funk, King
Products: Alice Mutton, Crab Meat, Ipoh Coffee

Result:
Davis  | Alice Mutton
Davis  | Crab Meat
Davis  | Ipoh Coffee
Funk   | Alice Mutton
Funk   | Crab Meat
...and so on

Inner join = Cartesian product + filter (ON predicate). Outer join = Cartesian product + filter + add back non-matching rows with NULLs. Cross join = raw Cartesian product with no filter. Understanding this relationship explains why "forgetting the ON clause" accidentally turns an inner join into a Cartesian product.

Inner Joins

▶

INNER JOIN returns only rows where a match is found in BOTH input tables. Non-matching rows from either table are excluded. When the join predicate uses =, it is also called an equi-join.

-- Basic inner join: orders with their details
SELECT o.orderid, o.orderdate,
    od.productid, od.unitprice, od.qty
FROM Sales.Orders AS o
JOIN Sales.OrderDetails AS od
ON o.orderid = od.orderid;

-- Join on single attribute: products with their categories
SELECT C.categoryname, P.productname, P.unitprice
FROM Production.Categories AS C
JOIN Production.Products AS P
ON C.categoryid = P.categoryid;

-- Composite join: match on multiple columns
-- Cities and countries where both customers AND employees live
SELECT DISTINCT e.city, e.country
FROM Sales.Customers AS c
JOIN HR.Employees AS e
ON c.city = e.city
AND c.country = e.country;

-- Three-table join
SELECT o.orderid, c.companyname, p.productname
FROM Sales.Orders AS o
JOIN Sales.Customers AS c ON o.custid = c.custid
JOIN Sales.OrderDetails AS od ON o.orderid = od.orderid
JOIN Production.Products AS p ON od.productid = p.productid;

Table order in an INNER JOIN does not matter — the result is the same regardless of which table is listed first. Use table aliases to qualify column names when joining — especially when both tables have columns with the same name. Filter for join purposes goes in ON; filter for results goes in WHERE — this logical separation improves readability even when the query optimizer may treat them identically.

Outer Joins

▶

OUTER JOINs return all rows from the "preserved" table plus matching rows from the other table. Where no match exists, NULLs fill the missing columns.

LEFT OUTER JOIN

All rows from the LEFT table are preserved. Matching rows from right table retrieved. Non-matching right rows → NULL placeholders.

RIGHT OUTER JOIN

All rows from the RIGHT table are preserved. Matching rows from left table retrieved. Non-matching left rows → NULL placeholders.

FULL OUTER JOIN

All rows from BOTH tables preserved. NULLs fill in where either side has no match. Rarely needed but useful for finding discrepancies.

Anti-Join Pattern

LEFT OUTER JOIN + WHERE right.col IS NULL returns only rows from the left table with NO match in the right table. Classic pattern for "find customers with no orders."

-- All customers WITH order info (NULL if no orders)
SELECT c.custid, c.contactname, o.orderid, o.orderdate
FROM Sales.Customers AS c
LEFT OUTER JOIN Sales.Orders AS o
ON c.custid = o.custid;

-- Customers who have placed NO orders (anti-join)
SELECT c.custid, c.contactname
FROM Sales.Customers AS c
LEFT OUTER JOIN Sales.Orders AS o
ON c.custid = o.custid
WHERE o.orderid IS NULL;  -- filter to non-matching rows only

-- RIGHT OUTER JOIN: same result, different table order
SELECT c.custid, c.contactname, o.orderid
FROM Sales.Orders AS o
RIGHT OUTER JOIN Sales.Customers AS c
ON o.custid = c.custid;

Best practice: always use LEFT OUTER JOIN and place the preserved table on the left. Mixing LEFT and RIGHT joins in complex queries becomes confusing. Rewrite RIGHT joins as LEFT joins by swapping table order — the result is identical and the query is easier to read.

Cross Joins

▶

CROSS JOIN produces the Cartesian product of two tables — every row from the first table combined with every row from the second. No ON clause is used or needed.

-- ANSI SQL-92 CROSS JOIN syntax
SELECT e1.firstname, e2.lastname
FROM HR.Employees AS e1
CROSS JOIN HR.Employees AS e2;

-- ANSI SQL-89 equivalent (same result)
SELECT e1.firstname, e2.lastname
FROM HR.Employees AS e1, HR.Employees AS e2;

Cross joins are usually NOT desired in production queries — a 1,000-row table crossed with a 1,000-row table produces 1,000,000 rows. Legitimate uses: generating test data (all combinations of inputs), creating a numbers/dates table (sequence generator), or building reporting grids where every combination is needed.

Self Joins

▶

A self join joins a table to itself — creating two logical instances of the same physical table. At least one alias is required to distinguish the two instances. Used to compare rows within the same table.

-- Inner self join: employees and their managers
-- CEO has no manager → CEO's row is excluded (inner join)
SELECT e.empid, e.lastname, e.title,
    e.mgrid, m.lastname AS manager_lastname
FROM HR.Employees AS e
JOIN HR.Employees AS m
ON e.mgrid = m.empid;

-- Left outer self join: include CEO (no manager = NULL)
SELECT e.empid, e.lastname, e.title,
    m.lastname AS manager_lastname
FROM HR.Employees AS e
LEFT OUTER JOIN HR.Employees AS m
ON e.mgrid = m.empid;
-- CEO row: manager_lastname = NULL

Self joins are essential for hierarchical data (employees/managers, categories/subcategories, bill of materials). The key technique is giving the same table two different aliases — one representing the "child" role and one representing the "parent" role — then joining on the relationship column (mgrid = empid).
