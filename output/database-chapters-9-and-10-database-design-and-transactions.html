Chapter 9 — Database Design

Systems Development Life Cycle (SDLC)

▶

The SDLC traces the history of an information system and provides the framework within which database design and application development are mapped and evaluated. It is iterative rather than sequential.

Planning

General overview of the company and its objectives. Identifies the need for and extent of the information system.

Analysis

Problems defined during planning are examined in detail. Requirements gathering and analysis of current systems.

Detailed Systems Design

Designer completes the design of the system's processes, interfaces, and data structures.

Implementation

Hardware, DBMS software, and application programs are installed. Database design is physically implemented.

Testing

Ensures software performs as expected and meets stated requirements. Includes integration, performance, and user acceptance testing.

Maintenance

Corrective (fix bugs), adaptive (respond to environment changes), and perfective (improve performance) maintenance.

Database Life Cycle (DBLC)

▶

The DBLC contains six phases describing the history of the database within an information system:

PhaseDescription
1. Initial StudyDefine problems, constraints, objectives, scope, and boundaries
2. Database DesignEnsure the final product meets user and system requirements
3. Implementation & LoadingDBMS installed, database created, data loaded or converted
4. Testing & EvaluationDatabase tested, fine-tuned, and evaluated. Backup strategy established.
5. OperationProblems identified and solutions implemented in production
6. Maintenance & EvolutionPreventative, corrective, adaptive improvements

Three backup types in Phase 4: Full backup (all objects), Differential backup (only changes since last full), Transaction log backup (only uncommitted log operations). Database failures can be caused by software bugs, hardware failures, programming exceptions, deadlocks, or external disasters (fire, flood, lightning).

Conceptual Design

▶

Goal: design a database independent of database software and physical details. The conceptual data model describes main entities, attributes, relationships, and constraints in a software/hardware-independent way.

Minimum data rule: All that is needed is there, and all that is there is needed.

Steps to develop the conceptual model using ER diagrams:

  1. Identify, analyze, and refine the business rules
  2. Identify the main entities from Step 1
  3. Define relationships among entities
  4. Define attributes, primary keys, and foreign keys for each entity
  5. Normalize the entities
  6. Complete the initial ER diagram
  7. Validate the ER model against end users' requirements
  8. Modify the ER model based on validation results

Data model verification checks the model against proposed system processes. Key concepts: Module (IS component handling a specific business function), Cohesivity (strength of relationships among module's entities), Module coupling (degree of independence between modules — low coupling is better).

Distributed database design: Portions of the database may reside in different physical locations. A database fragment is a subset stored at a given location. Ensures integrity, security, and performance across distributed environments.

Logical & Physical Design

▶

Logical Design goal: design an enterprise-wide database based on a specific data model but independent of physical-level details. Maps all conceptual model objects to the constructs used by the selected database model.

Logical design steps:

  1. Map the conceptual model to logical model components (strong entities, supertype/subtype, weak entities, binary/higher-degree relationships)
  2. Validate using normalization
  3. Validate integrity constraints
  4. Validate against user requirements

Physical Design deals with actual data storage organization and access characteristics — defining storage organization, integrity and security measures, and performance optimizations. This is where indexes, partitioning, and physical file layout decisions are made.

Design Strategies

▶

Top-Down Design

Starts by identifying data sets (entities) then defines data elements (attributes) for each. Works from the big picture down to details. Suitable for smaller, well-understood problem domains.

Bottom-Up Design

First identifies data elements (attributes) then groups them into data sets (entities). Works from details up to the big picture. Useful when starting from existing data or documents.

Centralized Design

All design decisions made by a small central group. Suitable for top-down approaches with smaller, single-unit problem domains. Consistent and controlled.

Decentralized Design

Conceptual design models subsets of requirements independently, then aggregated into a complete design. Typical for complex systems with many objects and procedures across multiple departments.

Chapter 10 — Transaction Management and Concurrency Control

What Is a Transaction?

▶

A transaction is a logical unit of work that must be entirely completed or aborted — no partial completion permitted. The database must begin and end in a consistent state.

Transactions end when one of four events occurs: COMMIT reached, ROLLBACK reached, end of program, or abnormal termination.

-- Without transactions: Order can exist without Payment
BEGIN TRY
    INSERT INTO Sales.Orders ...       -- succeeds
    INSERT INTO Sales.OrderDetails ... -- fails
END TRY
BEGIN CATCH
    -- Order row still exists! Data integrity violated.
    SELECT ERROR_NUMBER();
END CATCH;

-- With transactions: all-or-nothing
BEGIN TRY
    BEGIN TRANSACTION;
        INSERT INTO Sales.Orders ...
        INSERT INTO Sales.OrderDetails ...
    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    ROLLBACK TRANSACTION;  -- undo everything
    SELECT ERROR_NUMBER();
END CATCH;

Individual DML statements (INSERT, UPDATE, DELETE) are automatically treated as stand-alone transactions. Explicit transactions group multiple statements into a single logical unit. Transactions are local to a connection — they cannot span connections. Keep transactions as short as possible to minimize locking.

XACT_ABORT: SET XACT_ABORT ON causes SQL Server to automatically roll back the entire transaction when any runtime error occurs — simplifying error handling by eliminating the need for explicit ROLLBACK in each CATCH block.

ACID Properties

▶

Atomicity

All operations of a transaction must complete — or the entire transaction is aborted. No partial completion. The "all or nothing" property.

Consistency

The database must be in a consistent state before and after a transaction. All integrity constraints must be satisfied. A transaction cannot leave the database in a broken state.

Isolation

Data used by a transaction cannot be used by a second transaction until the first completes. Concurrent transactions appear to execute serially from each other's perspective.

Durability

Once a transaction is committed, changes are permanent and cannot be undone or lost — even if the system fails immediately after commit. Enforced by the transaction log.

Serializability: the result of concurrently executing transactions must be equivalent to some serial (one-at-a-time) execution order. This is the gold standard for concurrency correctness.

Concurrency Control Problems

▶

Concurrent transactions on shared data create three main problems:

Lost Update

Two transactions read the same value, both compute a new value based on it, and both write back — the first write is overwritten by the second, losing the first transaction's change.

Uncommitted Data (Dirty Read)

Transaction T2 reads data modified by T1 before T1 commits. If T1 rolls back, T2 has worked with data that never officially existed.

Inconsistent Retrieval

A transaction reads the same data twice and gets different results because another transaction modified it between the two reads. Also called a non-repeatable read.

Phantom Read

A transaction runs the same query twice and gets additional rows the second time — because another transaction inserted matching rows between the two query executions.

The scheduler establishes execution order for concurrent transactions, interleaving operations to ensure serializability. It creates a serializable schedule where the interleaved result equals some serial execution.

Locking & Deadlocks

▶

Shared Lock (S)

Multiple transactions can hold simultaneously. Allows concurrent reads. Blocks exclusive locks — prevents writes while reading.

Exclusive Lock (X)

Only one transaction can hold. Blocks all other locks. Required for writes. No other transaction can read or write the locked resource.

Lock Granularity

Row locking: highest concurrency, highest overhead. Page locking: middle ground. Table locking: lowest concurrency, lowest overhead. SQL Server automatically escalates locks when too many row locks exist.

Two-Phase Locking

Growing phase: acquire all needed locks, release none. Shrinking phase: release locks, acquire none. Guarantees serializability — but NOT deadlock prevention.

Deadlock (deadly embrace): T1 holds lock A and waits for lock B. T2 holds lock B and waits for lock A. Neither can proceed. SQL Server's Lock Manager detects deadlocks every 5 seconds, selects a victim, and terminates it. Use SQL Server Profiler's deadlock graphs to analyze and prevent deadlocks.

Time stamping assigns a global unique timestamp to each transaction — older transactions take priority. Wait/Die: older waits, younger dies. Wound/Wait: older preempts younger, younger waits for older. Optimistic concurrency: assumes conflicts are rare — no locks, validate before commit, roll back if conflict detected.

Database Recovery

▶

Database recovery restores the database from a given state to a previously consistent state. Based on the atomic transaction property — if a transaction cannot complete, it must be rolled back entirely.

Write-Ahead Log Protocol

Transaction logs are always written BEFORE the data pages are updated. Ensures recovery can reconstruct any committed transaction even if data pages weren't flushed to disk.

Checkpoints

SQL Server periodically writes all modified in-memory buffers to disk. Checkpoints reduce recovery time — transactions committed before the last checkpoint don't need to be redone.

Deferred Write (Deferred Update)

Transaction operations don't immediately update the physical database — only the log is updated. Changes applied only after successful commit.

Write-Through (Immediate Update)

Database is immediately updated during transaction execution. Log records the "before" image for rollback if needed.

Recovery process:

  • Find last checkpoint in transaction log
  • If committed BEFORE checkpoint: nothing to do — already on disk
  • If committed AFTER checkpoint: REDO the transaction from log
  • If ROLLBACK after checkpoint: UNDO operations using "before" values in log

The transaction log is the foundation of all recovery. Redundant transaction logs ensure that a physical disk failure won't prevent recovery. Always maintain multiple backup types (full, differential, transaction log) and test recovery procedures regularly.
