OOP vs Procedural Programming

▶

Procedural Programming

Focus is on the process — what actions occur. Data and functions are separate. If data structures change, many functions must change too. Complex function hierarchies become hard to maintain and extend.

Object-Oriented Programming

Focus is on data and the functions that operate on it. Objects are instances of ADTs that bundle data and behavior together. Changes to one class don't ripple through unrelated code.

OOP key concepts: Encapsulation — bundling data and functions together and hiding internal details. Data hiding — restricting access to certain members. Public interface — the functions the outside world can use to interact with the object. A class is like a blueprint; objects are the houses built from it.

Classes and Objects

▶

A class is like a struct but can include both member variables (attributes) and member functions (methods/behaviors). An object is an instance of a class.

// Class declaration
class Rectangle
{
private:
    double width;      // member variables (attributes)
    double length;
public:
    void setWidth(double w);    // member function prototypes
    void setLength(double l);
    double getWidth() const;
    double getLength() const;
    double getArea() const;
};

// Define an object (instance of the class)
Rectangle r;

// Access public members with dot operator
r.setWidth(5.2);
r.setLength(10.0);
cout << r.getArea();

// Pointer to an object
Rectangle *rPtr = nullptr;
rPtr = &r;
rPtr->setLength(12.5);
cout << rPtr->getLength();

// Dynamically allocated object
Rectangle *dynRect = new Rectangle(10, 20);
delete dynRect;

A class declaration only defines the TYPE — no memory is allocated. Defining an object (Rectangle r;) allocates the memory. The dot operator (.) accesses members of an object. The arrow operator (->) accesses members through a pointer. Attempting to access a private member from outside the class causes a compiler error.

Access Specifiers

▶
SpecifierWho Can AccessTypical Use
publicAny code, inside or outside the classMember functions that form the public interface
privateOnly member functions of the class itselfData members, internal helper functions
class Rectangle
{
private:                    // data is hidden from outside
    double width;
    double length;
public:                     // interface is accessible from outside
    void setWidth(double w);
    void setLength(double l);
    double getWidth() const;
    double getArea() const;
private:                    // can appear multiple times, any order
    void validate(double);  // private helper function
};

If no access specifier is listed in a class, members default to private. This is different from struct, where the default is public. Always make data members private — this protects them from being directly modified by code outside the class, preventing data corruption.

Member Functions: Accessors, Mutators & Inline

▶

Mutator: sets or changes a private member variable. Accessor: retrieves a value — should be marked const since it doesn't change object data.

// Prototype goes inside the class declaration
class Rectangle
{
private:
    double width = 0.0;    // in-place initialization (C++11)
    double length = 0.0;
public:
    void setWidth(double w);           // mutator
    double getWidth() const;           // accessor — marked const
    double getArea() const;            // computed, never stored (avoids stale data)

    // Inline: defined directly in the class (small functions)
    double getLength() const { return length; }
};

// Definition outside class: use scope resolution operator ::
void Rectangle::setWidth(double w)
{
    width = w;
}

double Rectangle::getWidth() const
{
    return width;
}

double Rectangle::getArea() const
{
    return width * length;    // computed on demand — never stale
}

Avoid storing computed values (like area) as member variables — if width or length changes and you forget to update area, the stored value becomes stale. Instead, compute them in const member functions on demand. Inline functions are copied into the call site at compile time — faster execution but larger binary. Use inline only for short, simple function bodies.

Constructors

▶

A constructor is a member function automatically called when an object is created. It initializes the object. Constructor name = class name. No return type.

class Rectangle
{
private:
    double width;
    double length;
public:
    Rectangle();                    // default constructor (no args)
    Rectangle(double w, double l);  // parameterized constructor
    Rectangle(double = 0, double = 0); // default args — also a default constructor
};

// Default constructor definition
Rectangle::Rectangle()
{
    width = 0.0;
    length = 0.0;
}

// Parameterized constructor
Rectangle::Rectangle(double w, double l)
{
    width = w;
    length = l;
}

// Creating objects
Rectangle r1;           // calls default constructor
Rectangle r2(10, 5);    // calls parameterized constructor

// Overloaded constructors — different parameter lists
Rectangle();
Rectangle(double);
Rectangle(double, double);

If you write NO constructor, C++ generates a default constructor that does nothing. If you write ANY constructor that takes arguments but NO default constructor, then Rectangle r; becomes a compiler error — you must pass arguments. Constructor delegation (C++11) lets one constructor call another in the same class to avoid code duplication. Never define both Rectangle() and Rectangle(int = 0) — the compiler can't tell which to call with no arguments.

Destructors

▶

A destructor is automatically called when an object is destroyed (goes out of scope or deleted). Name is ~ClassName. No return type, no parameters, cannot be overloaded — only one per class.

class InventoryItem
{
private:
    char *description;   // dynamically allocated
public:
    InventoryItem(const char *desc);
    ~InventoryItem();    // destructor prototype
};

// Destructor definition — releases dynamic memory
InventoryItem::~InventoryItem()
{
    delete [] description;    // free memory allocated in constructor
}

// With dynamic objects:
Rectangle *r = new Rectangle(10, 20);
// ... use r ...
delete r;   // destructor called here, then memory freed

If a constructor allocates dynamic memory with new, the destructor MUST release it with delete. Without a destructor, the dynamically allocated memory leaks — it's never freed. Local objects have their destructors called automatically when they go out of scope. Dynamically allocated objects need explicit delete.

Arrays of Objects

▶ 
// Array of objects — default constructor called for each
InventoryItem inventory[40];

// Initialize with constructor arguments (initializer list)
InventoryItem inventory[3] = { "Hammer", "Wrench", "Pliers" };

// Multi-argument constructor — use function call syntax
InventoryItem inventory[3] =
{
    InventoryItem("Hammer", 5, 9.99),
    InventoryItem("Wrench", 10, 14.99),
    InventoryItem("Pliers", 7, 7.99)
};

// Different constructors per element
InventoryItem inventory[3] =
{
    InventoryItem("Hammer", 5, 9.99),
    InventoryItem("Wrench"),
    InventoryItem()
};

// Access: subscript + dot operator
inventory[2].setUnits(30);
cout << inventory[2].getUnits();

// Dynamically allocated array — default constructor per element
InventoryItem *inv = new InventoryItem[howMany];
delete [] inv;

Separating Specification from Implementation

▶

Professional C++ code separates class declaration from implementation into two files:

Rectangle.h (Header)

Contains the class declaration — member variable definitions and function prototypes. Programs that use the class #include this file.

Rectangle.cpp (Implementation)

Contains the full member function definitions. Must #include "Rectangle.h". Compiled and linked with the program.

// Rectangle.h — header guard prevents multiple inclusion
#ifndef RECTANGLE_H
#define RECTANGLE_H

class Rectangle
{
private:
    double width;
    double length;
public:
    Rectangle();
    Rectangle(double, double);
    void setWidth(double);
    void setLength(double);
    double getWidth() const;
    double getLength() const;
    double getArea() const;
};

#endif

// Rectangle.cpp
#include "Rectangle.h"

Rectangle::Rectangle()
{
    width = 0.0;
    length = 0.0;
}
// ... rest of definitions ...

The #ifndef / #define / #endif header guard prevents the header from being included multiple times in the same translation unit — which would cause duplicate declaration errors. This is standard practice for all C++ header files.
