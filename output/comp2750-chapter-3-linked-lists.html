Introduction — Why Linked Lists?

▶

Arrays suffer from two significant limitations that linked lists overcome:

Fixed Size

Array size must be known at compile time. Too large wastes memory. Too small causes overflow. Linked lists grow and shrink dynamically.

Expensive Insertion/Deletion

Array elements are contiguous in memory — inserting or deleting requires shifting potentially all elements. Linked lists only update a few pointers.

A linked list is a collection of independent memory locations called nodes. Each node stores data and a link (pointer/address) to another node. Moving between nodes means following these links — the nodes can be anywhere in memory.

The most common implementation uses pointers, providing great flexibility. The tradeoff: linked lists lose random access (you can't jump directly to element i like an array — you must traverse from the head). Arrays win for random access; linked lists win for frequent insertions and deletions.

Singly Linked Lists

▶

Each node contains data (info) and a pointer to the next node. A single head pointer accesses the entire list. The last node's next pointer is null.

// Node definition — self-referential class
class IntSLLNode {
public:
    int info;               // data stored in this node
    IntSLLNode *next;       // pointer to next node (same type!)

    IntSLLNode() {
        next = nullptr;     // initialize to null — no dangling pointer
    }
    IntSLLNode(int i, IntSLLNode *in = nullptr) {
        info = i;
        next = in;
    }
};

// List class — encapsulates the node class
class IntSLList {
    IntSLLNode *head;   // pointer to first node
    IntSLLNode *tail;   // pointer to last node (avoids traversal to end)
public:
    // Core operations
    bool isEmpty() { return head == nullptr; }
    void addToHead(int);
    void addToTail(int);
    int  deleteFromHead();
    int  deleteFromTail();
    void deleteNode(int);   // delete by value
    bool isInList(int);     // search
};

Insertion at the beginning:

void IntSLList::addToHead(int el) {
    head = new IntSLLNode(el, head);  // new node's next = old head
    if (tail == nullptr)              // list was empty
        tail = head;
}
// Steps: create node → set next to old head → update head

Insertion at the end:

void IntSLList::addToTail(int el) {
    if (isEmpty()) {
        head = tail = new IntSLLNode(el);
    } else {
        tail->next = new IntSLLNode(el);  // old tail points to new node
        tail = tail->next;                // update tail
    }
}
// Steps: create node → old tail's next = new node → update tail

Deletion from beginning:

int IntSLList::deleteFromHead() {
    int el = head->info;           // save value to return
    IntSLLNode *tmp = head;
    if (head == tail)              // only one node
        head = tail = nullptr;
    else
        head = head->next;        // advance head
    delete tmp;                    // free memory
    return el;
}

Deletion from end: requires traversal to find the second-to-last node (no prev pointer!)

// Must traverse entire list to find node before tail
IntSLLNode *tmp = head;
while (tmp->next != tail)   // find predecessor of tail
    tmp = tmp->next;
// Now tmp is second-to-last
delete tail;
tail = tmp;
tail->next = nullptr;

Delete specific node by value: requires two pointers

// pred trails behind tmp during traversal
IntSLLNode *pred = head, *tmp = head->next;
while (tmp != nullptr && tmp->info != el) {
    pred = tmp;
    tmp = tmp->next;
}
if (tmp != nullptr) {
    pred->next = tmp->next;  // "link around" the target node
    if (tmp == tail)
        tail = pred;
    delete tmp;
}

Special cases to always handle: empty list (deletion fails gracefully), single node (head and tail both become null), deleting the first or last node (head/tail pointers need updating). Missing any of these leads to memory leaks or dangling pointers.

Doubly Linked Lists

▶

Adding a prev pointer to each node allows traversal in both directions. This eliminates the need to scan forward to find the predecessor when deleting from the end or middle.

class IntDLLNode {
public:
    int info;
    IntDLLNode *next;
    IntDLLNode *prev;   // NEW — points to previous node

    IntDLLNode(int i, IntDLLNode *n = nullptr, IntDLLNode *p = nullptr)
        : info(i), next(n), prev(p) { }
};

Insertion at end:

// New node's prev = current tail
// New node's next = nullptr
// Old tail's next = new node
// Update tail to new node
// Special case: if list was empty, head = tail = new node

new_node->prev = tail;
new_node->next = nullptr;
if (tail) tail->next = new_node;
else head = new_node;
tail = new_node;

Deletion from end: no traversal needed!

int el = tail->info; IntDLLNode *tmp = tail; tail = tail->prev; // jump directly to predecessor if (tail) tail->next = nullptr; else head = nullptr; // list now empty delete tmp; return el;

The key advantage of doubly linked lists: deletion from the end or middle is O(1) instead of O(n). With a singly linked list, finding the predecessor requires O(n) traversal. With a doubly linked list, every node already has a prev pointer — no traversal needed. The cost: each node requires an extra pointer (more memory overhead).

The STL implements its list container as a doubly linked list with head and tail pointers — use #include <list>.

Circular Lists

▶

In a circular list, the last node's next pointer points back to the first node — forming a ring. Only one permanent pointer (tail) is needed, since tail→next gives access to the head.

void addToTail(int el) {
    if (isEmpty()) {
        tail = new IntSLLNode(el);
        tail->next = tail;           // points to itself!
    } else {
        tail->next = new IntSLLNode(el, tail->next);
        // new node's next = old head (tail->next was head)
        tail = tail->next;           // advance tail to new node
    }
}

Circular lists are useful for applications that cycle through data repeatedly — round-robin scheduling, buffers, game loops. The drawback: deleting from the end still requires finding the predecessor by traversal (like singly linked lists). Making the circular list doubly linked solves this by providing two rings — one forward through next pointers, one backward through prev pointers.

Skip Lists

▶

The biggest drawback to linked lists is sequential search — finding an element always requires starting from the head and scanning one by one. A skip list adds "express lanes" — higher-level pointers that skip over multiple nodes.

Structure

Nodes have varying numbers of pointers (levels). Higher-level pointers skip more nodes. Level 1 = direct links. Level 2 = skip every other. Level 4 = skip every 8th. etc.

Search

Start at the highest level. Follow pointers until you overshoot or find target. Drop to next level. Repeat. Much faster than sequential — O(log n) average.

Insertion

Assign new node a random level (using probability distribution). Number of nodes at each level stays balanced: 8 at level 1, 4 at level 2, 2 at level 3, 1 at level 4 (for 15 elements).

Node Structure

Each node points to an array of pointers whose size equals the node's level. Avoids giving every node the maximum number of pointers (wasteful).

Skip lists trade insertion/deletion complexity for search speed. The level of a new node is determined randomly, which maintains the probabilistic balance without requiring expensive restructuring. Skip lists achieve O(log n) average search time — the same as a balanced binary search tree — but with simpler implementation.

Self-Organizing Lists

▶

Instead of a fixed structure, self-organizing lists dynamically reorganize as they are used — moving frequently accessed items closer to the front to reduce future search time.

TechniqueWhen element foundWhen element not found
Move-to-FrontMove target to head of listInsert at end
TransposeSwap target with its predecessorInsert at end
CountIncrement access counter; reorder by countInsert at end with count=1
OrderingNo change — list is ordered on data valueInsert in sorted position

Move-to-front and count perform best in practice. The goal: items accessed frequently end up near the head, reducing average search length. This works well when access patterns are non-uniform (some items accessed much more than others). For small lists, simple sequential search is fine — these techniques pay off as list size and access frequency increase.

Sparse Tables

▶

A sparse table is a table that is mostly unoccupied. Storing it as a full 2D array wastes enormous amounts of memory.

Example: Student grades — 8,000 students × 300 classes

ApproachStorageWasted
Full 2D array (1 byte/grade)2,400,000 bytes~99% (if avg 4 classes/student)
Two parallel arrays417,000 bytes~17% — less waste but inflexible, data duplicated
Two arrays of linked lists288,000 bytes0% wasted — no fixed allocation, fully flexible

Linked list approach: Two arrays of linked lists — one indexed by student (lists all classes a student takes), one indexed by class (lists all students in that class). Each node stores:

  • Student number
  • Class number
  • Grade
  • Pointer to next class for this student
  • Pointer to next student in this class

The linked list approach uses only 10% of the original table's space with NO wasted storage — and the lists expand or contract as enrollment changes, unlike fixed arrays. The tradeoff: slightly more complex code and pointer overhead per node. For truly sparse data, this is almost always the right choice.

Lists in the Standard Template Library

▶

The STL implements std::list as a generic doubly linked list with head and tail pointers.

#include <list>

list<int> myList;

// Common operations
myList.push_front(10);    // insert at beginning
myList.push_back(20);     // insert at end
myList.pop_front();       // remove from beginning
myList.pop_back();        // remove from end
myList.front();           // access first element
myList.back();            // access last element
myList.empty();           // check if empty
myList.size();            // number of elements
myList.clear();           // remove all elements

// Iterate with iterator
for (auto it = myList.begin(); it != myList.end(); ++it)
    cout << *it << " ";

// Range-based for
for (int x : myList)
    cout << x << " ";

Arrays vs Linked Lists

▶
OperationArrayLinked List
Random access by indexO(1) — directO(n) — must traverse
Search (unsorted)O(n)O(n)
Insert at beginningO(n) — shift allO(1)
Insert at endO(1) amortizedO(1) with tail pointer
Delete at beginningO(n) — shift allO(1)
Delete at end (singly)O(1)O(n) — find predecessor
Delete at end (doubly)O(1)O(1)
Memory overheadLow — just dataHigher — data + pointer(s)
Size flexibilityFixed at compile timeDynamic — grows/shrinks

Use arrays when: you need random access by index, data size is known and stable, memory overhead matters. Use linked lists when: frequent insertions/deletions at arbitrary positions, data size varies significantly, or you're implementing stacks/queues/deques. The STL provides both — vector for array-like needs, list for linked-list needs.
