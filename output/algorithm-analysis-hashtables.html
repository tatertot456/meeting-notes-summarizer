Basics of Hashing

▶

Hashing transforms a key (typically a string) into a numeric index, enabling direct access to stored values — no searching required.

Hashing achieves O(1) average-case for search, insert, and delete — better than any tree structure. The tradeoff: no natural order, harder to enumerate, performance depends on the hash function quality.

Hash Functions

▶

A good hash function must be:

  • Reflexive: Always returns the same value for the same object
  • Symmetric: If x.equals(y), then x and y must have the same hash code
  • Uniformly distributed: Maps keys as evenly as possible across all buckets
// Basic Java hash function
private int myhash(String inputString) {
    int hashVal = inputString.hashCode();
    int hashResult = hashVal % maxSize;
    return hashResult;
}

// Fixed version — handles negative hashCode values
private int myhash(String inputString) {
    int hashVal = inputString.hashCode();
    hashVal = hashVal & 0x7fffffff;  // 31-bit mask
    int hashResult = hashVal % maxSize;
    return hashResult;
}

The bug in the basic version: Java's hashCode() can return negative integers. The modulo of a negative number is negative in Java — giving an invalid array index. The 31-bit mask (0x7fffffff) strips the sign bit, ensuring a non-negative result.

Collisions

▶

A collision occurs when two different keys hash to the same bucket. This is unavoidable — by the pigeonhole principle, if there are more possible keys than buckets, some must collide.

Chaining

Each bucket contains a linked list. Colliding elements are chained together. Handles unlimited collisions but requires extra memory for list pointers.

Open Addressing (Linear Probing)

Scan forward from the collision bucket until an empty slot is found. No linked lists — memory is pre-allocated. Sensitive to clustering.

Dual Hashing

Use a second hash function to determine a 2D position — manages an N×M matrix instead of an array. Better distribution than linear probing.

Robin Hood Hashing

A variant of open addressing that reduces variance by "stealing" positions from richer (shorter probe length) entries to give to poorer ones.

Implementing Hashtable with Chaining

▶

Each bucket is a linked list. The HashTableElement needs a next pointer:

class HashTableElement {
    String key;
    Object value;
    HashTableElement next;  // linked list pointer
}

// Get: hash to bucket, traverse list for key match
// Insert: hash to bucket, traverse list:
//   - if key exists: update value
//   - if not found: insert at appropriate position
// Remove: hash to bucket, traverse list,
//   maintain previous pointer to bridge the gap

Good code practice: make insert and remove return Boolean instead of void. This lets callers detect failure (key not found for remove, insert failed) without exception handling — testing for false is simpler than catching exceptions.

The initial hash gets you to the right bucket in O(1). The linked list traversal within the bucket is O(k) where k is the chain length. With a good hash function and load factor, k averages O(1).

Open Addressing (Linear Probing)

▶

Instead of linked lists, scan the array for the next open slot:

  • Calculate hash bucket
  • If occupied, scan left/right until empty slot found
  • For retrieval, same scan — stop when key matches or empty slot found

Advantage

Memory is pre-allocated — no dynamic list allocation. Better cache performance since data is contiguous in memory.

Disadvantage

Primary and secondary clustering can degrade performance. Deletion is tricky — can't just clear a slot (would break probe chains).

Unlike arrays, you cannot simply iterate through a hashtable with a for loop. Enumeration requires a custom method that visits every bucket and traverses any chains — this is not particularly fast.

Performance Considerations

▶

Hashtable performance depends critically on two factors:

Load Factor

Entries ÷ Buckets. As this approaches 1, performance degrades. Java's default load factor is 0.75 — rehash when 75% full.

Hash Quality

Keys must distribute uniformly. Clustered keys cause long chains. Poor hash functions can turn O(1) operations into O(n).

Hashtables have no natural order. Sorting keys requires extracting all entries and sorting separately. If ordered iteration is important, consider a balanced BST (Red-Black Tree) instead — it gives O(log n) operations with natural order.

When to use a hashtable: any time you associate objects with labels (key-value pairs), TreeView UI components, dictionary objects, database indexing — and you don't need ordered access.

Java Hashtable Classes

▶

Java's java.util package provides several hashtable implementations:

Hashtable()

Default capacity 11, load factor 0.75. Thread-safe (synchronized). Legacy class — prefer HashMap for new code.

Hashtable(int initialCapacity)

Specify initial capacity. Load factor defaults to 0.75. rehash() called automatically when capacity exceeded.

HashMap

Non-synchronized, allows null keys/values. Generally preferred over Hashtable for single-threaded use.

hashCode() in Java

Inherited from Object. Implemented for all basic types (String, Integer, Date, URL). Override for custom objects — must be consistent with equals().

Calling rehash() frequently incurs significant performance penalties. Size your hashtable appropriately upfront — if you know you'll store n items, initialize with capacity n/0.75 to avoid rehashing entirely.
