Simple SELECT Statements

▶

The SELECT statement retrieves data from one or more tables or views. Every SELECT query has these core clauses evaluated in logical processing order:

ClauseExpressionLogical Order
FROM<table or view>1st
WHERE<search condition>2nd
GROUP BY<group by list>3rd
HAVING<search condition>4th
SELECT<select list>5th
ORDER BY<order by list>6th
-- Specify both schema and object name (best practice)
SELECT companyname, country
FROM Sales.Customers;

-- Wildcard: returns all columns
-- NOT recommended in production code
SELECT *
FROM Sales.Customers;

-- Always specify only the columns you need
SELECT custid, companyname, country
FROM Sales.Customers;

Always specify schema and object name together: Sales.Customers, not just Customers. This avoids ambiguity, improves performance (no schema resolution search), and makes intent clear. End all T-SQL statements with a semicolon.

Using Calculations in SELECT

▶

Scalar expressions in the SELECT clause return one value per row — calculated on the fly without storing the result.

OperatorDescription
+Add numbers or concatenate strings
-Subtract
*Multiply
/Divide
%Modulo (remainder)
-- Calculate line total per order detail row
SELECT unitprice, qty, (qty * unitprice) AS line_total
FROM Sales.OrderDetails;

-- String concatenation with +
SELECT firstname + ' ' + lastname AS fullname
FROM HR.Employees;

-- Date arithmetic
SELECT orderid, orderdate,
    DATEADD(day, 30, orderdate) AS due_date
FROM Sales.Orders;

Calculated columns return one value per row (scalar). They don't permanently store values — recalculated each query. Give calculated columns an alias using AS so results have meaningful column names instead of no name at all.

Eliminating Duplicates with DISTINCT

▶

SQL query results are not truly relational — rows are NOT guaranteed to be unique and have NO guaranteed order. Even unique source rows can produce duplicate values for selected columns.

-- Without DISTINCT: same country appears multiple times
SELECT country
FROM Sales.Customers;
-- Argentina, Argentina, Belgium, Austria, Austria...

-- With DISTINCT: each unique value appears once
SELECT DISTINCT country
FROM Sales.Customers;
-- Argentina, Austria, Belgium...

-- DISTINCT across multiple columns: unique combinations
SELECT DISTINCT companyname, country
FROM Sales.Customers;

DISTINCT removes duplicates based on the combination of ALL selected columns — not from the source table. A row is only eliminated if every selected column matches an already-seen row. Performance note: some queries improve performance by filtering duplicates early in the logical processing pipeline — before the full SELECT clause executes.

DISTINCT is applied AFTER WHERE, HAVING, and GROUP BY — so it only operates on the rows that survive those filters, not on all source rows.

Column and Table Aliases

▶

Aliases give alternative names to columns or tables. They improve readability and are essential for calculated columns.

-- Column aliases: three equivalent syntaxes
SELECT orderid, qty AS quantity         -- preferred: explicit AS
FROM Sales.OrderDetails;

SELECT orderid, quantity = qty          -- = syntax (less common)
FROM Sales.OrderDetails;

SELECT orderid, unitprice quantity      -- accidental alias! (space only)
FROM Sales.OrderDetails;               -- easy to misread — avoid

-- Table aliases in FROM clause
SELECT SO.custid, SO.orderdate
FROM SalesOrders AS SO;                -- with AS (preferred)

SELECT SO.custid, SO.orderdate
FROM SalesOrders SO;                   -- without AS (also valid)

Critical impact of logical processing order: FROM, WHERE, and HAVING are processed BEFORE SELECT. This means aliases created in the SELECT clause are NOT visible to WHERE or HAVING — only to ORDER BY. If you need to filter on a calculated value, repeat the expression in WHERE or use a CTE/derived table.

Column Alias Purpose

Give meaningful names to calculated columns. Rename columns for display. Required when a SELECT expression has no natural column name.

Table Alias Purpose

Shorten long table names. Required when self-joining (two references to same table need different aliases). Required with some table expressions.

CASE Expressions

▶

CASE expressions return a single scalar value based on conditions. They are NOT control-of-flow — they are expressions that evaluate to a value, usable anywhere a value expression is valid: SELECT list, WHERE, HAVING, ORDER BY.

Simple CASE — compares one value to a list of possible values, returns first match:

-- Simple CASE: compare one value to known options
SELECT orderid, custid,
    CASE shipcountry
        WHEN 'USA'    THEN 'Domestic'
        WHEN 'Canada' THEN 'North America'
        WHEN 'Mexico' THEN 'North America'
        ELSE 'International'
    END AS shipping_region
FROM Sales.Orders;

Searched CASE — evaluates a set of predicates, returns first TRUE:

-- Searched CASE: evaluate different conditions per branch
SELECT orderid, freight,
    CASE
        WHEN freight < 10   THEN 'Low'
        WHEN freight < 100  THEN 'Medium'
        WHEN freight >= 100 THEN 'High'
        ELSE 'Unknown'
    END AS freight_category
FROM Sales.Orders;

-- CASE in ORDER BY: custom sort order
SELECT productname, categoryid
FROM Production.Products
ORDER BY
    CASE categoryid
        WHEN 1 THEN 'Beverages'
        WHEN 2 THEN 'Condiments'
        ELSE 'Other'
    END;

CASE evaluation rules: returns the value from the FIRST matching condition — subsequent conditions are not evaluated. If no condition matches and there is no ELSE, the result is NULL. In the SELECT clause, CASE behaves as a calculated column and requires an alias for a meaningful column name.

Simple CASE

One input value compared to a list. Equivalent to a series of = comparisons. Cleaner when checking one column against known values.

Searched CASE

Each WHEN evaluates a full predicate (can involve any expression). More flexible — different columns or complex conditions per branch.
