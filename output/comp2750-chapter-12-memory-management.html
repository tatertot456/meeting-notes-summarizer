The Heap & Fragmentation

▶

When a program dynamically requests memory (via new in C++, malloc() in C), it is allocated from a region of main memory called the heap. The memory manager administers this region. In C++, memory management is the programmer's responsibility — new to allocate, delete to release.

External Fragmentation

Wasted space BETWEEN allocated segments. Even with sufficient total free memory, a request may fail because no single contiguous block is large enough. The heap is broken into small scattered pieces.

Internal Fragmentation

Wasted space WITHIN an allocated segment. Occurs when more memory is allocated than was requested (e.g., rounding to block sizes). The extra bytes inside the block are unused but unavailable.

After a series of allocations and deallocations, the heap becomes disorganized — allocated blocks mixed with free blocks of varying sizes. External fragmentation can refuse requests even with sufficient total free memory. Internal fragmentation wastes memory inside allocated blocks. Good memory management minimizes both.

Sequential-Fit Methods

▶

All free blocks are maintained in a doubly linked list. When a request arrives, the list is searched for a suitable block. How the list is searched distinguishes the methods.

MethodStrategyProsCons
First-FitAllocate the FIRST block large enoughFast — stops at first match. Most efficient overall.Scatters small fragments at the front of the list over time.
Best-FitAllocate the CLOSEST-SIZE blockMinimizes wasted space in allocated block.Leaves tiny unusable leftover fragments everywhere. Worst in practice.
Worst-FitAllocate the LARGEST blockLeftover fragments are larger and more usable.Quickly consumes large blocks needed for big requests.
Next-FitStart search where LAST search endedSimilar speed to first-fit. Avoids rescanning front repeatedly.More external fragmentation than first-fit — reaches end of list sooner.

First-fit is empirically the most efficient. Best-fit sounds optimal but produces many tiny unusable fragments — it leaves "almost perfect" fits that waste surrounding space. The order blocks are maintained matters: best-fit/worst-fit benefit from size-ordered lists; first-fit/next-fit use address-ordered lists. Returned blocks are merged with neighboring free blocks to reduce fragmentation.

Non-Sequential Fit Methods

▶

For large memories, sequential search becomes too slow. Non-sequential methods organize free blocks into multiple specialized lists.

Ross's Method (1967)

Divide memory into multiple lists, each holding same-size blocks. Larger blocks split to create smaller ones. Lists organized as a tree for efficient lookup.

Cartesian Tree Variants

Leftmost-fit and better-fit variants use a Cartesian tree structure. Only advantageous for large numbers of memory blocks (Stephenson, 1983).

Adaptive Exact-Fit (1985)

Tracks most popular block sizes from recent allocations. Maintains short lists of popular sizes. Requests satisfied immediately from list; fall back to sequential search if needed. Blocks evicted from list if unused for T allocations.

// Adaptive Exact-Fit:
// size-list = block-list entries of sizes used in last T allocations
// On request for size s:
//   if size-list has entry for s:
//     detach and return block immediately (O(1))
//   else:
//     search memory sequentially for block >= s (slow)
//     add size s to size-list
// On return of block size s:
//   if size-list has entry for s: add to list
//   else: return to general heap
// Evict size-list entry if no request for that size in last T allocations

Buddy Systems

▶

Buddy systems allocate memory in power-of-2 blocks and pair each block with a "buddy" of equal size. Two free buddies are always merged back together.

Binary Buddy System (Knowlton, 1965):

// Memory of 2^m locations: addresses 0...2^m-1
// avail[i] = doubly linked list of free blocks of size 2^i

// Block's BUDDY address: flip bit (i+1) from right in block's address
// For block at address X with size 2^i:
//   buddy(X) = X XOR 2^i

// Example (8 locations = 2^3):
// Size 1: addresses {000,001,010,011,100,101,110,111}
// Size 2: {000,010,100,110}  — buddy pairs: (000,010),(100,110)
// Size 4: {000,100}         — buddy pair:  (000,100)
// Size 8: {000}             — no buddy (entire memory)

// ALLOCATION (request size s):
// Find smallest k where 2^k >= s
// If avail[k] is not empty: take a block from avail[k]
// If empty: look in avail[k+1], split it into two buddies
//           put one in avail[k], return the other
// Continue splitting up until a block is found or fail

// DEALLOCATION (return block of size 2^k at address X):
// Check if buddy is free
// If yes: merge, remove buddy from avail[k], put merged block in avail[k+1]
// Check new buddy at level k+1... repeat until buddy not free or reach top

Binary buddy system internal fragmentation: a request for 33 locations gets a 64-location block — nearly 50% waste! Requests slightly above a power of 2 are worst case. External fragmentation can also occur: two adjacent free blocks of different sizes can't be merged (they're not buddies). Long-term, up to 50% of memory can be wasted with pathological request patterns.

Fibonacci Buddy System (Hirschberg, 1973)

Block sizes are Fibonacci numbers: 1,2,3,5,8,13,... More block size options reduces internal fragmentation. Finding buddies is more complex — need bit flags (buddy-bit and memory-bit) per block.

Weighted Buddy System (Shen & Peterson, 1974)

Block sizes: 2^k AND 3×2^k. Even more size options. Simulations show worse external fragmentation and slower than binary buddy. Requires 2-bit flag per block.

Dual Buddy System (Page & Hagins, 1986)

Two separate memory areas: one binary buddy, one with sizes 3,6,...,3×2^j. Internal fragmentation midway between binary and weighted. External fragmentation similar to binary.

Internal vs External Tradeoff

Inverse relationship: reducing internal fragmentation (more block sizes) tends to create more external fragmentation (small splinter blocks). No perfect solution exists.

Garbage Collection — Overview

▶

Some languages (LISP, Java, Python) handle memory reclamation automatically. The programmer uses new freely and simply abandons unneeded memory — the garbage collector reclaims it automatically.

Root Set

All root pointers — the starting points from which all reachable data is accessible. Everything reachable from the root set is "live" (in use). Everything else is garbage.

Marking Phase

Identify all currently used (reachable) cells by traversing the root set. Mark every reachable cell. Unreachable cells = garbage.

Reclamation Phase

Return all unmarked cells to the free memory pool. May also include compaction — moving live cells together to reduce fragmentation.

LISP Cell Structure

Cells have head (car) and tail (cdr) pointers, plus header with atom/nonatom flag and marked/unmarked flag. Variable-size cells also store size in header.

Mark-and-Sweep (McCarthy, 1960)

▶

The classical garbage collection approach — two clear phases.

// SIMPLE MARKING (recursive — like preorder traversal):
marking(node):
    if node is not marked:
        mark node
        if node is not an atom:
            marking(head(node))
            marking(tail(node))
// Call for every element of root set
// PROBLEM: Can overflow the runtime stack for long lists!

// STACK-BASED MARKING (avoids stack overflow):
markingWithStack(node):
    push(node)
    while stack not empty:
        node = pop()
        while node is unmarked nonatom:
            mark node
            push(tail(node))   // save tail for later
            node = head(node)  // follow head immediately
        if node is unmarked atom: mark node
// Still can overflow if array-based stack is too small

// SCHORR-WAITE (1967) — NO STACK, uses link inversion:
// Reverse links during traversal to "remember" the way back
// tag bit: 0 = coming from head path, 1 = coming from tail path
// When marked node or atom reached: follow reversed links back
// Visits each cell TWICE — slower but uses O(1) extra memory
// Handles cycles correctly

Sweep phase:

sweep():
    for each location from last to first:
        if mark(location) == 0:
            insert location at front of availList  // it's garbage
        else:
            mark(location) = 0    // reset for next GC cycle

Compaction (Two-Pointer Method):

// Like quicksort partitioning — two pointers from opposite ends
// Left pointer: scans forward, finds unmarked (free) cells
// Right pointer: scans backward, finds marked (live) cells
// When both found: copy live cell to free location
//   Record new location in old location (forwarding address)
// After pointers cross: scan compacted area, update all pointers
//   If pointer refers to old location: look up forwarding address

// Cost: 2.5 passes (mark + move + update pointers)
// Benefit: eliminates external fragmentation, contiguous free space

Mark-and-sweep drawback: it stops program execution entirely while collecting. For long-running or real-time systems this pause is unacceptable. The Schorr-Waite algorithm trades speed (visits each cell twice) for stack safety (no overflow possible). The fastmark algorithm (Kurokawa, 1981) improves speed but can still fail on some inputs, making Schorr-Waite more reliable despite being slower.

Copying Methods — Stop-and-Copy

▶

Instead of marking garbage and sweeping it, copying methods copy all LIVE cells to a new area — garbage is simply left behind.

// STOP-AND-COPY (Fenichel & Yochelson, 1969):
// Divide heap into two equal semispaces (semispace1, semispace2)
// Allocate from semispace1 until full
// When full: STOP program, COPY all live cells to semispace2
//   - Traverse from root set
//   - Copy each reachable cell to semispace2 (leave forwarding address in old location)
//   - Update all pointers to refer to new locations
// semispace2 becomes the active space, program resumes

// CHENEY'S BREADTH-FIRST COPY (1970):
// No stack, no separate marking phase!
// Use two pointers in tospace: scan (processes cells) and free (next allocation spot)
// Algorithm:
//   copy all root-referenced cells to tospace
//   while scan != free:
//     for each pointer in cell at scan:
//       if pointer refers to fromspace cell not yet copied:
//         copy it to tospace (at free), leave forwarding address
//       else if already copied: use forwarding address
//     advance scan
// BFS order: naturally handles cycles (forwarding addresses prevent infinite loops)
// NEVER visits garbage — only traverses reachable cells!

Counter-intuitive property: the more garbage in memory, the FASTER the copying collector works. It only copies live cells — garbage is simply ignored. As heap size increases, the frequency and cost of collection decreases. Appel (1987) showed that with a truly large heap, no GC is needed at all. The cost: half the total memory is always reserved as "tospace" — 50% memory overhead.

Incremental Garbage Collection

▶

Standard GC suspends the program entirely — unacceptable for real-time systems. Incremental GC interleaves collector and program (mutator) execution in small chunks.

Mutator

The running program. Can modify the heap while the collector is partially done. Changes may invalidate partially-processed structures.

Cost

Incremental collectors need ~2× the processing power of stop-the-world collectors (Wadler, 1976) due to re-processing modified structures.

Baker's Incremental Copying Algorithm (1978):

// Uses fromspace and tospace (both active simultaneously)
// Two collector pointers: scan (processes cells) and bottom (free space in tospace)
// On each memory allocation:
//   1. Allocate new cell at top of tospace
//   2. Copy k cells from fromspace to tospace (collector's work quota)
//   3. BFS copies descendants, places them at end of queue (after bottom)

// READ BARRIER: Before mutator uses any pointer:
//   if pointer refers to fromspace:
//     check for forwarding address in tail field
//     if exists: use it (already copied)
//     if not: copy cell to tospace NOW, set forwarding address

// When all live cells copied: swap roles (fromspace↔tospace)

// Stanchina-Meyer improvement (2007):
// Object space (fromspace) holds objects
// Handle space (tospace) holds references to objects
// Live objects compacted IN PLACE (fromspace)
// Dead objects overwritten
// Cleanup phase removes handles and restores direct pointers

Yuasa's Noncopying Incremental GC (1990):

// Incremental mark-and-sweep — avoids copying and address changes
// k1 = cells marked per GC invocation
// k2 = cells swept per GC invocation
// Invoked after each memory allocation

// WRITE BARRIER: Before mutator overwrites head(p) or tail(p):
updateTail(p, q):
    if collector in marking phase:
        mark tail(p)       // save old value before overwriting
        st.push(tail(p))   // push onto GC stack
    tail(p) = q

// This prevents destroying paths to reachable cells not yet marked
// "Floating garbage": cells freed during sweep phase missed —
//   collected in NEXT cycle (acceptable delay)

Generational Garbage Collection

▶

Key observation: most allocated objects die young — a few survive long-term. Generational GC exploits this by collecting young generations frequently and old generations rarely.

Minor Collections

Frequent — collect young generation only. Most objects in young generation ARE garbage. Quick, cheap, handles most GC work.

Major Collections

Infrequent — collect entire heap. Expensive but rarely needed. Old objects that survived many minor collections are likely long-lived.

Tenuring

When a young object survives n collections it is "promoted" (tenured) to the older generation. Value of n depends on the GC implementation.

Remembrance Sets

Track pointers FROM older generations TO younger ones (and vice versa). Used to find all references to young generation objects without scanning the entire heap.

Train Algorithm (Hudson & Moss, 1992) — for mature objects:

// Divides MATURE object space into areas (cars) linked into trains
// One area collected per cycle — bounded pause time!

// Collection rules:
// 1. If no outside reference to any object in a train → entire train is garbage
// 2. If object reachable from root: copy to another train
//    Then copy all objects in same area reachable from it
// 3. If object referenced from another area: copy to END of that area's train
// Objects move between trains based on who references them
// Eventually circular structures gather into one train → collectible when cut off

// Remembrance sets: one per area
// Track all cross-area references
// Allow finding referenced objects without full heap scan

Generational GC efficiency depends critically on the assumption that most objects die young. If a program allocates objects with long lifetimes, minor collections become expensive as few objects are collected. The train algorithm solves the major collection pause problem by making mature object collection incremental — only one area per cycle, with bounded pause time. Hardware support can reduce GC overhead from ~50% to near-zero of program runtime.
