What Is Inheritance?

▶

Inheritance provides a way to create a new class from an existing class. The new class is a specialized version of the existing (more general) class. It establishes an "is a" relationship.

  • A poodle is a dog
  • A car is a vehicle
  • An UnderGrad is a Student
  • A Mammal is an Animal
// Base class (parent)
class Student
{
private:
    string name;
    int studentID;
public:
    void setName(string n) { name = n; }
    string getName() const { return name; }
};

// Derived class (child) — uses public inheritance
class UnderGrad : public Student
{
private:
    int hoursCompleted;
public:
    void setHours(int h) { hoursCompleted = h; }
    int getHours() const { return hoursCompleted; }
};

// An UnderGrad object has ALL Student members PLUS its own
UnderGrad ug;
ug.setName("Alice");      // inherited from Student
ug.setHours(45);          // defined in UnderGrad

A derived class object has ALL members defined in both the child AND parent class. It can use all PUBLIC members from both. Private members of the base class are inherited but cannot be accessed directly by the derived class — only through public/protected accessor functions.

Access Specifiers & Protected Members

▶

protected is a new access level — like private, but accessible to derived classes directly.

Base Memberpublic inheritanceprotected inheritanceprivate inheritance
publicpublic in derivedprotected in derivedprivate in derived
protectedprotected in derivedprotected in derivedprivate in derived
privateinaccessibleinaccessibleinaccessible

public inheritance

Most common. Derived object CAN be treated as a base class object. Maintains the "is a" relationship. Use this in most cases.

protected inheritance

More restrictive. Allows derived classes to know parent details but outside code can't treat derived as base.

private inheritance

Most restrictive. Derived objects CANNOT be treated as base class objects. Breaks the "is a" relationship.

protected members

Accessible by the class itself AND by derived classes. Invisible to outside code. Useful when derived classes need direct access to parent data.

Constructors & Destructors in Inheritance

▶

When a derived object is created, the base class constructor runs first, then the derived class constructor. When destroyed, derived destructor runs first, then base class destructor.

// Pass arguments to base class constructor
// using initializer list syntax on derived constructor
Square::Square(int side) : Rectangle(side, side)
{
    // Rectangle constructor already ran with (side, side)
    // add Square-specific initialization here
}

// If base class has NO default constructor, MUST use initializer list
class Rectangle
{
public:
    Rectangle(double w, double l);  // no default constructor
};

class Square : public Rectangle
{
public:
    Square(int side) : Rectangle(side, side) { }  // REQUIRED
};

// Constructor inheritance (C++11)
class MyBase
{
public:
    MyBase(int);
    MyBase(int, double);
};

class MyDerived : public MyBase
{
public:
    using MyBase::MyBase;   // inherit all base constructors
    // MyDerived now has MyBase(int) and MyBase(int, double)
};

The default constructor, copy constructor, and move constructor cannot be inherited — you must define those in the derived class if needed. Base class constructors are called in the order listed in the class declaration (for multiple inheritance), not the order written in the initializer list.

Redefining Base Class Functions

▶

A derived class can redefine a function from the base class using the same name and parameter list. Objects of the base class use the base version; objects of the derived class use the derived version.

class GradedActivity
{
public:
    virtual char getLetterGrade() const;  // base version
    void setScore(double s) { score = s; }
protected:
    double score;
};

class PassFailActivity : public GradedActivity
{
public:
    char getLetterGrade() const override;  // redefined version
};

// Objects use their own class's version
GradedActivity exam;
exam.setScore(75.0);
cout << exam.getLetterGrade();  // uses GradedActivity version

PassFailActivity quiz;
quiz.setScore(75.0);
cout << quiz.getLetterGrade(); // uses PassFailActivity version

Redefining vs. Overriding: a non-virtual redefined function is statically bound (resolved at compile time based on pointer/reference type — can call wrong version). A virtual overridden function is dynamically bound (resolved at runtime based on actual object type — always calls correct version). This distinction is the core of polymorphism.

Virtual Functions

▶

A virtual function is dynamically bound — at runtime, C++ determines the actual type of the object and calls the correct version. Declared with the virtual keyword in the base class.

class GradedActivity
{
public:
    // virtual — will be dynamically bound
    virtual char getLetterGrade() const;
    virtual ~GradedActivity();   // virtual destructor — always do this!
};

class PassFailActivity : public GradedActivity
{
public:
    // override keyword (C++11): tells compiler this overrides a virtual
    char getLetterGrade() const override;
};

class SpecialExam : public PassFailActivity
{
public:
    // final: cannot be overridden in further derived classes
    char getLetterGrade() const override final;
};

The problem virtual solves:

// Without virtual — static binding picks wrong version
void displayGrade(GradedActivity &activity)
{
    cout << activity.getLetterGrade();  // always calls GradedActivity version!
}

PassFailActivity quiz;
quiz.setScore(75.0);
displayGrade(quiz);   // prints 'C' instead of 'P' — WRONG!

// With virtual — dynamic binding picks correct version
// Same displayGrade function, but getLetterGrade is now virtual
displayGrade(quiz);   // correctly prints 'P' — RIGHT!

Once a function is declared virtual in a base class, it is automatically virtual in ALL derived classes — you don't need to repeat the keyword. Always make destructors virtual if the class may be a base class — otherwise delete through a base pointer won't call the derived destructor, causing a resource leak. Use override to catch typos: the compiler will error if no matching virtual function exists in the base class.

Polymorphism

▶

Polymorphism means "the ability to take many forms." A base class pointer or reference can refer to derived class objects and call the correct overridden virtual function at runtime.

// Base class pointer can point to any derived object
GradedActivity *ptr;
GradedActivity exam;
PassFailActivity quiz;

ptr = &exam;
cout << ptr->getLetterGrade();   // calls GradedActivity version

ptr = &quiz;
cout << ptr->getLetterGrade();   // calls PassFailActivity version (polymorphism!)

// Function taking base class reference accepts ANY derived object
void displayGrade(GradedActivity &activity)
{
    cout << activity.getLetterGrade();  // correct version called at runtime
}

displayGrade(exam);  // GradedActivity version
displayGrade(quiz);  // PassFailActivity version — polymorphism!

Polymorphism ONLY works through pointers or references. If you use a base class variable directly (not pointer/reference), object slicing occurs — the derived part is lost. Always use pointers or references when you need polymorphic behavior. This is why function parameters for polymorphic use are typically references.

Static Binding (compile time)

Non-virtual functions. Call resolved based on type of pointer/reference — may call wrong version for derived objects.

Dynamic Binding (runtime)

Virtual functions. Call resolved based on actual type of object at runtime — always calls the correct version.

Abstract Base Classes & Pure Virtual Functions

▶

A pure virtual function has no implementation in the base class — derived classes MUST override it. A class with at least one pure virtual function is an abstract base class — no objects of it can be created.

class Shape   // abstract base class
{
public:
    // Pure virtual functions — = 0 means no base implementation
    virtual double getArea() const = 0;
    virtual double getPerimeter() const = 0;
    virtual void draw() const = 0;

    // Can still have non-pure virtual functions
    virtual void printInfo() const
    {
        cout << "Area: " << getArea() << endl;
    }

    virtual ~Shape() { }   // virtual destructor
};

// Derived class MUST implement all pure virtual functions
class Circle : public Shape
{
private:
    double radius;
public:
    Circle(double r) : radius(r) { }
    double getArea() const override { return 3.14159 * radius * radius; }
    double getPerimeter() const override { return 2 * 3.14159 * radius; }
    void draw() const override { cout << "Drawing circle\n"; }
};

// Shape s;       // ERROR — cannot create abstract class object
Shape *ptr = new Circle(5.0);   // OK — pointer to derived
cout << ptr->getArea();         // polymorphism works

Abstract base classes define an interface that all derived classes must implement. They're a powerful design tool — you can write functions that accept a Shape* or Shape& and work with any shape: Circle, Rectangle, Triangle, etc., without knowing the specific type. If a derived class doesn't implement ALL pure virtual functions, it also becomes abstract.

Multiple Inheritance

▶

A derived class can inherit from more than one base class. Each base class gets its own access specifier.

class Square
{
public:
    Square(int side);
    int getSide() const;
};

class RectSolid
{
public:
    RectSolid(int l, int w, int h);
    int getVolume() const;
};

// Cube inherits from BOTH Square and RectSolid
class Cube : public Square, public RectSolid
{
public:
    // Pass arguments to BOTH base constructors
    Cube(int side) : Square(side), RectSolid(side, side, side) { }
};

// Base constructors called in ORDER LISTED in class declaration
// Square constructor runs first, then RectSolid

Naming conflicts: if both base classes define a function with the same name:

class A { public: void show() { cout << "A\n"; } };
class B { public: void show() { cout << "B\n"; } };
class C : public A, public B
{
public:
    // Option 1: redefine in derived class
    void show() { cout << "C\n"; }

    // Option 2: use scope resolution to call specific base version
    void showA() { A::show(); }
    void showB() { B::show(); }
};

C obj;
obj.show();   // calls C's version — unambiguous
obj.A::show(); // explicitly call A's version

Multiple inheritance is powerful but complicated. Name conflicts, the "diamond problem" (two base classes sharing a common ancestor), and complex constructor ordering make it easy to get wrong. Use it sparingly and prefer composition (aggregation) or interface-style abstract base classes when possible.
