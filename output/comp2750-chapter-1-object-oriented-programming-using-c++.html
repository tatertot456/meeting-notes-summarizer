Abstract Data Types (ADTs)

▶

Proper planning is essential to successful program implementation. Good design methodology focuses on what needs to be done — not how it's done. This attention to operations before implementation is the foundation of ADTs.

A data structure is a technique of storing and organizing data so it can be used efficiently. In program models, data structures are described by ADTs — defined indirectly through operations, not inner structure.

Example — Stack ADT (LIFO: Last In, First Out):

PUSH

Add an item to the top of the stack.

POP

Remove the item at the top of the stack.

TOP

Return the value of the item at the top — without removing it.

EMPTY

Determine if the stack is empty.

CREATE

Create a new empty stack.

These operations describe WHAT we can do with a stack — not HOW it's implemented internally. The implementation details (array? linked list? dynamic memory?) are reserved for later. ADTs are then implemented through class definitions in C++. This separation of interface from implementation is the foundation of good software design.

Encapsulation

▶

The fundamental notion of OOP is the object — a data structure combined with the operations pertinent to that structure. In C++, objects are defined through classes.

Member Variables / Data Members

The data stored inside the class — the object's state.

Methods / Member Functions

The operations the object can perform — its behavior.

Public Interface

Methods and data accessible from outside the object. What users interact with.

Private

Implementation details hidden from outside. Protected from side effects and accidental corruption.

Information hiding: every object is treated as a black box. You use it based on what it does, not how it does it. This confines implementation errors to the methods of a particular class — making bugs easier to detect and fix. Objects communicate through message passing (method calls) — the degree of interaction is precisely defined by these communication methods.

Advantages of encapsulation: stronger link between data and operations mirrors real-world behavior; errors are confined to specific classes; implementation details are hidden preventing side effects; objects can be replaced with more efficient versions while keeping the same public interface.

Inheritance

▶

Inheritance is the technique of reusing existing class definitions to derive new classes. Derived classes inherit attributes and behavior from base classes — an "is a" relationship.

Access SpecifierEffect on Inherited Members
publicPreserves base class access levels. Everyone aware of Base knows Child inherits from it.
protectedPublic and protected base members become protected in derived. Only Child and its children know about the inheritance.
privateAll base members become private in derived. No one outside Child knows about the inheritance.

Multiple Inheritance & the Diamond Problem:

// B and C both inherit from A
// D inherits from both B and C
// If B and C both override a method from A,
// D could inherit two different versions — compiler error!

// Solution: virtual inheritance
class B : virtual public A { ... };
class C : virtual public A { ... };
class D : public B, public C { ... };
// Now only ONE copy of A exists in D

Java avoids the diamond problem by prohibiting multiple inheritance entirely. C++ allows it but requires virtual inheritance to resolve ambiguity. Derived classes can add, delete, and modify (override) methods and data members from the base class. The inheritance hierarchy is represented in UML with a triangle/arrow pointing to the base class.

Pointers

▶

A pointer is a variable whose value is the address of another variable in memory. Every variable has two attributes: its value (what it stores) and its location (where it is stored).

int i = 15, j, *p, *q;   // p and q are pointers to int
// i and j are regular ints

p = &i;      // p now holds the ADDRESS of i (&i = "address of i")
*p = 20;     // dereference p: changes the VALUE at the address (i = 20)

// Dynamic memory
p = new int;    // allocate memory at runtime, p holds the address
*p = 42;
delete p;       // release the memory
p = nullptr;    // ALWAYS set to nullptr after delete — avoids dangling pointer!

// Dynamic array
int *arr;
arr = new int[n];      // allocate array of size n
arr[0] = 5;            // array notation
*(arr + 1) = 10;       // pointer arithmetic — same as arr[1]
delete[] arr;          // delete array — note the []

Dangling Reference

Pointer still holds address of deleted memory. Dereferencing causes undefined behavior. Fix: set to nullptr after delete.

Memory Leak

Allocating memory without deleting the previous allocation, or a pointer going out of scope. Memory becomes inaccessible — accumulates until none is available.

Pointer Arithmetic

*(arr + n) is equivalent to arr[n]. The array name IS a pointer to the first element. Add offset to base address, dereference to access any element.

Copy Constructors

Default copy copies pointer ADDRESS — both objects point to same memory. Write a copy constructor to deep-copy the pointed-to object. Also override operator= for the same reason.

// Pointer vs reference — both give access to original variable
int n = 5, *p = &n, &r = n;
n = 7;     // n, *p, and r are all now 7
*p = 7;    // same result
r = 7;     // same result

// Const pointer vs pointer to const
int* const ptr1 = &n;    // constant pointer — can't change where it points
const int* ptr2 = &n;    // pointer to constant — can't change value through pointer

// Pointer to function
void temp() { ... }
void (*funcPtr)() = &temp;
(*funcPtr)();    // call the function through the pointer

Objects that contain pointer members MUST have a destructor defined. When a local pointer variable goes out of scope, the pointer's memory (the address) is freed — but the object it POINTS TO remains allocated, causing a memory leak. The destructor explicitly deletes that pointed-to memory. Similarly, the copy constructor must deep-copy to prevent two objects sharing the same dynamically allocated memory.

Polymorphism

▶

Polymorphism allows objects of different types to respond to method calls with the same name — each according to its own type-specific behavior.

Static Binding (Early)

Method chosen at COMPILE TIME based on the pointer's declared type. Default behavior in C++. Faster but inflexible.

Dynamic Binding (Late)

Method chosen at RUNTIME based on the actual type of the object being pointed to. Enabled with the virtual keyword. More flexible.

class Base {
public:
    void Method1() { cout << "Base::Method1" << endl; }
    virtual void Method2() { cout << "Base::Method2" << endl; }
};

class Derived : public Base {
public:
    void Method1() { cout << "Derived::Method1" << endl; }
    void Method2() { cout << "Derived::Method2" << endl; }
};

Base* obj = new Derived();   // pointer is Base type, object is Derived

obj->Method1();   // "Base::Method1"    — static binding, uses pointer type
obj->Method2();   // "Derived::Method2" — dynamic binding, uses object type

Without virtual: early binding — the compiler decides at compile time based on the POINTER TYPE (Base), so Base::Method1 runs even though the object is a Derived. With virtual: late binding — the runtime checks the actual object type (Derived) and calls Derived::Method2. This is the foundation of runtime polymorphism in C++ — you send a message (method call) without knowing the exact receiver type, and the object handles it appropriately.

Aggregation (Composition)

▶

Aggregation models a "has a" relationship — one class contains another as a member. Distinct from inheritance ("is a"). Represented in UML with a diamond.

class Address {
    string street, city, state;
};

class Phone {
    string number;
};

class Person {
    Address permanentAddress;   // Person HAS AN Address
    Address campusAddress;
    Phone cell;                 // Person HAS A Phone
    Phone landline;
};

Aggregation vs Inheritance: A Student IS A Person (inheritance). A Student HAS AN Address (aggregation). Choose the right relationship for the design — using inheritance when aggregation is more appropriate leads to fragile, tightly coupled designs. If in doubt, prefer aggregation ("has a") over inheritance ("is a").

The Standard Template Library (STL)

▶

The STL provides three generic entities plus function objects, implemented through templates enabling static binding polymorphism — often more efficient than dynamic binding.

Containers

Data structures that hold objects of the same type. Implemented as template classes. Methods specify operations on the data AND the structure itself.

Iterators

Objects that access container elements. Five types: input, output, forward, bidirectional, random. Essentially generalized pointers — can be dereferenced and manipulated like pointers.

Algorithms

~70 generic functions for searching, sorting, etc. Independent of containers — work on any container that provides an iterator interface. Reduces library complexity dramatically.

Function Objects

Classes that overload operator(). Maintain state information in functions passed to other functions. More powerful than raw function pointers for some STL manipulations.

Iterator TypeCapabilities
InputRead a sequence of values
OutputWrite a sequence of values
ForwardRead, write, move forward
BidirectionalLike forward, plus move backward
RandomMove freely in any direction — most powerful

Vectors in the STL

▶

A vector is one of the simplest STL containers — a dynamic array. Elements stored contiguously in memory. Supports random access via index. Useful when list length is unknown but removal is rare.

#include <vector>

// Create vectors
vector<int> v1;                    // empty vector
vector<int> v2(5);                 // 5 elements, default-initialized
vector<int> v3(5, 10);             // 5 elements, all = 10

// Add elements
v1.push_back(42);     // add to end — O(1) amortized
v1.push_back(7);

// Access elements
v1[0];                // array notation — no bounds checking
v1.at(0);             // bounds-checked access — throws out_of_range

// Size and capacity
v1.size();            // current number of elements
v1.capacity();        // allocated storage (may be larger than size)
v1.empty();           // true if size == 0

// Iterator access
for (auto it = v1.begin(); it != v1.end(); ++it) {
    cout << *it;      // dereference iterator like a pointer
}

// Range-based for (C++11)
for (int x : v1) {
    cout << x;
}

// Remove last element
v1.pop_back();

// Remove all elements
v1.clear();

Vectors resize automatically when elements are added beyond capacity — but this involves allocating new memory and copying all elements, which is O(n). Normally push_back is O(1) amortized because vectors double their capacity when they resize, keeping the average cost constant. Use reserve(n) to pre-allocate if you know the approximate size — avoids repeated reallocations. The STL algorithms (sort, find, count, etc.) all work with vectors via iterators.
